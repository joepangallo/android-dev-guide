val vs var — declaring variables

val is a read-only (assign-once) reference — prefer it by default. var is reassignable. Types are usually inferred, so you rarely write them; add an explicit type after a colon when you want to be clear or the value is ambiguous.

Kotlin 
val name = "Ada"            // String, inferred, cannot be reassigned
var score = 0               // Int, inferred, reassignable
score = score + 10          // OK
// name = "Grace"           // ❌ compile error: val cannot be reassigned

val pi: Double = 3.14159    // explicit type when you want it
const val MAX = 100         // compile-time constant (top-level / object only)
In Java this was… final String name = "Ada"; for val, and a plain int score = 0; for var. Kotlin flips the default: read-only is the easy, short option and mutability is the one you opt into.
valvarconsttype inference

Basic types

Kotlin has no primitive/object split in your code — everything is an object (Int, Double, Boolean, Char, String), and the compiler optimizes to primitives under the hood. Numbers do not auto-widen; convert explicitly.

Kotlin 
val count: Int = 42
val big: Long = 42L
val ratio: Double = 1.5
val small: Float = 1.5f
val flag: Boolean = true
val letter: Char = 'K'

val asLong: Long = count.toLong()   // explicit — no silent widening
val asDouble = count.toDouble()
// val x: Long = count              // ❌ Int is NOT a Long automatically
In Java this was… int vs Integer, and long x = anInt; widened silently. Kotlin requires .toLong() on purpose, which removes a whole class of accidental-conversion bugs.
IntLongDoubleBooleantoLong()

String templates & multiline strings

Embed values directly with $; wrap expressions in ${...}. Triple-quoted strings are raw (no escaping) and multiline — great for JSON literals or SQL.

Kotlin 
val user = "Ada"
val items = 3
println("Hi $user, you have $items items.")          // Hi Ada, you have 3 items.
println("Total: ${items * 2} (doubled)")             // expression in ${ }
println("Cost is \$5")                                // literal dollar sign

val json = """
    {
        "user": "$user",
        "items": $items
    }
""".trimIndent()                                      // raw + multiline
In Java this was… "Hi " + user + ", you have " + items + " items." (or String.format). Python's f-strings are the closest analog to Kotlin templates.
$template${expr}trimIndentmultiline

A complete program & comments

A runnable Kotlin file just needs a top-level main function — no class wrapper required. Comments use // and /* */, with /** */ for KDoc.

Kotlin 
// Single-line comment
/* Block comment */

/** KDoc: greets the user. */
fun main() {
    val name = "Android"
    println("Hello, $name!")   // top-level main — no class needed
}
In Java this was… public static void main(String[] args) inside a public class. Kotlin drops the ceremony; functions can live at the top level of a file.
main()println// commentKDoc

Declaring functions

Use fun. Parameters are name: Type; the return type follows the parameter list after a colon. A function that returns nothing returns Unit (you can omit it).

Kotlin 
fun add(a: Int, b: Int): Int {
    return a + b
}

fun greet(name: String) {        // returns Unit (≈ void), can be omitted
    println("Hi $name")
}
In Java this was… int add(int a, int b) — Java puts the type before the name. Kotlin puts it after (a: Int), and void becomes Unit.
funUnitreturn

Expression-body functions

When a function is a single expression, drop the braces and return — use =. The return type is inferred.

Kotlin 
fun add(a: Int, b: Int) = a + b              // type inferred as Int
fun square(x: Int): Int = x * x              // type written for clarity
fun isAdult(age: Int) = age >= 18            // returns Boolean
In Java this was… No equivalent before — Java always needs a body and an explicit return. This is one of the most-used Kotlin shorthands.
= expressionsingle-expression

Default & named arguments

Give parameters defaults so callers can omit them, and pass arguments by name to skip ahead or self-document. Together these replace most method overloads.

Kotlin 
fun createUser(
    name: String,
    age: Int = 0,
    isAdmin: Boolean = false
) { /* ... */ }

createUser("Ada")                                  // age=0, isAdmin=false
createUser("Ada", 30)                              // positional
createUser("Ada", isAdmin = true)                  // named — skip age
createUser(name = "Ada", age = 30, isAdmin = true) // all named
In Java this was… Several overloaded methods (createUser(name), createUser(name, age)…) or a builder. Kotlin collapses them into one function. This is everywhere in Compose: Text("Hi", color = Color.Red, fontSize = 18.sp).
default argsnamed argsname = value

Varargs

vararg accepts any number of arguments. Spread an existing array into a vararg with the * operator.

Kotlin 
fun sumAll(vararg nums: Int): Int = nums.sum()

sumAll(1, 2, 3, 4)              // 10
val arr = intArrayOf(5, 6, 7)
sumAll(*arr)                    // spread operator
In Java this was… int... nums — same idea. Java had no spread operator; you'd pass the array directly.
vararg*spread

Extension functions

Add a method to a type you don't own — even String or Int — without subclassing. Inside, this is the receiver. Compose and the standard library use these constantly.

Kotlin 
fun String.shout(): String = this.uppercase() + "!"

"hello".shout()                 // "HELLO!"

// Extension property
val String.firstChar: Char get() = this[0]
"Android".firstChar             // 'A'
In Java this was… Impossible cleanly — you'd write a StringUtils.shout(String s) static helper. Extensions let you call it as if it were a real method.
extension funreceiverthis

Local (nested) functions

Functions can be defined inside other functions and can capture the enclosing variables — handy for breaking up logic without polluting the class.

Kotlin 
fun process(values: List<Int>): Int {
    var total = 0
    fun accumulate(x: Int) {     // local fn, captures `total`
        total += x
    }
    values.forEach { accumulate(it) }
    return total
}
In Java this was… Not directly possible (you'd use a private helper method or a lambda). Python supports nested def, so Python users will recognize this.
local funclosure

Nullable vs non-null types

Kotlin's biggest win over Java: the type system tracks null. A normal type can never be null. Append ? to allow null. The compiler then forces you to handle the null case.

Kotlin 
var name: String = "Ada"
// name = null               // ❌ compile error — String can't be null

var nickname: String? = "Addie"   // nullable
nickname = null                   // ✅ allowed

val len = name.length             // ✅ safe, never null
// val n = nickname.length        // ❌ must handle null first
In Java this was… Every reference could be null, and NullPointerException at runtime was a constant threat. Kotlin moves the check to compile time.
Type?nullablenon-nullNPE

Safe call ?.

The safe-call operator calls a member only if the receiver is non-null; otherwise the whole expression is null. Chains short-circuit on the first null.

Kotlin 
val nickname: String? = null
val len: Int? = nickname?.length         // null (no crash)

// chaining
val city: String? = user?.address?.city  // null if user OR address is null

// safe call + apply action only when non-null
nickname?.let { println("Has nickname: $it") }
In Java this was… if (user != null && user.getAddress() != null) ... nested null checks. ?. collapses that to one line.
?.safe callchaining

Elvis operator ?:

Provide a fallback when the left side is null. Reads as "this, or else that." Often combined with return or throw for early exits.

Kotlin 
val nickname: String? = null
val display = nickname ?: "Anonymous"     // "Anonymous"

val len = nickname?.length ?: 0           // 0 when null

fun firstChar(s: String?): Char {
    val text = s ?: return ' '            // early return on null
    return text[0]
}
In Python this was… display = nickname or "Anonymous" (similar idea). In Java: nickname != null ? nickname : "Anonymous".
?:elvisdefault

Not-null assertion !! (use sparingly)

!! tells the compiler "trust me, this isn't null" — and throws NullPointerException if you're wrong. It defeats the point of null safety, so reach for ?. or ?: first.

Kotlin 
val nickname: String? = getNickname()
val len = nickname!!.length      // crashes with NPE if nickname is null

// Prefer one of these instead:
val safe = nickname?.length ?: 0
nickname?.let { use(it) }
⚠️
Warning

Treat !! as a code smell. If you find yourself writing it often, your types probably should be non-null, or you should handle the null with ?:/?.let.

!!assertionNPE

Smart casts & null narrowing

After you check for null (or a type) and the variable can't change, the compiler "smart casts" it to the non-null/narrowed type — no manual cast needed.

Kotlin 
fun describe(s: String?) {
    if (s != null) {
        // inside here, s is smart-cast to non-null String
        println("Length is ${s.length}")
    }
}

fun area(shape: Any): Double {
    if (shape is Circle) {       // type check
        return 3.14 * shape.r * shape.r   // smart-cast to Circle
    }
    return 0.0
}
In Java this was… if (shape instanceof Circle) { Circle c = (Circle) shape; ... } — you cast manually. Kotlin does it for you once the check proves the type.
smart castis!= null

if is an expression

In Kotlin if returns a value, so it replaces the ternary ?: from C/Java. The last expression in each branch is the result.

Kotlin 
val a = 5; val b = 9
val max = if (a > b) a else b          // if as expression

val label = if (max > 5) {
    "big"
} else {
    "small"
}                                       // multiline branches still return
In Java this was… int max = (a > b) ? a : b; — Kotlin has no ?: ternary because if/else already returns a value.
if expressionno ternary

when — the supercharged switch

when replaces switch and is far more flexible: match values, ranges, types, or arbitrary boolean conditions. It's also an expression that returns a value (then else is required for exhaustiveness).

Kotlin 
val grade = when (score) {
    in 90..100 -> "A"
    in 80..89  -> "B"
    in 70..79  -> "C"
    else       -> "F"
}

// when with no subject = a clean if/else-if chain
val sign = when {
    n > 0  -> "positive"
    n < 0  -> "negative"
    else   -> "zero"
}

// match types
when (x) {
    is Int    -> println("int")
    is String -> println("len ${x.length}")   // smart-cast
    else      -> println("other")
}
In Java this was… switch with break after each case (and only constant labels). Kotlin's when needs no break, allows ranges/types/conditions, and returns a value.
whenswitchin rangeis type

Ranges

Build ranges with .. (inclusive), ..< / until (exclusive end), downTo (descending), and step. Use them in loops and with in.

Kotlin 
1..5            // 1,2,3,4,5  (inclusive)
1..<5           // 1,2,3,4    (exclusive end — modern syntax)
1 until 5       // 1,2,3,4    (older equivalent)
10 downTo 1     // 10,9,...,1
0..10 step 2    // 0,2,4,6,8,10

val ok = 7 in 1..10          // true — membership test
val notLetter = 'x' !in 'a'..'f'  // true
In Python this was… range(1, 6) / range(10, 0, -1). Kotlin ranges are first-class objects you can also test with in.
1..5untildownTostepin

for loops

Kotlin's for iterates over anything iterable — ranges, lists, maps. There's no C-style for (int i = …); use a range instead, and withIndex() when you need the index.

Kotlin 
for (i in 1..3) println(i)              // 1 2 3

val fruits = listOf("apple", "pear")
for (f in fruits) println(f)

for ((index, f) in fruits.withIndex())  // index + value
    println("$index: $f")

for (i in fruits.indices) println(i)     // 0,1 — valid indices

val ages = mapOf("Ada" to 30, "Al" to 25)
for ((name, age) in ages) println("$name=$age")
In Java this was… The enhanced for (for (String f : fruits)) for items, and a C-style index loop for indices. Kotlin replaces the index loop with indices / withIndex().
for inwithIndexindices

while, do…while, and repeat

while and do/while work as you'd expect. For a fixed count, repeat(n) { } is the idiomatic shortcut. break and continue behave normally and support labels.

Kotlin 
var i = 0
while (i < 3) { println(i); i++ }

do { println("at least once") } while (false)

repeat(3) { index -> println("tick $index") }   // 0,1,2

outer@ for (a in 1..3) {
    for (b in 1..3) {
        if (a * b > 4) break@outer    // labeled break
    }
}
In Java this was… Same while/do-while; Java has labeled break too. Kotlin adds repeat(n){} as a clean fixed-count loop.
whiledo-whilerepeatbreak@label

Classes & the primary constructor

Declare properties right in the header — the primary constructor. Marking a parameter val/var makes it a property automatically. Use an init {} block for setup logic.

Kotlin 
class User(val name: String, var age: Int) {
    init {
        require(age >= 0) { "age must be non-negative" }
    }
    fun birthday() { age++ }
}

val u = User("Ada", 30)     // no `new` keyword
println(u.name)             // property access, no getter call
u.birthday()                // age is now 31
In Java this was… A field, a constructor that assigns it, plus getName()/setAge() — dozens of lines. Kotlin generates the backing field and accessors; new isn't needed.
classprimary ctorinitno new

data class — model objects for free

A data class auto-generates equals(), hashCode(), toString(), copy(), and destructuring components. This is the standard way to model UI state, API responses, and entities in Android.

Kotlin 
data class User(val name: String, val age: Int)

val a = User("Ada", 30)
val b = User("Ada", 30)
println(a == b)               // true — value equality, not reference
println(a)                    // User(name=Ada, age=30)

val older = a.copy(age = 31)  // immutable update — key for Compose state
val (name, age) = a           // destructuring
In Java this was… A POJO with hand-written or IDE-generated equals/hashCode/toString (or a Java record in 16+). Kotlin's copy() for immutable updates has no direct Java equivalent and is essential for Compose UiState.
data classcopy()equalsdestructuring

object & companion object

object declares a singleton (one instance, ever). A companion object inside a class holds members you'd call statically in Java — constants and factory functions.

Kotlin 
object Analytics {                 // singleton
    fun track(event: String) { /* ... */ }
}
Analytics.track("open")            // call on the object directly

class User private constructor(val name: String) {
    companion object {             // "static" members live here
        const val MAX_NAME = 32
        fun create(name: String) = User(name.take(MAX_NAME))  // factory
    }
}
val u = User.create("Ada")         // User.create — looks static
In Java this was… An enum-or-private-constructor singleton, and static fields/methods. Kotlin has no static keyword — use object / companion object instead.
objectcompanion objectsingletonstatic

enum class

Enums model a fixed set of constants and can carry properties and methods. They pair naturally with when (which becomes exhaustive over the enum).

Kotlin 
enum class Status(val label: String) {
    LOADING("Loading…"),
    SUCCESS("Done"),
    ERROR("Failed");

    fun isFinished() = this != LOADING
}

val s = Status.SUCCESS
println(s.label)                   // "Done"

val msg = when (s) {               // no `else` needed — exhaustive
    Status.LOADING -> "wait"
    Status.SUCCESS -> "ok"
    Status.ERROR   -> "retry"
}
In Java this was… Nearly identical — Java enums also take fields/methods. The big difference is Kotlin's when is checked for exhaustiveness, so adding a case forces you to handle it.
enum classexhaustive when

sealed class / sealed interface

A sealed type has a closed set of subtypes known at compile time. Perfect for representing UI state or a result (Loading / Success / Error). when over a sealed type is exhaustive, so the compiler catches missed cases.

Kotlin 
sealed interface UiState {
    data object Loading : UiState
    data class Success(val data: List<String>) : UiState
    data class Error(val message: String) : UiState
}

fun render(state: UiState) = when (state) {
    UiState.Loading      -> "Spinner"
    is UiState.Success   -> "Show ${state.data.size} items"  // smart-cast
    is UiState.Error     -> "Oops: ${state.message}"
}   // no `else` — all subtypes covered
In Java this was… An abstract base class plus instanceof chains, with no compiler guarantee you handled every subtype. Sealed types + exhaustive when are a core Android architecture pattern (you'll use this for ViewModel state in Sessions 4–5).
sealedUiStateexhaustivedata object

Inheritance & interfaces

Classes are final by default — mark them open to allow subclassing, and override members explicitly. Interfaces can have default method bodies.

Kotlin 
interface Greeter {
    fun greet(): String                 // abstract
    fun hello() = "Hi from ${greet()}"  // default body
}

open class Animal(val name: String) {   // `open` = subclassable
    open fun sound(): String = "..."
}

class Dog(name: String) : Animal(name), Greeter {
    override fun sound() = "Woof"
    override fun greet() = name
}
In Java this was… extends / implements, with classes open by default. Kotlin reverses that: closed by default for safety; : Base() replaces extends and also calls the super-constructor.
openoverrideinterface: Base()

Lists, sets, maps — read-only vs mutable

Kotlin separates read-only interfaces (List, Set, Map) from mutable ones (MutableList…). Builders: listOf/mutableListOf, setOf, mapOf. Prefer read-only unless you truly need to mutate.

Kotlin 
val nums = listOf(1, 2, 3)              // read-only List<Int>
val mutable = mutableListOf(1, 2)       // can add/remove
mutable.add(3)

val set = setOf(1, 1, 2)                // {1, 2} — no duplicates
val ages = mapOf("Ada" to 30, "Al" to 25)   // Map via `to`
println(ages["Ada"])                    // 30 (nullable get)
println(nums[0])                        // index access
In Java this was… new ArrayList<>(), List.of(...), new HashMap<>(). Kotlin's read-only/mutable split is enforced by the type, not just convention.
listOfmutableListOfmapOfto

map — transform each element

map returns a new list with each element transformed by the lambda. The implicit single parameter is it.

Kotlin 
val nums = listOf(1, 2, 3, 4)
val doubled = nums.map { it * 2 }           // [2, 4, 6, 8]
val names = listOf("ada", "al")
val caps = names.map { it.uppercase() }     // [ADA, AL]
val lengths = names.map { it.length }       // [3, 2]
In Java this was… nums.stream().map(n -> n * 2).collect(Collectors.toList()). Kotlin's version is built into collections — no .stream()/.collect() boilerplate.
mapittransform

filter — keep matching elements

filter keeps only elements for which the predicate is true; filterNot is the inverse. Chain freely with map and friends.

Kotlin 
val nums = listOf(1, 2, 3, 4, 5, 6)
val evens = nums.filter { it % 2 == 0 }      // [2, 4, 6]
val odds  = nums.filterNot { it % 2 == 0 }   // [1, 3, 5]

// chaining: even numbers, doubled
val result = nums.filter { it % 2 == 0 }
                 .map { it * 2 }             // [4, 8, 12]
In Python this was… [n for n in nums if n % 2 == 0] or filter(lambda n: n%2==0, nums). Same concept, fluent chaining in Kotlin.
filterfilterNotpredicate

forEach & quick checks

forEach runs an action per element (side effects). any, all, none, count, and find answer common questions without a manual loop.

Kotlin 
val nums = listOf(1, 2, 3, 4)
nums.forEach { println(it) }

nums.any { it > 3 }        // true  — at least one
nums.all { it > 0 }        // true  — every element
nums.none { it > 10 }      // true  — not a single one
nums.count { it % 2 == 0 } // 2     — how many match
nums.find { it > 2 }       // 3     — first match or null
In Java this was… forEach exists on streams; anyMatch/allMatch/noneMatch/count/findFirst are the stream equivalents. Kotlin names are shorter and work directly on the collection.
forEachanyallfindcount

Aggregating: sum, fold, reduce

Specialized aggregates (sumOf, maxOrNull, average) cover most needs; fold lets you accumulate with a custom seed and operation.

Kotlin 
val nums = listOf(3, 1, 4, 1, 5)
nums.sum()                       // 14
nums.maxOrNull()                 // 5  (null if empty)
nums.average()                   // 2.8
nums.sumOf { it * it }           // 52 — sum of squares

// fold(initial) { acc, item -> ... }
val product = nums.fold(1) { acc, n -> acc * n }   // 60
In Java this was… stream().reduce(...) or mapToInt(...).sum(). fold ≈ reduce with an identity value; Python's functools.reduce is the analog.
sumsumOffoldreduceaverage

Organizing: groupBy, sortedBy, associateBy

Reshape collections in one call: group into a map, sort by a key, build a lookup map, drop duplicates.

Kotlin 
data class User(val name: String, val age: Int)
val users = listOf(User("Ada", 30), User("Al", 30), User("Bo", 25))

users.groupBy { it.age }          // {30=[Ada, Al], 25=[Bo]}
users.sortedBy { it.age }         // by age ascending
users.sortedByDescending { it.name }
users.associateBy { it.name }     // Map<name, User> — fast lookup
users.map { it.age }.distinct()   // [30, 25]
In Java this was… Collectors.groupingBy(...), sorted(Comparator.comparing(...)), toMap(...). Kotlin's one-word forms are dramatically shorter.
groupBysortedByassociateBydistinct

Lambda syntax & it

A lambda is { params -> body }. When there's exactly one parameter, you can drop it and refer to it as it. The last expression is the return value.

Kotlin 
val square: (Int) -> Int = { x -> x * x }
square(5)                                  // 25

val nums = listOf(1, 2, 3)
nums.map { it * 2 }                        // single param -> use `it`
nums.mapIndexed { index, value -> "$index:$value" }  // two params
In Java this was… x -> x * x — Java lambdas use the arrow too, but wrap them in functional interfaces. Kotlin lambdas live in { } and provide the implicit it.
{ -> }itlambda

Trailing-lambda syntax

If the last parameter is a function, move that lambda outside the parentheses. If it's the only argument, drop the parens entirely. This is what makes Compose read like a layout DSL.

Kotlin 
// these are all the same call:
nums.filter({ it > 2 })
nums.filter() { it > 2 }
nums.filter { it > 2 }              // idiomatic

// why Compose looks the way it does:
Button(onClick = { count++ }) {     // trailing lambda = content
    Text("Clicked $count times")
}
In Java this was… No equivalent — the lambda always stayed inside the parentheses. Trailing-lambda syntax is key to reading Jetpack Compose code.
trailing lambdaCompose DSL

Higher-order functions

Functions that take or return other functions. Declare a function-type parameter as (In) -> Out. This is how you write reusable callbacks — exactly like Compose's onClick: () -> Unit.

Kotlin 
fun applyTwice(x: Int, op: (Int) -> Int): Int = op(op(x))

applyTwice(3) { it + 1 }              // 5

// returning a function
fun multiplier(factor: Int): (Int) -> Int = { n -> n * factor }
val triple = multiplier(3)
triple(10)                            // 30

// function reference with ::
fun isEven(n: Int) = n % 2 == 0
listOf(1, 2, 3, 4).filter(::isEven)   // [2, 4]
In Java this was… Function<Integer,Integer> / BiFunction functional interfaces, and method references like this::isEven. Kotlin's (Int) -> Int type is a built-in, lighter syntax.
higher-order(In)->Out::refcallback

The five scope functions — pick the right one

Scope functions run a block "on" an object. They differ in two ways: how you refer to the object (it vs this) and what they return (the lambda result vs the object itself). This table is the whole game:

letitlambda resultnull-safe transform of a value

applythisthe objectconfigure / build an object

Memory aid

Returns the object? → apply / also. Returns the result? → let / run / with. Uses this? → run / with / apply. Uses it? → let / also.

let — null-safe transforms

Run a block only when a value is non-null (with ?.let), referring to it as it and returning the block's result.

Kotlin 
val name: String? = getName()
val length = name?.let { it.trim().length } ?: 0   // only runs if non-null

// also good for scoping a temporary
val upper = "hello".let { it.uppercase() }         // "HELLO"
let?.letit

apply — configure & return the object

Set up an object's properties inside the block (using this) and get the same object back. The classic "builder without a builder."

Kotlin 
val intent = Intent(context, DetailActivity::class.java).apply {
    putExtra("id", 42)
    putExtra("title", "Hello")
    flags = Intent.FLAG_ACTIVITY_NEW_TASK
}   // returns the configured Intent
In Java this was… Repeating the variable name on every line (intent.putExtra(...); intent.putExtra(...);) or a builder pattern. apply removes the repetition.

Why coroutines? (the big idea)

Network calls and database reads must not run on the main/UI thread (it freezes the app and can crash with NetworkOnMainThreadException). Coroutines let you write asynchronous code that looks sequential — no callbacks — while running off the main thread. You'll use these in Session 5 (networking) and Session 6 (Room).

🧵
Mental model

A coroutine is a "lightweight thread" you can suspend (pause) and resume without blocking a real OS thread. Thousands can run on a small thread pool.

coroutineasyncmain thread

suspend functions

A suspend function can pause and resume. It can only be called from another suspend function or from a coroutine. The compiler handles the pause/resume — your code reads top-to-bottom.

Kotlin 
suspend fun fetchUser(): User {
    val response = api.getUser()      // suspends here, doesn't block the thread
    return response.toUser()          // resumes when the result arrives
}

// reads sequentially, runs asynchronously
suspend fun loadProfile() {
    val user = fetchUser()            // waits without blocking
    val posts = fetchPosts(user.id)   // then this
    show(user, posts)
}
In Java this was… Callbacks (onSuccess/onError), Future/CompletableFuture, or RxJava chains — easy to nest into "callback hell." suspend flattens it back to straight-line code.
suspendawaitsequential

Launching a coroutine (launch + a scope)

You start coroutines from a CoroutineScope. In Android you almost never make your own — you use viewModelScope (cancels when the ViewModel clears) or lifecycleScope. launch fires off work that returns no value.

Kotlin 
class ProfileViewModel(
    private val repo: UserRepository
) : ViewModel() {

    fun load() {
        viewModelScope.launch {        // scoped to this ViewModel
            val user = repo.fetchUser()    // suspend call — off main thread
            _uiState.value = UiState.Success(user)
        }
    }
}
💡
Structured concurrency

Because the coroutine is tied to viewModelScope, it is automatically cancelled when the screen goes away — no leaked work, no crashes from updating a dead UI.

launchviewModelScopeCoroutineScope

Dispatchers & withContext

A dispatcher decides which thread(s) a coroutine uses. Switch threads with withContext: Dispatchers.IO for network/disk, Dispatchers.Default for CPU work, Dispatchers.Main for UI. (Modern Retrofit/Room are already main-safe, but knowing this matters.)

Kotlin 
suspend fun readFile(path: String): String =
    withContext(Dispatchers.IO) {     // run this block on an IO thread
        File(path).readText()          // blocking call, safely off main
    }                                   // result handed back to caller's thread
In Java this was… AsyncTask (deprecated), an ExecutorService, or new Thread() plus a Handler to post back to the UI thread. withContext replaces all of that with one expression.
withContextDispatchers.IODispatchers.Main

async/await — run work in parallel

When two independent calls can happen at once, start each with async (returns a Deferred) and await both. They run concurrently instead of one-after-another.

Kotlin 
suspend fun loadDashboard() = coroutineScope {
    val user = async { repo.fetchUser() }     // starts immediately
    val feed = async { repo.fetchFeed() }     // also starts immediately
    show(user.await(), feed.await())          // waits for both, in parallel
}
asyncawaitparallelDeferred

Handling errors in coroutines

A suspend call can throw (network down, bad JSON). Wrap it in an ordinary try/catch — because the code reads sequentially, the catch works just like it does in synchronous code. Catch the specific exception and update state, and never swallow CancellationException (that's how structured concurrency cancels work).

Kotlin 
fun load() {
    viewModelScope.launch {
        _uiState.value = UiState.Loading
        try {
            val user = repo.fetchUser()        // suspend call may throw
            _uiState.value = UiState.Success(user)
        } catch (e: CancellationException) {
            throw e                            // ⬅ never swallow — let cancellation propagate
        } catch (e: IOException) {
            _uiState.value = UiState.Error("Network error")
        }
    }
}
⚠️
Structured-concurrency gotcha

If one child of a coroutineScope { } (say, one async { }) throws, it cancels its siblings and re-throws at the scope. So put the try/catch around the coroutineScope / await() — not deep inside one child — when you want the whole parallel block to fail together.

In Java this was… An onError callback or an RxJava .onErrorReturn() operator — error handling lived in a separate branch from the happy path. Coroutines fold it back into a normal try/catch right where the call is made. You'll lean on this in Session 5 (networking) and Session 6 (Room).
