    🔗
    Bridge to the Compose state lesson

    Rotation is a configuration change. The system destroys the Activity (onPause → onStop → onDestroy) and immediately recreates it (onCreate → onStart → onResume). Any value you held in a plain variable — or in a remember { } in Compose — is gone.

    That’s exactly why our next session teaches rememberSaveable (survives config changes) and the ViewModel (survives config changes, lives until the screen is truly finished). Try “Rotate device” above, then read the recreate count, then look at the code below.

    Teaching panel: the seven callbacks

    A quick-reference for each callback — when it runs, what’s guaranteed, and the one thing students must remember.

    onCreate() — the only one you must override

    Runs: exactly once per Activity instance, when the Activity is first created (cold start) and after every recreate (rotation, process restart).

    Guaranteed: this is where you receive the saved Bundle (or null the first time). The window isn’t visible yet.

    Do here: one-time setup — call setContent { } to attach your Compose UI, read savedInstanceState, obtain your ViewModel, wire up bindings.

    Minimal Compose Activity

    Kotlin 
    class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        Log.d("MainActivity", "onCreate() called")

        setContent {
            MyAppTheme {
                Surface { GreetingScreen() }
            }
        }
    }
}

    onStart() — the Activity becomes visible

    Runs: after onCreate(), and again every time the Activity returns from the background (after onRestart()). Can run many times in one instance’s life.

    Guaranteed: the Activity is now visible to the user but not yet in the foreground / interactive.

    Do here: start work that should only happen while the UI is on screen but that you can afford to do a moment before the user can touch it — register broadcast receivers that update the UI, begin showing content.

    onResume() — now in the foreground & interactive

    Runs: right after onStart(), and again whenever the app returns to the foreground (e.g. after a dialog closes, or coming back from Home).

    Guaranteed: the Activity is at the top of the stack and receiving user input — this is the Running state.

    Do here: resume anything you paused — camera preview, sensors, animations, location updates, a game loop. Pair every acquire-in-onResume with a release-in-onPause.

    Acquire / release pairing

    Kotlin 
    override fun onResume() {
    super.onResume()
    sensorManager.registerListener(
        listener, accelerometer,
        SensorManager.SENSOR_DELAY_UI
    )
}

override fun onPause() {
    super.onPause()
    sensorManager.unregisterListener(listener)
}

    onPause() — losing focus (keep it fast!)

    Runs: as soon as the Activity loses focus — a dialog appears, the user presses Home, another Activity comes to the front, or the device rotates.

    Guaranteed: the Activity may still be partially visible (e.g. behind a translucent dialog). The next state could be Resumed (focus came back) or Stopped (fully hidden).

    Do here: pause battery-/CPU-heavy work (camera, sensors, animations). Do NOT do heavy work or disk/network writes — the next Activity can’t resume until onPause() returns.

    Modern guidance: persist user data in onStop(), not here. onPause() is for quick releases only.

    onStop() — no longer visible

    Runs: when the Activity is fully hidden — the user went Home, switched apps, or this Activity is finishing.

    Guaranteed: the Activity is in the background. The instance is kept in memory, so it can come back via onRestart() — unless the system needs the memory and kills the process.

    Do here: persist data to a database / DataStore, release resources you don’t need while hidden, stop refresh work. This is the recommended place for relatively heavy save work.

    Persist on stop

    Kotlin 
    override fun onStop() {
    super.onStop()
    // Heavier persistence is OK here.
    lifecycleScope.launch {
        draftRepository.saveDraft(currentDraft)
    }
}

    onRestart() — returning from the background

    Runs: only when a stopped (still-alive) Activity is about to be shown again. It runs before onStart(). It never runs on first launch and never runs after a destroy/recreate.

    Guaranteed: the instance survived in memory — your onCreate() did not run again, so your fields still hold their values.

    Do here: usually nothing special; refresh anything that may be stale (re-query data, restart polling). Most logic belongs in onStart()/onResume() so the cold-start path is covered too.

    onDestroy() — the instance is going away

    Runs: when the Activity is finishing (the user pressed Back or you called finish()) or the system is destroying it to recreate it (configuration change) or to reclaim resources.

    Guaranteed: this is the last callback. Use isFinishing to tell a real finish from a config-change recreate.

    Do here: final cleanup — but don’t rely on it for saving user data, because a process kill can skip it. Use onStop() + onSaveInstanceState() for durability.

    Saving UI state across recreation

    There are three layers, from quick-and-transient to fully durable. Students confuse these constantly — clarify the boundaries explicitly.

    In Compose, remember { } survives recomposition but not recreation. rememberSaveable { } writes into the saved-instance Bundle, so it survives rotation and process death restore. Use it for small, screen-local UI state (a text field, a selected tab).

    Kotlin 
    @Composable
fun NameField() {
    // Lost on rotation:
    // var name by remember { mutableStateOf("") }

    // Survives rotation & process-death restore:
    var name by rememberSaveable { mutableStateOf("") }

    OutlinedTextField(
        value = name,
        onValueChange = { name = it },
        label = { Text("Your name") }
    )
}

    Rule of thumb: if it’s small and tied to this one screen, rememberSaveable is enough.

    A ViewModel is scoped to the screen and survives configuration changes without any save/restore code — the same instance is handed back after recreation. It’s the right home for UI state and the logic that produces it. It does not survive process death on its own (use SavedStateHandle for that).

    Kotlin 
    class CounterViewModel : ViewModel() {
    var count by mutableStateOf(0)
        private set

    fun increment() { count++ }
}

@Composable
fun CounterScreen(vm: CounterViewModel = viewModel()) {
    Column {
        Text("Count: ${vm.count}")
        Button(onClick = vm::increment) { Text("Add") }
    }
}
// Rotating the device keeps the count — same ViewModel instance.

    When the system kills your backgrounded process for memory, the ViewModel is gone too. The Bundle saved via onSaveInstanceState() (or, in a ViewModel, SavedStateHandle) is restored when the user returns. Keep it tiny — IDs and scroll positions, not whole lists.

    Kotlin 
    class SearchViewModel(
    private val state: SavedStateHandle
) : ViewModel() {
    // Survives both rotation AND process death:
    var query: String
        get() = state["query"] ?: ""
        set(value) { state["query"] = value }
}

    Anything the user expects to still be there next week — notes, settings, the offline cache — must go to durable storage in onStop() or as the user edits it: DataStore for key-value/preferences, Room for structured data. This is CLO 4 territory (data persistence) and the “local persistence” requirement in the course project.

    remember = recomposition · rememberSaveable = + rotation · ViewModel = + rotation (logic home) · SavedStateHandle = + process death · Room/DataStore = + app restart.

    Event → callback cheat sheet

    The exact callback sequence for each device event. This is the table to memorize for the quiz.

    Event Callback sequence Instance survives?
    Cold launch onCreate → onStart → onResume New instance
    Dialog covers app onPause (then onResume when dismissed) Yes
    Press Home / switch apps onPause → onStop Yes (in background)
    Return from background onRestart → onStart → onResume Yes (same instance)
    Rotate / config change onPause → onStop → onDestroy then onCreate → onStart → onResume No — destroyed & recreated
    Press Back / finish() onPause → onStop → onDestroy (isFinishing == true) No — gone for good
    Low-memory kill (backgrounded) No callbacks guaranteed; on return onCreate → onStart → onResume No — process killed
    ⚠️
    The trap on the exam

    Returning from a stopped app runs onRestart() first; returning after a process kill runs onCreate() (no onRestart). And onDestroy() is not guaranteed to run on a low-memory kill — so never put “save my data” logic there alone.

    Persist in onStop() (or as the user edits). Do not rely on onDestroy() — a process kill can skip it. Keep onPause() fast; don’t do disk/network writes there because the next Activity can’t resume until it returns.

    No — the system may kill the process without calling it. When it does run, check isFinishing: true means a genuine finish (Back/finish()); false means a recreate (e.g. config change).

    Paste-in logging template

    Kotlin 
    class MainActivity : ComponentActivity() {

    private val tag = "MainActivity"

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        Log.d(tag, "onCreate() — savedInstanceState " +
            if (savedInstanceState == null) "null (fresh)" else "restored")
        setContent { MyAppTheme { Surface { GreetingScreen() } } }
    }

    override fun onStart()   { super.onStart();   Log.d(tag, "onStart()") }
    override fun onResume()  { super.onResume();  Log.d(tag, "onResume()") }
    override fun onPause()   { super.onPause();   Log.d(tag, "onPause()") }
    override fun onStop()    { super.onStop();    Log.d(tag, "onStop()") }
    override fun onRestart() { super.onRestart(); Log.d(tag, "onRestart()") }

    override fun onDestroy() {
        super.onDestroy()
        Log.d(tag, "onDestroy() — isFinishing=$isFinishing")
    }
}

    Watch isFinishing: false on rotate, true on Back.

    Three scenarios you will hit in production

    For each one: read the symptom, predict which callbacks fire, then open the answer. Notice that every fix is one of the three state tools — that is the whole point.

    🔄 Rotation wipes the form

    A user fills in a long sign-up form, rotates the phone to landscape to see more, and every field resets to empty. They are furious. What happened, and what is the smallest fix?

    Callbacks: onPause → onStop → onDestroy (isFinishing == false), then onCreate → onStart → onResume on a brand-new instance.

    Why it broke: the field values lived in remember { mutableStateOf("") }, which dies with the instance. Smallest fix: switch each to rememberSaveable { } so the value is written into the saved-instance Bundle and restored after the recreate. This is exactly the "rotation loses state" bug — and the reason rememberSaveable exists.

    💀 Process death after a phone call

    The user is on screen 3 of your app, takes a long phone call, opens two other heavy apps, then returns 20 minutes later. The app relaunches at the home screen and their work is gone — even though a ViewModel held it. Why?

    Callbacks: while backgrounded the system did a low-memory kill — no guaranteed callbacks, onDestroy() may be skipped. On return: onCreate → onStart → onResume (no onRestart()).

    Why it broke: a ViewModel survives configuration changes but not process death — when the process dies, so does it. Fix: keep the small, restorable bits (selected id, query, scroll position) in a SavedStateHandle inside the ViewModel, and keep anything durable in Room/DataStore. onCreate() receives the saved Bundle to rebuild from.

    ↩️ Returning from the background

    The user presses Home (process not killed), then re-opens from Recents. Your screen needs to re-subscribe to a sensor and refresh a possibly-stale list. Which callbacks run, and where does each job go?

    Callbacks: onRestart → onStart → onResume. onCreate() does not run — the instance survived, so your fields still hold their values.

    Where each job goes: re-acquire the sensor in onResume() (and release it in onPause() — always pair them). Refresh stale data in onStart()/onResume() so the cold-start path is covered too, not only in onRestart(). The state itself lives in a ViewModel, so it is already correct when the screen reappears.

    ℹ️
    The one decision that links all three

    Every scenario is answered by choosing the right tool on the survival ladder: remember survives recomposition · rememberSaveable adds rotation · ViewModel adds rotation + is the home for logic · SavedStateHandle adds process death · Room/DataStore adds app restart. In production you ask one question — "how long must this value outlive the screen?" — and the ladder gives you the answer. That is why hoisting state out of a composable into a ViewModel matters: it puts the value where the lifecycle can protect it.

    Common wrong mental model

    "Returning to the app always runs onRestart()." It does not after a process kill — that path runs onCreate() instead, because there is no instance left to restart. Encoding that difference correctly below is the whole skill.
