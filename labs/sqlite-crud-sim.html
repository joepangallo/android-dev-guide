@Update

@Query("UPDATE tasks SET is_done = :done WHERE id = :id")

@Update uses the entity's primary key to build UPDATE tasks SET ... WHERE id = ?. For a single column, a custom @Query is leaner.

Kotlin · DAO + ViewModel
// DAO — delete by entity (matched on primary key)...
@Delete

// ...or delete by id, or wipe the table:

}

Same pattern: a suspend DAO function, called inside viewModelScope.launch, and the observing Flow updates the list automatically.

🏗 The three Room building blocks

Room is an abstraction layer over SQLite. You describe your data in plain Kotlin with three annotations and Room generates the boilerplate (cursor handling, SQL compilation, type mapping). Below is the full, runnable definition behind the simulator's tasks table.

① @Entity = a table

One Kotlin class ⇄ one SQL table. Each val is a column; the type maps to INTEGER/TEXT/REAL/BLOB.

② @Dao = the queries

An interface of data-access methods. @Insert/@Update/@Delete are generated; @Query lets you write SQL Room verifies at compile time.

③ @Database = the file

Ties entities + DAOs to one SQLite file (.db). Usually a single instance for the whole app.

Kotlin · TaskEntity.kt — @Entity
import androidx.room.ColumnInfo
import androidx.room.Entity
import androidx.room.PrimaryKey

@Entity(tableName = "tasks")
data class TaskEntity(
    @PrimaryKey(autoGenerate = true)
    val id: Int = 0,

    @ColumnInfo(name = "title")
    val title: String,

    // SQLite has no boolean type; Room stores Boolean as INTEGER 0/1
    @ColumnInfo(name = "is_done")
    val isDone: Boolean = false
)
// Generated table:
//   CREATE TABLE tasks (
//     id      INTEGER PRIMARY KEY AUTOINCREMENT NOT NULL,
//     title   TEXT NOT NULL,
//     is_done INTEGER NOT NULL DEFAULT 0
//   );

First: what a @Dao actually is

🧩
One-sentence definition

A @Dao (Data Access Object) is an interface of data-access methods — you only declare the method signatures and tag each with @Query, @Insert, @Update, or @Delete; Room's KSP processor writes the actual SQL-running implementation for you at build time.

Kotlin · TaskDao.kt — @Dao
import androidx.room.*
import kotlinx.coroutines.flow.Flow

@Dao
interface TaskDao {

    // READ — a Flow re-emits whenever the table changes (observable)
    @Query("SELECT * FROM tasks ORDER BY id ASC")
    fun getAll(): Flow<List<TaskEntity>>

    // READ one (suspend = one-shot, runs off the main thread)
    @Query("SELECT * FROM tasks WHERE id = :id")
    suspend fun getById(id: Int): TaskEntity?

    // CREATE
    @Insert(onConflict = OnConflictStrategy.IGNORE)
    suspend fun insert(task: TaskEntity): Long

    // UPDATE (matched on @PrimaryKey)
    @Update
    suspend fun update(task: TaskEntity)

    // DELETE
    @Delete
    suspend fun delete(task: TaskEntity)

    @Query("DELETE FROM tasks")
    suspend fun clearAll()
}

Reading the DAO, line by line

  • @Dao interface TaskDao { … } — it's an interface, not a class. You write no method bodies; the @Dao tag tells Room's code generator to produce a hidden TaskDao_Impl class that contains all the cursor handling and SQL execution. You never see or write that file.
  • @Query("SELECT * FROM tasks ORDER BY id ASC") — the SQL string is validated at compile time. Misspell tasks as tassks, or select a column that doesn't exist, and the build fails with a clear error — it never compiles into the app, so it can't crash a user at runtime.
  • fun getAll(): Flow<List<TaskEntity>> — an observable read. Returning a Flow tells Room to re-run this query and emit a fresh list every time the tasks table changes. No suspend needed: you collect the stream, you don't call it once.
  • @Query("SELECT * FROM tasks WHERE id = :id") suspend fun getById(id: Int): TaskEntity? — a one-shot read. :id binds the Kotlin parameter into the SQL (Room checks the name matches). suspend means it runs once, off the main thread, and returns a single value.
  • @Insert / @Update / @Delete … suspend fun … — the three write verbs. You don't write their SQL at all; Room derives INSERT/UPDATE/DELETE from the entity's columns and primary key. They're suspend because a write is a one-shot action that must not block the UI thread.

The split is the whole point: reads you want to watch return Flow; one-shot reads and all writes are suspend.

Stripped of everything else, that's just two method shapes on one interface:

Kotlin · the two DAO shapes, isolated
@Dao
interface NoteDao {
    // observable read — re-emits on every table change:
    @Query("SELECT * FROM notes")
    fun watchAll(): Flow<List<Note>>

    // one-shot write — runs once, off the main thread:
    @Insert
    suspend fun add(note: Note)
}
⚠️
Common confusion

You are not writing the code that runs the SQL, and you're not implementing the interface — Room/KSP generates both at build time. And unlike a raw db.rawQuery("SELCT * …") string (which only blows up when that line executes on a device), a @Query string is checked when you compile: a typo'd table or column is a build error you catch immediately, not a runtime crash a user hits.

First: why the database is a thread-safe singleton

🧩
One-sentence definition

This is the thread-safe singleton idiom — the same double-checked-locking pattern from Session 9 — that guarantees the whole app builds and shares exactly one AppDatabase, because opening a database is expensive and two instances on the same file cause conflicts.

Kotlin · AppDatabase.kt — @Database (singleton)
import android.content.Context
import androidx.room.Database
import androidx.room.Room
import androidx.room.RoomDatabase

@Database(entities = [TaskEntity::class], version = 1, exportSchema = false)
abstract class AppDatabase : RoomDatabase() {

    abstract fun taskDao(): TaskDao

    companion object {
        @Volatile private var INSTANCE: AppDatabase? = null

        fun getDatabase(context: Context): AppDatabase =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: Room.databaseBuilder(   // double-check inside the lock
                    context.applicationContext,
                    AppDatabase::class.java,
                    "app_database.db"      // <-- the SQLite file on disk
                )
                    // For class demos only; real apps add a Migration instead:
                    .fallbackToDestructiveMigration()
                    .build()
                    .also { INSTANCE = it }
            }
    }
}

The getDatabase idiom, chunk by chunk

  • @Volatile private var INSTANCE: AppDatabase? = null — the one cached instance. @Volatile means every thread sees writes to INSTANCE immediately (no per-CPU cached copy), so a database one thread just built is visible to all the others.
  • INSTANCE ?: synchronized(this) { … } — the elvis operator ?: returns the cached INSTANCE if it's already non-null, skipping the lock entirely. This is the fast path that runs almost every time after the first call.
  • synchronized(this) { … } — only reached when INSTANCE is still null. The lock lets just one thread at a time enter and build, so two threads racing on first launch can't each create a database.
  • (the second read of INSTANCE inside the lock) — this is the double-check: a thread that was waiting at the lock re-reads INSTANCE after acquiring it, in case the thread ahead of it already built the database while it waited.
  • Room.databaseBuilder(…).build() — actually opens the SQLite file and returns the AppDatabase. You call this builder; you never call the AppDatabase constructor yourself (it's abstract — Room supplies the implementation).
  • .also { INSTANCE = it } — caches the freshly built database into INSTANCE and returns that same object, so every later call takes the fast path above.

Same five moves as the Session 9 singleton: @Volatile for visibility → elvis returns the cache → synchronized lets one thread build → inner re-check is the double-check → .also caches and returns.

The pattern is independent of Room — here it is guarding any expensive object:

Kotlin · double-checked locking, isolated
@Volatile private var INSTANCE: Engine? = null

fun get(): Engine =
    INSTANCE ?: synchronized(this) {     // fast path returns the cache
        INSTANCE ?: buildExpensiveEngine() // one thread builds…
            .also { INSTANCE = it }        // …caches + returns it
    }
ℹ️
Common confusion

You never call AppDatabase() directly — build() is the only way in, and you funnel every caller through getDatabase() so they share the one instance. The inner INSTANCE ?: looks redundant next to the outer one, but it isn't: the outer check runs without the lock (fast), the inner check runs with it (safe). Drop either and you either lose thread-safety or pay the lock cost on every single call.

⚠️
Bump the version when the schema changes

Add or rename a column and you must increase version and supply a Migration, or Room throws IllegalStateException: Room cannot verify the data integrity. fallbackToDestructiveMigration() just drops and recreates the table — fine for a classroom demo, data loss in production.

⚙️ Project setup (KSP + dependencies)

Room uses an annotation processor to generate the implementation of your DAO and database. Current Android projects use KSP (Kotlin Symbol Processing), not the older kapt.

Kotlin DSL · build.gradle.kts (module)
plugins {
    id("com.android.application")
    id("org.jetbrains.kotlin.android")
    // KSP version = <Kotlin version>-<KSP release>. The prefix MUST match your Kotlin.
    id("com.google.devtools.ksp") version "2.2.20-2.0.2"   // Room's code generator
}

dependencies {
    val roomVersion = "2.8.4"
    implementation("androidx.room:room-runtime:$roomVersion")
    implementation("androidx.room:room-ktx:$roomVersion")   // coroutines + Flow support
    ksp("androidx.room:room-compiler:$roomVersion")         // generates DAO/DB impl

    // Compose + lifecycle (for collectAsStateWithLifecycle, viewModelScope)
    implementation("androidx.lifecycle:lifecycle-viewmodel-compose:2.9.4")
    implementation("androidx.lifecycle:lifecycle-runtime-compose:2.9.4")
}
🔢
The one version rule that trips everyone up

The KSP plugin version is two parts glued together: your Kotlin version, then KSP's own release. With Kotlin 2.2.20 you must use a KSP whose prefix is 2.2.20-… (here 2.2.20-2.0.2). Bump Kotlin and forget to bump KSP and the build fails with "ksp-X was compiled against a different Kotlin version". Room itself (2.8.x) tracks Kotlin separately — keep it current, but it does not have to match the Kotlin number.

ℹ️
Why suspend / Flow?

Database I/O is slow and must not run on the main thread — doing so freezes the UI and can crash with IllegalStateException: Cannot access database on the main thread. Marking DAO writes suspend lets Room run them on a background dispatcher from a coroutine; marking reads as Flow gives you an observable query that auto-updates Compose.

🧱 Wiring it into Compose (clean architecture)

This is the layering the course project requires: UI → ViewModel → Repository → DAO → SQLite. Compose never touches the database directly; it only reads state and sends events up.

Kotlin · TaskRepository.kt
class TaskRepository(private val dao: TaskDao) {
    fun observeTasks(): Flow<List<TaskEntity>> = dao.getAll()
    suspend fun add(title: String, done: Boolean) =
        dao.insert(TaskEntity(title = title, isDone = done))
    suspend fun update(task: TaskEntity) = dao.update(task)
    suspend fun delete(task: TaskEntity) = dao.delete(task)
    suspend fun clear() = dao.clearAll()
}
Kotlin · TaskViewModel.kt
class TaskViewModel(private val repo: TaskRepository) : ViewModel() {

    val tasks: StateFlow<List<TaskEntity>> =
        repo.observeTasks()
            .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())

    fun add(title: String, done: Boolean) = viewModelScope.launch {
        if (title.isNotBlank()) repo.add(title.trim(), done)
    }
    fun toggleDone(task: TaskEntity) = viewModelScope.launch {
        repo.update(task.copy(isDone = !task.isDone))
    }
    fun delete(task: TaskEntity) = viewModelScope.launch { repo.delete(task) }
}

The whole loop, in one sentence

User taps Add → ViewModel launches a coroutine → Repository calls the suspend DAO insert → SQLite changes → the Flow re-emits → collectAsStateWithLifecycle updates tasks → Compose recomposes the LazyColumn. You write zero refresh code.

📋 SQL ⇄ Room cheat sheet

Every Room action maps to one of the four classic SQL verbs. Memorize the mapping; the simulator's log uses these exact forms.

SQL · what @Insert runs
-- One-time: Room creates the table from your @Entity
CREATE TABLE IF NOT EXISTS tasks (
    id      INTEGER PRIMARY KEY AUTOINCREMENT NOT NULL,
    title   TEXT NOT NULL,
    is_done INTEGER NOT NULL DEFAULT 0
);

-- Each insert (id omitted -> AUTOINCREMENT assigns it):
INSERT INTO tasks (title, is_done) VALUES ('Finish Unit 6 reading', 0);
SQL · what @Query SELECT runs
-- Read everything (the default list):
SELECT * FROM tasks ORDER BY id ASC;

-- Read only completed (the "Show done only" filter):
SELECT * FROM tasks WHERE is_done = 1 ORDER BY id ASC;

-- Read a single row by id:
SELECT * FROM tasks WHERE id = 3;
SQL · what @Update / @Query runs
-- Full row update (what @Update generates, keyed on PK):
UPDATE tasks SET title = 'Submit milestone', is_done = 1 WHERE id = 2;

-- Targeted single-column update (custom @Query):
UPDATE tasks SET is_done = 1 WHERE id = 2;
SQL · what @Delete / @Query runs
-- Delete one row (what @Delete generates, keyed on PK):
DELETE FROM tasks WHERE id = 4;

-- Wipe the whole table (custom @Query clearAll()):
DELETE FROM tasks;

⚖️ DataStore (key-value) vs Room (structured)

Both are official persistence APIs, but they solve different problems. Choosing wrong is the most common Session 9 mistake — picking the right tool is itself a CLO 4 learning goal.

Rule of thumb: a handful of settings → DataStore. A growing collection of records you query/sort/filter → Room.
ConcernPreferences DataStoreRoom (over SQLite)
Shape of dataKey → value pairs (flat)Rows in typed tables with relationships
Best forSettings, flags, last-screen, onboarding-seen, theme, tokensLists/collections: tasks, notes, contacts, cached API results
QueryingRead the whole value by key — no filtering/sortingFull SQL: WHERE, ORDER BY, JOIN, aggregates
API styleFlow + suspend edit; no schema@Entity/@Dao/@Database; compile-checked SQL
ReplacesSharedPreferences (async, type-safe, no apply() pitfalls)Raw SQLiteOpenHelper + cursors
ScaleSmall (a few dozen keys)Thousands+ of rows, indexed

🐞 Common pitfalls & fixes

You called a non-suspend DAO method directly on the UI thread. Fix: make the DAO function suspend (writes) or return Flow (reads), and call it from viewModelScope.launch { ... }. Never use .allowMainThreadQueries() outside a quick test.

You changed the @Entity (added/renamed a column) without bumping version or providing a Migration. Fix for class: bump version and add .fallbackToDestructiveMigration(). Fix for production: write a real Migration(1, 2) with the ALTER TABLE statements.

Your read query is suspend (one-shot) instead of returning Flow. A one-shot query reads once and never updates. Change suspend fun getAll(): List<TaskEntity> to fun getAll(): Flow<List<TaskEntity>> and collect it with collectAsStateWithLifecycle().

SQLite only has INTEGER, TEXT, REAL, BLOB. Boolean works out of the box (stored as 0/1). For types like LocalDate or an enum, add a @TypeConverter that maps to/from one of the four primitives (e.g. store a date as an epoch Long).

Calling Room.databaseBuilder(...).build() in multiple places opens multiple files. Use the @Volatile + synchronized singleton shown above so the whole app shares one AppDatabase.

Drill 1 · Map a table to Room — then predict the SQL

You're given a plain-English spec for a new table. Before any code, decide the @Entity columns and the @Query SQL in your head — then implement the in-memory DAO so the harness confirms your mapping.

📋 The spec (a reminders app)

  • Table reminders with an auto-increment id (Int, primary key, starts at 1).
  • label: required text (NOT NULL).
  • priority: an Int, 0–2 (low/med/high), default 0.
  • is_archived: a Boolean stored as INTEGER 0/1, default false.
  • An "active list" read = all non-archived rows, highest priority first, then id ascending.
✍️
Predict first (retrieval, not recognition)

On paper, write (1) the @Entity class with its @ColumnInfo names and @PrimaryKey(autoGenerate = true), and (2) the exact SELECT … WHERE … ORDER BY … for the active list. Then open the panel below to check yourself — recalling beats re-reading.

Kotlin · ReminderEntity.kt — the mapping you should have written
@Entity(tableName = "reminders")
data class ReminderEntity(
    @PrimaryKey(autoGenerate = true)
    val id: Int = 0,

    @ColumnInfo(name = "label")
    val label: String,

    @ColumnInfo(name = "priority")
    val priority: Int = 0,            // 0 = low, 1 = med, 2 = high

    // SQLite has no boolean — Room stores it as INTEGER 0/1
    @ColumnInfo(name = "is_archived")

)
SQL · the active-list @Query Room compiles
-- @Query("SELECT * FROM reminders WHERE is_archived = 0 ORDER BY priority DESC, id ASC")
SELECT * FROM reminders
WHERE is_archived = 0
ORDER BY priority DESC, id ASC;

Note priority DESC (highest first) then id ASC as the tie-breaker — a stable order matters once you key a LazyColumn on id. The Boolean compares against 0, not false, because the on-disk type is INTEGER.

  • insert: val r = Reminder(nextId++, label, priority); rows += r; return r.id — increment after reading the id, just like AUTOINCREMENT.
  • archive: find the index with rows.indexOfFirst { it.id == id }; if >= 0, rows[i] = rows[i].copy(isArchived = true). The copy mirrors @Update writing a new row image.
  • activeList: rows.filter { !it.isArchived }.sortedWith(compareByDescending<Reminder> { it.priority }.thenBy { it.id }) — that compareByDescending(...).thenBy(...) is exactly ORDER BY priority DESC, id ASC.

Drill 3 · Flow vs suspend — which read do you reach for?

Both are correct tools; choosing wrong is the most common Room mistake. The rule: a screen that must stay live as the table changes needs an observable Flow read; a one-time fetch (pre-fill an edit form, a quick existence check) is a one-shot suspend read. Reason through the cases below, then prove it in code.

If the UI must react to later writes → Flow. If you need the value once, right now → suspend.
What the screen needsRight readReturn type
The whole task list, always reflecting inserts/deletesFlow (observable)Flow<List<TaskEntity>>
Load one row to pre-fill an "edit task" form, then leavesuspend (one-shot)suspend fun getById(id): TaskEntity?
A live "X items remaining" badge in the top barFlow (observable)Flow<Int>
"Does a task with this title already exist?" before insertingsuspend (one-shot)suspend fun countByTitle(t): Int
⚠️
The classic symptom of the wrong choice

If you use a one-shot suspend fun getAll(): List for the main list, the screen renders once and then goes stale: you add a task, the row is genuinely in SQLite (you can see it in the Database Inspector), but the LazyColumn won't change until you navigate away and back. A Flow read fixes it because Room re-runs the query and pushes a new list on every write.

Run this side-by-side model. A one-shot snapshot (suspend) is taken before a write and never changes; an observable read (Flow) sees the write. Predict each printed line, then Run.

Kotlin · runnable — watch one-shot go stale, Flow stay live
import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

fun main() = runBlocking {
    val table = MutableStateFlow(listOf("a", "b"))   // the "tasks" table

    // ONE-SHOT (suspend): read once, into a plain value — frozen forever.
    val oneShot: List<String> = table.value
    println("one-shot read at start: $oneShot")

    // OBSERVABLE (Flow): collect emissions as the table changes.
    val seen = mutableListOf<List<String>>()
    val job = launch { table.collect { seen += it } }
    yield()

    // a write happens (INSERT):
    table.value = table.value + "c"
    yield()

    println("one-shot AFTER insert (stale): $oneShot")          // still [a, b]
    println("Flow AFTER insert (live):      ${seen.last()}")    // [a, b, c]
    println("Flow emission count:           ${seen.size}")      // 2
    job.cancel()
}

The one-shot value is captured by reference to a snapshot — later writes can't touch it. The Flow is wired to the source, so it re-emits. That difference is suspend fun getAll(): List vs fun getAll(): Flow<List<…>> in your DAO.

Drill 4 · Store a LocalDate — write the @TypeConverter

SQLite has exactly four storage classes — INTEGER, TEXT, REAL, BLOB. Try to put a LocalDate (or an enum) straight on an @Entity and Room stops the build with "Cannot figure out how to save this field into database." The fix is a type converter: a pair of functions that map your type to/from one of those four primitives. Here we'll store a date as the Long count of days since 1970-01-01 (its epoch day).

📋 The contract

  • A dueDate: LocalDate? column must round-trip through a Long? with no data loss.
  • null in → null out (an optional due date is legal).
  • The converters are wired onto the database with @TypeConverters(Converters::class); Room then calls them automatically for every read and write of that column.
Kotlin · Converters.kt + the @Database wiring
import androidx.room.TypeConverter
import androidx.room.TypeConverters
import java.time.LocalDate

class Converters {
    // LocalDate -> the INTEGER column (epoch day)
    @TypeConverter
    fun toEpochDay(date: LocalDate?): Long? = date?.toEpochDay()

    // the INTEGER column -> LocalDate
    @TypeConverter
    fun fromEpochDay(value: Long?): LocalDate? = value?.let { LocalDate.ofEpochDay(it) }
}

@Database(entities = [TaskEntity::class], version = 2, exportSchema = false)
@TypeConverters(Converters::class)      // <-- register; Room calls them per read/write
abstract class AppDatabase : RoomDatabase() { abstract fun taskDao(): TaskDao }

// Now this compiles — the dueDate column is stored as INTEGER (epoch day):
@Entity(tableName = "tasks")
data class TaskEntity(
    @PrimaryKey(autoGenerate = true) val id: Int = 0,
    val title: String,
    @ColumnInfo(name = "due_date") val dueDate: LocalDate? = null   // needs the converter
)

Adding a column changed the schema, so version went 1 → 2 (and you'd supply a Migration(1, 2) in production). The converter pair is the only "extra" code; everything else is the same @Entity/@Dao you already know.
