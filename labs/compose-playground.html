    The four ideas behind every Compose screen

    Column

    Stacks children top→bottom. Main axis = vertical. Use verticalArrangement + horizontalAlignment.

    Row

    Lays children left→right. Main axis = horizontal. Use horizontalArrangement + verticalAlignment.

    Box

    Stacks children on the z-axis (overlapping). Position with contentAlignment and per-child Modifier.align().

    Kotlin 
    // The three layout primitives — pick by axis
Column { /* children flow vertically  */ }
Row    { /* children flow horizontally */ }
Box    { /* children stack / overlap   */ }

    Two different jobs that students constantly confuse:

    • Arrangement distributes children along the main axis (the direction the container flows). Think: spacing between and around children — spacedBy, SpaceBetween, Center.
    • Alignment positions children along the cross axis (perpendicular). Think: left / center / right (for a Column) or top / center / bottom (for a Row).
    ⚠️
    The flip students forget

    In a Column the main axis is vertical → verticalArrangement. In a Row it is horizontal → horizontalArrangement. Switch the container in the playground and watch the property name change.

    Kotlin 
    Column(
    verticalArrangement = Arrangement.spacedBy(8.dp), // main axis
    horizontalAlignment = Alignment.CenterHorizontally // cross axis
) { /* ... */ }

Row(
    horizontalArrangement = Arrangement.SpaceBetween,  // main axis
    verticalAlignment = Alignment.CenterVertically     // cross axis
) { /* ... */ }

    A Modifier is an ordered, chained description of size, layout, drawing, and behavior. Order matters because each modifier wraps the next.

    padding → background 
    Text(
    "Hi",
    modifier = Modifier
        .padding(16.dp)   // space reserved FIRST
        .background(Green) // green fills the INNER box
)
// → green patch is SMALL (inside the padding)
    background → padding 
    Text(
    "Hi",
    modifier = Modifier
        .background(Green) // green fills the OUTER box
        .padding(16.dp)    // padding is INSIDE the green
)
// → green patch is LARGE (padding is inside it)

    Compose is declarative: the UI is a function of state. You don't mutate views — you change state and Compose recomposes (re-runs the affected functions) to match.

    Kotlin 
    @Composable
fun Counter() {
    // remember survives recomposition; mutableStateOf is observable
    var count by remember { mutableStateOf(0) }

    Button(onClick = { count++ }) {
        Text("Clicked $count times")
    }
    // tapping changes `count` → Compose re-runs Counter() → UI updates
}
    ⚡
    For an Int, prefer mutableIntStateOf

    mutableStateOf(0) is the right baseline to learn, but it boxes the Int into a State<Int> object. For a primitive counter, Compose ships specialized, autoboxing-free variants: var count by remember { mutableIntStateOf(0) } (also mutableLongStateOf, mutableFloatStateOf, mutableDoubleStateOf). Reach for them for hot, frequently-updated primitive state; mutableStateOf stays correct for everything else.

    ⚠️
    remember vs rememberSaveable

    remember survives recomposition but is lost on configuration change (e.g. rotation). Use rememberSaveable to also survive that, and hoist state to a ViewModel for anything that should outlive the screen.

    Stateful counter. Real remember { mutableStateOf(...) } — edit it, add a row, change the theme, and re-run.

    Kotlin + Compose · runnable
    import androidx.compose.foundation.layout.*
import androidx.compose.material3.*
import androidx.compose.runtime.*
import androidx.compose.ui.Alignment
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp
import androidx.compose.ui.window.ComposeViewport
import kotlinx.browser.document

fun main() {
    ComposeViewport(document.body!!) { App() }
}

@Composable
fun App() {
    MaterialTheme {
        var count by remember { mutableIntStateOf(0) } // primitive Int: autoboxing-free
        Column(
            Modifier.fillMaxWidth().padding(24.dp),
            horizontalAlignment = Alignment.CenterHorizontally
        ) {
            Text("Count: $count", style = MaterialTheme.typography.headlineMedium)
            Spacer(Modifier.height(16.dp))
            Row {
                Button(onClick = { count-- }) { Text("Minus") }
                Spacer(Modifier.width(12.dp))
                Button(onClick = { count++ }) { Text("Plus") }
            }
        }
    }
}

    Guard the tiny cases first: if widths.size <= 1, the first child is at 0 (or the list is empty). Otherwise compute val gap = (total - widths.sum()) / (widths.size - 1). Walk the list keeping a running cursor: each child's offset is the cursor; then advance the cursor by width + gap for the next one.

    Kotlin
    fun spaceBetweenOffsets(total: Int, widths: List<Int>): List<Int> {
    if (widths.isEmpty()) return emptyList()
    if (widths.size == 1) return listOf(0)
    val gap = (total - widths.sum()) / (widths.size - 1)
    val out = mutableListOf<Int>()
    var cursor = 0
    for (w in widths) {
        out.add(cursor)
        cursor += w + gap
    }
    return out
}

    This is exactly the math the Compose layout pass runs for Arrangement.SpaceBetween. SpaceAround and SpaceEvenly only differ in where the leftover gaps go — try modelling those next.

    Drill 4 — The Modifier-order pitfall (size ⇄ padding)

    ⚠️
    Wrong mental model: "Modifiers are a settings bag — order doesn't matter."

    A Modifier chain is not a bag of independent settings. Each modifier wraps the next and constrains it. Swap two and you get a different pixel result — and in production this shows up as a tap target that's the wrong size, a ripple that bleeds past the padding, or a box that's 8 dp bigger than the designer asked for. The simulator's padding ⇄ background swap makes the colored patch grow or shrink for the same reason.

    size THEN padding 
    Box(
    Modifier
        .size(100.dp)     // box is fixed at 100×100
        .padding(16.dp)   // padding eats INTO the 100
)
// → outer footprint = 100.dp
//   inner content   = 68.dp  (100 − 16 − 16)
    padding THEN size 
    Box(
    Modifier
        .padding(16.dp)   // 16 reserved AROUND the box
        .size(100.dp)     // box is 100×100 INSIDE that
)
// → outer footprint = 132.dp  (100 + 16 + 16)
//   inner content   = 100.dp
    🏭
    In production

    Same numbers, two different layouts. The rule: a modifier applies to everything that comes after it in the chain. .size().padding() shrinks the content inside a fixed box; .padding().size() sizes the content and then surrounds it with space. Put .clickable {} before .padding() and the padding becomes a dead zone you can't tap; put it after and the padding is part of the tap target. Order is a design decision, not a style preference.

    Prove it. Model both chains. Given an inner box size and a padding, return the outer footprint the parent has to reserve for each order. Press Run until everything is green.
