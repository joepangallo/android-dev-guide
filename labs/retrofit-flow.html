🧩 The full, copy-paste pipeline in code

The same six layers as a complete, current Android example. Tabs follow the data top-to-bottom. Everything here compiles against current AndroidX + Retrofit 3.0 + kotlinx.serialization. (Retrofit 3.x is a drop-in, binary-compatible successor to the 2.x line — the API you write is identical; it just bumps its baseline Kotlin/OkHttp dependencies.)

Gradle dependencies + the INTERNET permission

Add the dependencies, the Kotlin serialization plugin, and — easy to forget — the manifest permission. Without the permission the request throws SecurityException at runtime.

kotlin · build.gradle.kts (:app) 
plugins {
    id("com.android.application")
    id("org.jetbrains.kotlin.android")
    // Compiler plugin that generates the JSON (de)serializers.
    // Pin it to your Kotlin version (here 2.2.20) — they must match.
    id("org.jetbrains.kotlin.plugin.serialization") version "2.2.20"
}

dependencies {
    // Retrofit — type-safe HTTP client (3.x is binary-compatible with 2.x)
    implementation("com.squareup.retrofit2:retrofit:3.0.0")

    // kotlinx.serialization + the Retrofit converter that bridges them
    implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.11.0")
    implementation("com.jakewharton.retrofit:retrofit2-kotlinx-serialization-converter:1.0.0")

    // OkHttp logging — print every request/response in Logcat while learning
    // (Retrofit 3 baselines on OkHttp 5, so match it here)
    implementation("com.squareup.okhttp3:logging-interceptor:5.3.0")

    // ViewModel + lifecycle-aware coroutine scope (viewModelScope)
    implementation("androidx.lifecycle:lifecycle-viewmodel-compose:2.8.7")
    implementation("androidx.lifecycle:lifecycle-runtime-compose:2.8.7")
}

⚠️
INTERNET is a "normal" permission

It is granted automatically at install time — there is no runtime permission dialog. So the only thing students must remember is to declare it in the manifest. If a call mysteriously fails with SecurityException: Permission denied (missing INTERNET permission?), this line is missing.

Layer 5 — the Kotlin data class the JSON maps onto

Each property maps to a JSON key. Use @SerialName when the JSON key is not a valid/idiomatic Kotlin name. Mark fields the server might omit as nullable with a default.

kotlin · PhotoDto.kt 
import kotlinx.serialization.SerialName
import kotlinx.serialization.Serializable

// Matches one object in the JSON array returned by GET /objects
@Serializable
data class PhotoDto(
    val id: String,
    val name: String,
    // JSON key is "data" — remap it and make it optional (may be null)
    @SerialName("data")
    val details: PhotoData? = null
)

@Serializable
data class PhotoData(
    // No @SerialName needed — the property name already matches the JSON key.
    val color: String? = null,
    val capacity: String? = null,
    val price: Double? = null
)

A common pattern is to keep the network DTO separate from the UI model and map between them, so a server change does not ripple into your Composables:

kotlin · Photo.kt (UI model + mapper) 
// Clean model the UI actually consumes
data class Photo(
    val id: String,
    val title: String,
    val subtitle: String
)

// Map the wire DTO -> UI model in one place
fun PhotoDto.toPhoto(): Photo = Photo(
    id = id,
    title = name,
    subtitle = details?.color ?: "Unknown"
)

Layer 3 — the Retrofit interface (your typed HTTP contract)

Annotations describe the HTTP request. Retrofit generates the implementation. suspend means the function can be called from a coroutine and yields without blocking the thread.

🧩
One-sentence definition

A Retrofit API is a Kotlin interface with no method bodies — the annotations (@GET, @Path, …) are the implementation, and Retrofit reads them to generate the real networking code for you at runtime.

kotlin · PhotoApiService.kt 
import retrofit2.http.GET
import retrofit2.http.Headers
import retrofit2.http.Path
import retrofit2.http.Query

interface PhotoApiService {

    // GET https://api.restful-api.dev/objects
    @Headers("Accept: application/json")
    @GET("objects")
    suspend fun getPhotos(): List<PhotoDto>

    // GET .../objects?id=3&id=5   (query parameters)
    @GET("objects")
    suspend fun getPhotosByIds(
        @Query("id") ids: List<Int>
    ): List<PhotoDto>

    // GET .../objects/{id}        (path parameter)
    @GET("objects/{id}")
    suspend fun getPhoto(
        @Path("id") id: String
    ): PhotoDto
}

Read it line by line — notice there is not a single line of HTTP code, only descriptions:

  • interface PhotoApiService — it is an interface, not a class. There are no method bodies and no = … after any function; you only describe what each call is, never how it runs.
  • @GET("objects") — the annotation that is the implementation: "this call is an HTTP GET to the objects path" (appended to the base URL you set on the Retrofit instance). Retrofit reads this and writes the request for you.
  • @Headers("Accept: application/json") — attaches a request header. The annotation family (@GET, @POST, @Path, @Query, @Body, @Headers) is the whole vocabulary for describing the request.
  • suspend fun getPhotos(): List<PhotoDto> — suspend makes the call a coroutine: it runs off the main thread and resumes with the already-parsed result. The return type is the data you want back (List<PhotoDto>) — Retrofit performs the GET, hands the JSON to the converter, and gives you Kotlin objects.
  • @Query("id") ids: List<Int> — binds a function parameter into the URL's query string, producing …/objects?id=3&id=5. Change the argument, the URL changes.
  • @GET("objects/{id}") + @Path("id") id: String — the {id} placeholder in the path is filled by the @Path parameter, so getPhoto("3") hits …/objects/3.

The smallest possible version isolates the one new idea — an interface whose annotation describes the call, with no body anywhere:

kotlin · the whole idea in three lines 
interface ApiService {
    @GET("users")                       // the annotation IS the implementation
    suspend fun getUsers(): List<User>  // no body — Retrofit writes it
}

You never call ApiService directly. You hand the interface to Retrofit and it returns the generated object that actually makes the calls — see the Retrofit instance tab, where retrofit.create(PhotoApiService::class.java) produces it.

⚠️
Common confusion

You do not write the method body, and you do not write a PhotoApiServiceImpl class or construct the interface yourself. An empty-looking interface is correct — that is the entire point. retrofit.create(PhotoApiService::class.java) returns Retrofit's generated implementation at runtime; the annotations are the only "code" you supply.

Building the Retrofit instance

One configured singleton: base URL, the JSON converter, and a logging interceptor so students can see real traffic in Logcat. Create it once and reuse it.

kotlin · NetworkModule.kt 
import com.jakewharton.retrofit2.converter.kotlinx.serialization.asConverterFactory
import kotlinx.serialization.json.Json
import okhttp3.MediaType.Companion.toMediaType
import okhttp3.OkHttpClient
import okhttp3.logging.HttpLoggingInterceptor
import retrofit2.Retrofit

object NetworkModule {

    private const val BASE_URL = "https://api.restful-api.dev/"

    // Tolerate extra JSON keys the app does not model
    private val json = Json { ignoreUnknownKeys = true }

    private val logging = HttpLoggingInterceptor().apply {
        level = HttpLoggingInterceptor.Level.BODY   // BASIC/NONE in production
    }

    private val client = OkHttpClient.Builder()
        .addInterceptor(logging)
        .build()

    private val retrofit = Retrofit.Builder()
        .baseUrl(BASE_URL)
        .client(client)
        .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
        .build()

    // Lazily created, reused everywhere
    val photoApi: PhotoApiService by lazy {
        retrofit.create(PhotoApiService::class.java)
    }
}

The repository — turn calls into a clean result

The repository is where the network exception becomes a value. Catching here keeps the ViewModel simple and is part of the project's "ViewModel + repository" architecture requirement.

kotlin · PhotoRepository.kt 
import java.io.IOException
import retrofit2.HttpException

class PhotoRepository(
    private val api: PhotoApiService = NetworkModule.photoApi
) {
    // Network work itself runs on Dispatchers.IO via Retrofit's suspend call.
    suspend fun fetchPhotos(): Result<List<Photo>> = try {
        val dtos = api.getPhotos()              // performs the HTTP GET
        Result.success(dtos.map { it.toPhoto() })
    } catch (e: IOException) {
        // No connection, timeout, DNS failure …
        Result.failure(e)
    } catch (e: HttpException) {
        // Non-2xx status code (404, 500, …)
        Result.failure(e)
    }
}

Layer 2 — the ViewModel + coroutine + UI state

A sealed interface models exactly three UI states. viewModelScope.launch runs the call off the main thread and is cancelled automatically when the screen goes away.

🧩
One-sentence definition

The _uiState / uiState pair is the same idiom from Session 8: a private MutableStateFlow only the ViewModel writes, exposed as a read-only StateFlow via .asStateFlow() that the UI only ever collects.

kotlin · PhotoViewModel.kt 
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow
import kotlinx.coroutines.launch

// Exactly the three states the UI can be in
sealed interface PhotoUiState {
    data object Loading : PhotoUiState
    data class Success(val photos: List<Photo>) : PhotoUiState
    data class Error(val message: String) : PhotoUiState
}

class PhotoViewModel(
    private val repository: PhotoRepository = PhotoRepository()
) : ViewModel() {

    private val _uiState = MutableStateFlow<PhotoUiState>(PhotoUiState.Loading)
    val uiState: StateFlow<PhotoUiState> = _uiState.asStateFlow()

    init { loadPhotos() }

    fun loadPhotos() {
        // 1) show the spinner immediately
        _uiState.value = PhotoUiState.Loading
        // 2) launch a coroutine — network runs OFF the main thread
        viewModelScope.launch {
            repository.fetchPhotos()
                .onSuccess { photos ->
                    _uiState.value = PhotoUiState.Success(photos)
                }
                .onFailure { e ->
                    _uiState.value =
                        PhotoUiState.Error(e.message ?: "Something went wrong")
                }
        }
    }
}

The two state lines are the whole pattern — read them as a pair:

  • private val _uiState = MutableStateFlow<PhotoUiState>(PhotoUiState.Loading) — the writable holder, kept private so only this ViewModel can change it (the leading _ is just a naming convention for "the private backing one"). The ViewModel writes it by setting _uiState.value = ….
  • val uiState: StateFlow<PhotoUiState> = _uiState.asStateFlow() — the public, read-only view. .asStateFlow() hands out the same stream but with the write API removed, so the UI can collect the value but can never set it.
ℹ️
Common confusion — state down, events up

The UI never writes uiState (it has no setter). To change state, the screen sends an event up — onClick = { viewModel.loadPhotos() } — and the ViewModel is the only thing that writes _uiState. Same one-directional rule as Session 8.

Layers 1 & 6 — the Compose UI that triggers and renders

collectAsStateWithLifecycle() turns the StateFlow into Compose state; a when over the sealed state renders one of three UIs. The LazyColumn only appears in the Success branch.

kotlin · PhotoScreen.kt 
import androidx.compose.foundation.layout.*
import androidx.compose.foundation.lazy.LazyColumn
import androidx.compose.foundation.lazy.items
import androidx.compose.material3.*
import androidx.compose.runtime.Composable
import androidx.compose.ui.Alignment
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp
import androidx.lifecycle.compose.collectAsStateWithLifecycle

@Composable
fun PhotoScreen(viewModel: PhotoViewModel) {
    // Observe the StateFlow as Compose state (lifecycle-aware)
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()

    Scaffold(
        topBar = {
            TopAppBar(
                title = { Text("Photo Gallery") },
                actions = {
                    // LAYER 1: the UI only ASKS the ViewModel to load
                    IconButton(onClick = { viewModel.loadPhotos() }) {
                        Icon(Icons.Default.Refresh, contentDescription = "Refresh")
                    }
                }
            )
        }
    ) { padding ->
        // LAYER 6: render exactly one UI per state
        when (val state = uiState) {
            is PhotoUiState.Loading -> LoadingView(Modifier.padding(padding))
            is PhotoUiState.Error   -> ErrorView(
                message = state.message,
                onRetry = { viewModel.loadPhotos() },
                modifier = Modifier.padding(padding)
            )
            is PhotoUiState.Success -> PhotoList(
                photos = state.photos,
                modifier = Modifier.padding(padding)
            )
        }
    }
}

@Composable
private fun PhotoList(photos: List<Photo>, modifier: Modifier = Modifier) {
    LazyColumn(modifier = modifier.fillMaxSize()) {
        items(photos, key = { it.id }) { photo ->
            ListItem(
                headlineContent = { Text(photo.title) },
                supportingContent = { Text(photo.subtitle) }
            )
            HorizontalDivider()
        }
    }
}

@Composable
private fun LoadingView(modifier: Modifier = Modifier) {
    Box(modifier.fillMaxSize(), contentAlignment = Alignment.Center) {
        CircularProgressIndicator()
    }
}

@Composable
private fun ErrorView(message: String, onRetry: () -> Unit, modifier: Modifier = Modifier) {
    Column(
        modifier.fillMaxSize().padding(24.dp),
        horizontalAlignment = Alignment.CenterHorizontally,
        verticalArrangement = Arrangement.Center
    ) {
        Text("⚠️ $message")
        Spacer(Modifier.height(12.dp))
        Button(onClick = onRetry) { Text("Retry") }
    }
}

MethodMeansRetrofit

POSTCreate a new resource (has a body)@POST + @Body

RangeFamilyExamples

In Retrofit, any non-2xx response with a suspend body call throws HttpException; no connection at all throws IOException. Catch both.

ℹ️
HTTPS, not HTTP

Since Android 9 (API 28) cleartext HTTP is blocked by default. Always use https:// base URLs. If a teaching API is HTTP-only you must opt in via a network-security-config — a good moment to discuss why plaintext traffic is unsafe.

🧵 Why network work must leave the main thread

Android draws the UI on a single main (UI) thread roughly every 16 ms (60 fps). A network call can take hundreds of milliseconds to several seconds. If you block the main thread waiting for it, the UI freezes — and Android will actually throw to stop you:

🛑
NetworkOnMainThreadException

Do a network call directly on the main thread and the framework throws this immediately. It is a guardrail, not a bug — it is telling you to move the work to a background dispatcher.

Coroutines solve this cleanly: suspend functions can pause without blocking, and viewModelScope.launch runs them on a background dispatcher while keeping the code readable top-to-bottom.

The UI thread stays free during the slow part, so the app keeps animating. The result hops back to the main thread to update state.

kotlin · what suspend buys you 
// ❌ NEVER — blocks the UI thread, throws NetworkOnMainThreadException
fun bad() {
    val photos = URL("https://api.restful-api.dev/objects").readText()
}

// ✅ Coroutine: launched off the main thread, written like straight-line code
fun PhotoViewModel.good() = viewModelScope.launch {
    val photos = repository.fetchPhotos()   // suspends here, thread is freed
    // …resumes back on the main dispatcher to update state
}

🐛 Common pitfalls & how to debug them

Tap to expand. These are the failures students hit in the first networking lab, in roughly the order they hit them.

The <uses-permission android:name="android.permission.INTERNET" /> line is missing from AndroidManifest.xml. Add it and rebuild. INTERNET is a normal permission — no runtime prompt needed.

You ran the call on the main thread. Make the API function suspend and call it inside viewModelScope.launch { … } (Retrofit dispatches suspend calls on a background dispatcher automatically). Never use runBlocking on the main thread.

Your property name does not match the JSON key. Either rename the property to match exactly, or annotate it with @SerialName("json_key"). Add Json { ignoreUnknownKeys = true } so extra server fields do not crash parsing, and mark optional fields nullable with a default (val x: String? = null).

Your base URL is http://. Use https://. Only for a throwaway local/test server should you add a network-security-config to allow cleartext — never ship it.

A non-2xx status was returned. Check the URL path, query params, and any required API key/headers. Turn on the OkHttp HttpLoggingInterceptor at Level.BODY and read the exact request and response in Logcat. 401/403 = auth, 404 = wrong URL, 429 = rate limited, 5xx = server's problem (handle gracefully, maybe retry).

You set state to Loading but a thrown exception escaped before you set Success or Error. Catch in the repository and always emit a terminal state (use Result / try-catch as shown). An uncaught exception in a coroutine leaves the UI stuck.

Drill 1 — write the data class for this JSON auto-checked

Here is a single object from the server's JSON array:

json · one element of the response 
{

Model it as Kotlin data classes. Two snags real JSON throws at you are baked in: the nested object lives under the key "data" (not a great Kotlin name), and data can be absent on some items.             id: String, name: String, hasData: Boolean,
            dataColor: String?, dataCapacity: String?
        ): PhotoDto = PhotoDto(
            id = id,
            name = name,
            details = if (hasData) PhotoData(dataColor, dataCapacity) else null
        )
    }
}

//   @Serializable
//   data class PhotoData(val color: String? = null, val capacity: String? = null)
//
//   @Serializable
//   data class PhotoDto(
//       val id: String,
//       val name: String,
//       @SerialName("data") val details: PhotoData? = null
//   )

Two production habits are encoded here: rename awkward JSON keys with @SerialName so your Kotlin stays idiomatic, and mark anything the server might omit as nullable with a default so a missing field never crashes parsing.

Drill 2 — sketch the suspend repository's result mapping auto-checked

⚠️
The mental model that bites: "any failure means no internet"

Treating every exception as "offline → retry" hammers a server that already answered with 404 or 500. The canonical split: IOException = the call never reached the server (offline, timeout, DNS) → retryable; HttpException = it reached the server, which returned a non-2xx → don't blind-retry; anything else → a generic bug. The repository turns those exceptions into a Result, and the ViewModel maps that Result into a sealed UiState.

The real suspend fun fetchPhotos(): Result<List<Photo>> does its try/catch in the repository, then the ViewModel reduces that Result into a UI state. That reduce step is the pure logic — implement it so the four UI states come out correctly, including the Empty state for a successful-but-empty list.

fun toUiState(result: Result<List<Photo>>): UiState =
    result.fold(
        onSuccess = { if (it.isEmpty()) UiState.Empty else UiState.Success(it) },
        onFailure = {
            when (it) {
                is IOException   -> UiState.Error("offline", retryable = true)
                is HttpException -> UiState.Error("server ${'$'}{it.code()}", retryable = false)
                else             -> UiState.Error("unexpected", retryable = false)
            }
        }
    )

Drill 3 — "why is this on the main thread?" debug

Read this code the way Android's runtime does. A loadPhotos() is supposed to keep the UI thread free while the network call runs. First, predict the bug; then fix the runnable model so the work is correctly marked as off-main-thread.

kotlin · the buggy ViewModel (what NOT to do) 
fun loadPhotos() {
    _uiState.value = PhotoUiState.Loading
    // The result is read IMMEDIATELY, inline, on the calling thread:
    val photos = repository.fetchPhotosBlocking()   // blocks until HTTP returns
    _uiState.value = PhotoUiState.Success(photos)
}

Now fix it. The runnable model below records which thread each step ran on. Move the slow work into the coroutine so "network" is tagged background, not main, while the state updates stay on main.

Kotlin · runnable challenge — get the network off the main thread
// A tiny model of "what ran where". launchCoroutine { } simulates
// viewModelScope.launch: whatever you do INSIDE it runs on "background".
// Anything you do OUTSIDE it runs on "main" (the UI thread).

val log = mutableListOf<Pair<String, String>>()   // step -> thread

var currentThread = "main"
fun record(step: String) { log.add(step to currentThread) }

fun launchCoroutine(block: () -> Unit) {
    val previous = currentThread
    currentThread = "background"   // coroutine body runs off the main thread
    block()
    currentThread = previous       // resume on the main thread afterwards
}

fun loadPhotos() {
    record("setLoading")           // state update — stays on main, that's fine

    // YOUR CODE — the network call is slow. Run it OFF the main thread by
    // doing it INSIDE launchCoroutine { }. Inside the coroutine, call
    // record("network") for the HTTP work, then record("setSuccess")
    // for the state update that follows it.

    record("network")              // <- BUG: this runs on main. Move it.
    record("setSuccess")           // <- and this.
}

// ---- test harness (do not edit) ----
fun main() {

    val on = log.toMap()
