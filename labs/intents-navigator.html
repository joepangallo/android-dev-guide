Explicit vs Implicit Intents Explorer

An Intent is a message that asks the Android system to do something. Pick an action, edit the data, and watch the system either go straight to a named class (explicit) or resolve which installed app can handle it via intent filters (implicit) — with the equivalent Kotlin generated live.

CLO 1 CLO 2 CLO 5 Supports Session 9 (Navigation) & Session 5 (Components) Kotlin · Android SDK

Why it matters: Intents are how an app starts its own screens and how it taps the wider device — open a webpage, dial a number, share text, drop a map pin. Understanding explicit vs implicit resolution is foundational for in-app navigation, deep links, and integrating system/cloud services later in the course.

⚠️
Always handle "no app found"

An implicit intent can fail if no installed app declares a matching intent filter (common on bare emulators with no browser, dialer, or maps app). Calling startActivity() blindly throws ActivityNotFoundException and crashes your app. Guard it — see the try/catch or resolveActivity() pattern in the generated code.

🔎 How the system "resolves" an implicit intent

When you call startActivity(intent) with an implicit intent, you do not name a class. Instead, the Android Package Manager compares your intent's action, data (URI scheme + MIME type), and categories against the <intent-filter> elements every installed app declared in its manifest. Apps that match are candidates.

Your intent

action = VIEW
data = https://...
category = DEFAULT
→

An app's intent-filter (manifest)

action VIEW
category DEFAULT, BROWSABLE
data scheme="https"

All three tests must pass for a filter to match:

  • Action test — the filter must list your action (e.g. android.intent.action.VIEW), or list none.
  • Category test — every category in your intent must appear in the filter. Implicit intents launched via startActivity() implicitly add CATEGORY_DEFAULT, so the target's filter must include it.
  • Data test — the URI scheme (https, tel, geo, mailto…), host, path, and/or MIME type must match the filter's <data> rules.
ℹ️
One match vs many

If exactly one app matches (or the user set a default), Android launches it directly. If several match, the system shows a disambiguation dialog ("Open with…"). You can force that chooser every time — useful for "Share via" — by wrapping your intent in Intent.createChooser(intent, "Share via").

💡
Rule of thumb

Staying inside your app? In modern Compose apps you usually use Navigation-Compose (routes), not explicit Intents — but explicit Intents are still correct for launching a specific Activity, Service, or another module you own. Leaving your app to use a device capability (browser, dialer, maps, share sheet, camera)? Use an implicit Intent — and always guard it.

🧠 When to reach for each — and common pitfalls

  • You are starting a screen in your own app (a settings screen, a detail Activity) and you are not using Navigation-Compose for it.
  • You are starting your own Service (e.g. a foreground download service) or sending an internal broadcast.
  • You need to pass data with putExtra() to a destination whose contract you control.
  • Security matters: an explicit intent cannot be intercepted by another app, so it is the safe choice for internal messaging.
kotlin 
// From a Composable button, navigating to your own DetailActivity
val context = LocalContext.current
Button(onClick = {
    val intent = Intent(context, DetailActivity::class.java).apply {
        putExtra("ITEM_ID", 42)
        putExtra("ITEM_TITLE", "Trail Runner")
    }
    context.startActivity(intent)
}) {
    Text("Open details")
}

// In modern Compose apps, prefer Navigation-Compose for in-app screens:
navController.navigate("detail/42")
  • You want to open a website in whatever browser the user prefers (ACTION_VIEW + https: URI).
  • You want to dial / call a number (ACTION_DIAL + tel: — no permission; or ACTION_CALL which needs CALL_PHONE).
  • You want to share text or a file to any app via the system share sheet (ACTION_SEND + a MIME type, wrapped in createChooser()).
  • You want to show a map location in any maps app (ACTION_VIEW + geo: URI).
  • You want to compose an email (ACTION_SENDTO + mailto:), pick a contact, or capture a photo.

These delegate to apps you do not own. Always assume the handler app might be missing or might return unexpected data.

  • Crash: ActivityNotFoundException — you called startActivity() on an implicit intent with no handler. Fix: wrap in try/catch, or check intent.resolveActivity(packageManager) != null first (mind the API 30+ <queries> caveat).
  • "ACTION_CALL needs a permission" — ACTION_CALL places the call immediately and requires the CALL_PHONE runtime permission. Fix: use ACTION_DIAL, which just opens the dialer pre-filled and needs no permission.
  • Share sheet not appearing — you launched ACTION_SEND directly, so a saved default skipped the chooser. Fix: wrap with Intent.createChooser(sendIntent, "Share via").
  • geo: opens nothing on emulator — a bare emulator image may have no maps app. Fix: use an emulator with Google Play / Google APIs, or guard with a fallback to https://www.google.com/maps.
  • Wrong text not shared — you set the data URI instead of EXTRA_TEXT. Fix: for ACTION_SEND, the payload goes in putExtra(Intent.EXTRA_TEXT, ...) with type = "text/plain".

So far each intent is fire-and-forget. Often you need a result back — pick a photo, take a picture, choose a contact. The modern way is the Activity Result API: rememberLauncherForActivityResult in Compose, which replaces the deprecated startActivityForResult() / onActivityResult() pair. It is type-safe (the contract declares the input and result types), survives configuration changes, and keeps the launch + callback in one place instead of a far-away override.

kotlin 
// MODERN: get a result back without startActivityForResult / onActivityResult
@Composable
fun PickPhotoButton(onPicked: (Uri?) -> Unit) {
    // GetContent() is a built-in ActivityResultContract; the lambda is the callback.
    val launcher = rememberLauncherForActivityResult(
        contract = ActivityResultContracts.GetContent()
    ) { uri: Uri? -> onPicked(uri) }   // uri is null if the user backs out

    Button(onClick = { launcher.launch("image/*") }) {   // launch the implicit pick intent
        Text("Choose a photo")
    }
}

// LEGACY (do NOT use in new code) — split across two methods, not type-safe:
//   startActivityForResult(intent, REQUEST_PICK)              // deprecated
//   override fun onActivityResult(req, res, data) { ... }     // deprecated

There are ready-made contracts for the common jobs (GetContent, TakePicture, PickContact, RequestPermission), and StartActivityForResult() for an arbitrary Intent — but you call .launch() on a launcher, never the old startActivityForResult() method.

📄 Bonus: make YOUR app a candidate (declare an intent filter)

The flip side of resolution: if you want your Activity to appear in the "Open with…" list — for example, to handle https://yourapp.example.com deep links — you declare an <intent-filter> in AndroidManifest.xml. This is the manifest your app exposes to the Package Manager.

xml 
<!-- AndroidManifest.xml — make DetailActivity handle https deep links -->
<activity
    android:name=".DetailActivity"
    android:exported="true">   <!-- exported=true is REQUIRED for filters that other apps trigger -->

    <intent-filter android:autoVerify="true">
        <action android:name="android.intent.action.VIEW" />

        <!-- BROWSABLE lets a web link launch it; DEFAULT lets startActivity() reach it -->
        <category android:name="android.intent.category.DEFAULT" />
        <category android:name="android.intent.category.BROWSABLE" />

        <data
            android:scheme="https"
            android:host="yourapp.example.com"
            android:pathPrefix="/item" />
    </intent-filter>
</activity>

Read incoming intent data in the launched Activity:

kotlin 
class DetailActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // Came from a deep link? intent.data holds the matched Uri.
        val deepLink: Uri? = intent.data
        val itemId = deepLink?.lastPathSegment       // e.g. "/item/42" -> "42"

        // Or extras passed by an explicit intent inside our own app:
        val titleFromExtra = intent.getStringExtra("ITEM_TITLE")

        setContent {
            DetailScreen(itemId = itemId, title = titleFromExtra)
        }
    }
}
⚠️
android:exported is mandatory

Since Android 12 (API 31), any component with an <intent-filter> must explicitly set android:exported="true" (reachable by other apps) or "false". Omitting it is a build error. Only export what genuinely needs to be reached from outside.

Bug 1 — the unguarded implicit intent that crashes

⚠️
Wrong mental model → why it bites in production

"An implicit intent always finds an app, so I can just call startActivity()." On a bare emulator (or a locked-down device) no app may declare a matching filter — and an unguarded launch throws ActivityNotFoundException and crashes. Model the guard: if nothing handles the action, fall back instead of blowing up.

Kotlin · runnable challenge — never crash on a missing handler
// Models startActivity() for an implicit intent: pick the handler, or a safe fallback.
val installedHandlers = mapOf(
    "VIEW" to "Browser",
    "DIAL" to "Dialer"
    // note: no app handles "SEND" on this device
)

// Return the app that launches, or — if nothing matches — the SAFE FALLBACK
// "showToast" instead of crashing (the model of a try/catch or resolveActivity guard).
fun launch(action: String): String {
    // ----- YOUR CODE -----
    // Look up `action` in installedHandlers.
    // If an app handles it, return that app's name.
    // If NOT, return "showToast" (do NOT crash / throw).
    return installedHandlers[action]!!   // BUG: !! crashes when there is no handler
    // ----- END YOUR CODE -----
}

// ---------- test harness (do not edit) ----------
fun main() {
    val cases = listOf(
        "VIEW" to "Browser",
        "DIAL" to "Dialer",
        "SEND" to "showToast"   // nothing handles SEND -> must fall back, not crash
    )
    var allPass = true
    for ((action, expected) in cases) {
        val got = try { launch(action) } catch (e: Exception) { "CRASH(${e::class.simpleName})" }
        if (got == expected) {
            println("✓ passed: $action -> $got")
        } else {
            allPass = false
            println("✗ FAILED: $action -> expected $expected, got $got")
        }
    }
    println(if (allPass) "ALL TESTS PASSED 🎉" else "Keep going — what happens when nothing handles the action?")
}

The !! operator is the bug: it throws when the lookup is null — exactly the crash an unguarded startActivity() would cause. Swap it for the Elvis operator so a missing handler becomes the fallback: installedHandlers[action] ?: "showToast". That ?: is the pure-Kotlin model of if (intent.resolveActivity(packageManager) != null) … else fallback.

Kotlin — reference
fun launch(action: String): String {
    // No handler? Fall back gracefully instead of throwing.
    return installedHandlers[action] ?: "showToast"
}

In real Compose this is the guard you wrap every implicit intent in: try { context.startActivity(intent) } catch (e: ActivityNotFoundException) { /* toast */ }. The ?: branch is the crash you're preventing.

Bug 2 — ACTION_SEND text in the wrong place

⚠️
Wrong mental model → why it bites in production

"For a share intent, the text I want to send is the intent's data." It isn't. For ACTION_SEND the payload lives in putExtra(Intent.EXTRA_TEXT, …) with type = "text/plain"; the data Uri is for file/content shares. Put the message in the wrong slot and the receiving app shares an empty or wrong payload. Build the SEND spec correctly below.

Kotlin · runnable challenge — put the payload in EXTRA_TEXT
// A tiny model of an ACTION_SEND intent: its action, MIME type, and a bag of extras.
data class SendIntent(
    val action: String,
    val type: String?,
    val extras: Map<String, String>
)

const val EXTRA_TEXT = "android.intent.extra.TEXT"

// Build a correct "share plain text" intent for the given message.
fun buildShare(message: String): SendIntent {
    // ----- YOUR CODE -----
    // The intent must have:
    //   action = "SEND"
    //   type   = "text/plain"
    //   extras = the message stored under the EXTRA_TEXT key (NOT in a data Uri)
    return SendIntent(action = "SEND", type = null, extras = emptyMap()) // BUG: no type, no payload
    // ----- END YOUR CODE -----
}

// ---------- test harness (do not edit) ----------
fun main() {
    val msg = "I shipped my first Android app! 🚀"
    val i = buildShare(msg)
    var allPass = true
    fun check(name: String, cond: Boolean, expected: String, got: String) {
        if (cond) println("✓ passed: $name")
        else { allPass = false; println("✗ FAILED: $name -> expected $expected, got $got") }
    }
    check("action is SEND", i.action == "SEND", "SEND", i.action)
    check("type is text/plain", i.type == "text/plain", "text/plain", "${i.type}")
    check("payload in EXTRA_TEXT", i.extras[EXTRA_TEXT] == msg, msg, "${i.extras[EXTRA_TEXT]}")
    println(if (allPass) "ALL TESTS PASSED 🎉" else "Keep going — where does ACTION_SEND text belong?")
}

Two fixes: set type = "text/plain" so resolution can match text apps, and store the message under the EXTRA_TEXT key: extras = mapOf(EXTRA_TEXT to message). In real Kotlin this is Intent(ACTION_SEND).apply { type = "text/plain"; putExtra(Intent.EXTRA_TEXT, message) }.

Kotlin — reference
fun buildShare(message: String): SendIntent {
    return SendIntent(
        action = "SEND",
        type = "text/plain",
        extras = mapOf(EXTRA_TEXT to message)   // payload goes in EXTRA_TEXT, not a data Uri
    )
}

Then wrap it: startActivity(Intent.createChooser(sendIntent, "Share via")) so the picker always appears.

Bug 3 — implicit where you needed explicit (in-app navigation)

⚠️
Wrong mental model → why it bites in production

"Any time I leave a screen I should declare an action and let the system resolve it." When the destination is a component you own, an implicit intent is wrong: it may resolve to another app, pop a chooser, or fail — and it's a security hole (another app could register the same action). Name your own component explicitly. Classify each route correctly below.

"Own component → explicit" isn't just style: an explicit intent stays in your sandbox and can't be intercepted. Implicit is for reaching the device (browser, dialer, maps, share) — never for your own screens.

3 · Classic Intents vs Navigation-Compose

Intents launch components (whole Activities, or other apps). Navigation-Compose swaps composable destinations inside a single Activity using string routes — no new Activity, no Intent, no system resolution. The most common real-world mistake is reaching for an Intent when a route is the right tool (and vice-versa). This table is the decision you make on almost every screen change.

Dimension Classic Intent Navigation-Compose
Unit of navigation An Activity (or another app) — a whole new screen + lifecycle. A composable destination inside one Activity's NavHost.
How you trigger it context.startActivity(Intent(context, Screen::class.java)) navController.navigate("detail/42")
Passing data putExtra(...) → read back with intent.getStringExtra(...). Typed route arguments: navigate("detail/$id"), read via navArgument / backStackEntry.
Back stack Managed by the OS task/back stack of Activities. Managed by the NavController in-process; popBackStack(), launchSingleTop, etc.
Best for Leaving your app (browser, dialer, maps, share) or launching a separate Activity/module you own. Moving between screens within one Compose app — the in-app default.
Resolution Explicit = direct; implicit = system resolves via intent filters (can fail → guard it). No system resolution — routes are looked up in your own NavHost; a bad route fails fast in dev.
ℹ️
The decision you make in production

Modern Compose apps are usually single-Activity: nearly every in-app screen change is a navController.navigate(route), not a new Activity. You still drop to a classic Intent for exactly two jobs: (1) leaving your app to use a device capability (implicit), or (2) launching a whole separate Activity/module you own when routes aren't enough (explicit). Picking the wrong tool isn't a style nit — using an Activity where a route belongs bloats your back stack and lifecycle; using a route where you needed to leave the app simply can't reach the browser or dialer.

⚠️
Misconception: "Navigation-Compose uses Intents under the hood, so it's the same thing"

It isn't. A NavController swaps composables inside one Activity — no Intent, no startActivity, no Package Manager resolution. That's why an in-app route can't "fail to resolve" the way an implicit Intent can, and why route navigation is faster and lighter than spinning up a new Activity. Intents and routes solve different problems: components/other apps vs composable screens in your app.

Here's the same "go to a detail screen" idea expressed both ways — a single-Activity Compose app that navigates by route, with a comment showing where a classic explicit Intent would be used instead. This is a reference snippet for Android Studio, not a runnable in-browser demo: Navigation-Compose ships in the Android SDK, which isn't on the live Kotlin runner's classpath. Read it, predict what each destination renders, then paste it into a real Compose project to see it work.

Kotlin · Compose — Navigation-Compose vs a classic Intent (Android Studio)
// Android imports — Navigation-Compose lives in the Android SDK (not the web runner).
import androidx.compose.foundation.layout.*
import androidx.compose.material3.MaterialTheme
import androidx.compose.material3.Button
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp
import androidx.navigation.NavType
import androidx.navigation.compose.NavHost
import androidx.navigation.compose.composable
import androidx.navigation.compose.rememberNavController
import androidx.navigation.navArgument

@Composable
fun App() {
    MaterialTheme {
        // ONE Activity. The NavController swaps composable destinations by ROUTE —
        // no Intent, no startActivity, no system resolution.
        val nav = rememberNavController()
        NavHost(navController = nav, startDestination = "list") {

            composable("list") {
                Column(Modifier.padding(16.dp)) {
                    Text("Items", style = MaterialTheme.typography.titleLarge)
                    Spacer(Modifier.height(8.dp))
                    Button(onClick = { nav.navigate("detail/42") }) {
                        Text("Open item 42 (route navigation)")
                    }
                    // Classic Intent would instead be:
                    //   context.startActivity(Intent(context, DetailActivity::class.java)
                    //       .putExtra("ITEM_ID", 42))
                    // …but that launches a whole new Activity. Inside one Compose app, prefer the route above.
                }
            }

            composable(
                route = "detail/{itemId}",
                arguments = listOf(navArgument("itemId") { type = NavType.IntType })
            ) { backStackEntry ->
                val id = backStackEntry.arguments?.getInt("itemId") ?: 0
                Column(Modifier.padding(16.dp)) {
                    Text("Detail for item #$id", style = MaterialTheme.typography.titleLarge)
                    Spacer(Modifier.height(8.dp))
                    Button(onClick = { nav.popBackStack() }) { Text("Back") }
                }
            }
        }
    }
}

Summary (handout)

Explicit intent: Intent(context, Target::class.java) — names a specific component you own; for in-app navigation. Cannot fail to resolve.

Implicit intent: Intent(ACTION_VIEW, uri) — declares an action + data; the system resolves a handler via intent filters. Can throw ActivityNotFoundException; always guard it.
