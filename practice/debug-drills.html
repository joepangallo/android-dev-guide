Debug & Code-Review Drills

Reading broken code is the skill that separates a senior engineer from a junior. Each drill below is a realistic snippet that either crashes, misbehaves at runtime, or would get flagged in review. Diagnose it first — name the bug, predict the symptom — then reveal the precise fix and the principle behind it.

How to run a drill

Resist the spoiler. The diagnosis only sticks if your brain commits to an answer first.

  1. Read the snippet and the prompt. Some are true bugs (a crash, a stale screen, a frozen UI); some are code-review smells — it compiles and runs, but you'd push back in review.
  2. Say the diagnosis out loud: name the broken line, then predict the exact symptom (crash? wrong number? UI never updates? jank?).
  3. For runnable drills, press Run to watch the bug happen (or the fix pass) on JetBrains' compiler.
  4. Reveal the diagnosis + corrected code + principle. If you were right for the right reason, move on. If not, that miss is the most valuable rep on the page.

Kotlin fundamentals

Bugs that crash or quietly compute the wrong answer before any UI is involved.

Drill 01

The confident !!

Crashnull-safety
Kotlin · runnable — predict the output, then Run
fun lookupAge(people: Map<String, Int>, name: String): Int {
    // We "know" the name is in the map, so just force it.
    return people[name]!!
}

fun main() {
    val ages = mapOf("Ada" to 36, "Linus" to 54)
    println("Ada: "   + lookupAge(ages, "Ada"))
    println("Grace: " + lookupAge(ages, "Grace"))   // not in the map…
}

In review: the author swears “the name is always there.” What is wrong, and what would you change?

Diagnosis

people["Grace"] returns null (Grace isn't in the map). The !! operator asserts non-null and, when that assertion is false, throws a NullPointerException on that exact line. The first println succeeds (Ada: 36); the second force-closes. !! doesn't make a value non-null — it just turns “might be null” into “crash if null,” usually far from where the real mistake is.

Corrected code

Kotlin · fixed — handle absence explicitly
fun lookupAge(people: Map<String, Int>, name: String): Int? =
    people[name]                       // honest: may be null

fun main() {
    val ages = mapOf("Ada" to 36, "Linus" to 54)
    // Decide what "missing" means at the call site:
    println("Ada: "   + (ages["Ada"]   ?: "unknown"))
    println("Grace: " + (ages["Grace"] ?: "unknown"))   // -> unknown, no crash
}

Principle — Treat !! as a code smell, not a tool. Reach for the safe operators instead: ?. (safe call), ?: (Elvis / default), or ?.let { }. Reserve !! for the rare case where a null is a genuine programmer bug you want to crash on — and even then, prefer requireNotNull(x) { "why" } so the failure explains itself.

Drill 02

The average that's always too low

Bugnumeric types
Kotlin · runnable — what does this print?
fun average(scores: List<Int>): Double {
    val total = scores.sum()
    return total / scores.size      // looks fine…
}

fun main() {
    val grades = listOf(90, 91)   // true average is 90.5
    println("Average = " + average(grades))
}

Predict: the function returns Double, so it should print a decimal. What actually prints, and why?

Diagnosis

total and scores.size are both Int, so total / scores.size is integer division — it happens before anything is widened to Double. For listOf(90, 91): 181 / 2 = 90 (the remainder is thrown away), and then that 90 is widened to 90.0 — so Run prints Average = 90.0 instead of the true 90.5. The Double return type only widens the already-truncated Int; it cannot recover the lost remainder. Nasty inputs hide it (listOf(90, 92) → 182 / 2 = 91, which happens to be exact), which is exactly why this ships unnoticed — a wrong answer, not a crash, which is worse.

Corrected code

Kotlin · fixed — widen before dividing
fun average(scores: List<Int>): Double {
    if (scores.isEmpty()) return 0.0        // also: don't divide by zero
    return scores.sum().toDouble() / scores.size
}

fun main() {
    println(average(listOf(90, 91)))   // 90.5, correct
}

Principle — In Kotlin, Int / Int is integer division and the operand types decide the math, not the return type. Convert before the operation (.toDouble() on one side is enough). And any division needs a zero-denominator guard — an empty list here would throw ArithmeticException.

Drill 03

val ≠ immutable

Review smellimmutability
Kotlin · review this
class Cart {
    val items = mutableListOf<String>()   // exposed straight to callers

    fun add(item: String) { items.add(item) }
}

// Elsewhere, a teammate writes:
val cart = Cart()
cart.add("Cable")
cart.items.clear()        // …compiles. Wipes the cart from the outside.
cart.items[0] = "hacked"  // …also compiles, if non-empty

In review: items is a val, so it's safe to expose — true or false? What would you change?

Diagnosis

False. val only freezes the reference — you can't reassign items to a new list, but the MutableList it points to is fully editable. Exposing a mutableListOf publicly lets any caller clear(), reassign elements, or reorder your internal state, bypassing every invariant Cart tries to enforce. It compiles and runs; it's a design defect a reviewer should block.

Corrected code

Kotlin · fixed — encapsulate, expose read-only
class Cart {
    private val _items = mutableListOf<String>()   // private, mutable
    val items: List<String> get() = _items          // public, read-only view

    fun add(item: String) { _items.add(item) }
    fun clear() { _items.clear() }                   // mutation only through the API
}

val cart = Cart()
cart.add("Cable")
// cart.items.clear()   // ❌ won't compile — List has no clear()

Principle — val controls the binding, not the object. Keep mutable collections private (the _name backing-field convention) and expose an immutable type (List, Set, Map) so all changes funnel through methods you control. This same pattern reappears in Compose (private val _uiState; val uiState) and in the repository layer below.

Drill 04

The when that compiles today, breaks tomorrow

Review smellsealed / when
Kotlin · review this
sealed interface Payment {
    data class Card(val last4: String) : Payment
    data object Cash : Payment
    // (next sprint: data object ApplePay : Payment)
}

fun label(p: Payment): String {
    when (p) {                       // used as a STATEMENT, not an expression
        is Payment.Card -> return "Card ****" + p.last4
        Payment.Cash    -> return "Cash"
        else            -> return "Unknown"   // catch-all
    }
}

In review: there's an else, so every case is handled — what's the latent bug, and what would you change?

Diagnosis

The else is exactly the problem. When you add ApplePay next sprint, this when silently routes it to "Unknown" — no compile error, no warning, a wrong label shipped to users. Because when is used as a statement here (with early returns), the compiler doesn't enforce exhaustiveness, and the catch-all hides the gap. A reviewer wants the compiler to fail loudly the day a new variant appears.

Corrected code

Kotlin · fixed — exhaustive expression, no else
fun label(p: Payment): String = when (p) {   // EXPRESSION: must be exhaustive
    is Payment.Card -> "Card ****" + p.last4
    Payment.Cash    -> "Cash"
    // No else. Add ApplePay -> the compiler errors here until you handle it.
}

Principle — Over a sealed type, write when as an expression (it returns a value) and omit else. That makes the compiler require exhaustiveness, turning “forgot a case” from a silent runtime bug into a build error. Reserve else for genuinely open inputs (an arbitrary Int, a String) where new cases can't be enumerated.

Compose & state

The recomposition model is unforgiving: a plain variable, a wrong modifier order, or a hardcoded color all read as “works on my machine” until they don't.

Drill 05

The counter that snaps back to 0

Bugstateremember
Compose · what happens when you tap +1?
@Composable
fun Counter() {
    var count = 0                       // a plain local variable

    Button(onClick = { count++ }) {     // increments… then vanishes
        Text("Count: $count")
    }
}

Predict: you tap +1 five times. What does the button show, and why?

Diagnosis

The button keeps showing Count: 0. Two failures compound here. (1) count is a plain var, so Compose doesn't observe it — mutating it never schedules a recomposition. (2) Even ignoring that, var count = 0 is re-initialized to 0 on every recomposition, because the function body runs again top-to-bottom each time. So even if something else triggered a recompose, your increment would be thrown away. State that must survive recomposition has to live in remember, and it has to be a type Compose watches.

Corrected code

Compose · fixed — observable, remembered state
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }   // observed + survives recompose

    Button(onClick = { count++ }) {
        Text("Count: $count")
    }
}

Principle — Compose only reacts to reads of snapshot state (mutableStateOf, StateFlow, etc.), and only remember preserves a value across recompositions. The combo var x by remember { mutableStateOf(...) } gives you both: an observable value that triggers recomposition and survives one. (To also survive configuration changes like rotation, reach for rememberSaveable — see Drill 09.)

Drill 06

The padding that does nothing

Bugmodifiers
Compose · the tap target feels wrong
// Intent: a green box with 16dp of inner breathing room,
// where tapping anywhere INSIDE the padded box registers.
Box(
    Modifier
        .padding(16.dp)               // ① pad first
        .background(Color.Green)      // ② then color
        .clickable { onTap() }        // ③ then make it clickable
)

Diagnose: the box renders, but the padding looks like a transparent gap around the green, and taps in that gap miss. Which line order is wrong?

Diagnosis

Modifiers apply in order, outside-in. Here padding runs before background and clickable, so the 16dp is added outside the colored, clickable region: you get a transparent margin around a smaller green box, and that margin isn't part of the click target. The intent — colored area first, padding inside it, ripple covering the whole thing — requires the opposite order. This compiles and renders; it's a behavior bug you only catch by looking (or tapping).

Corrected code

Compose · fixed — order encodes layout intent
Box(
    Modifier
        .background(Color.Green)      // ① paint the area
        .clickable { onTap() }        // ② whole area is tappable (ripple included)
        .padding(16.dp)               // ③ inset the CONTENT, inside the box
)

Principle — Modifier order is not cosmetic — each modifier wraps the next, so the chain reads outside-in. Rule of thumb: size/background/clickable come before the padding that should sit inside them; a padding placed before clickable shrinks the touch target. When spacing or hit areas look off, re-read the modifier chain top-to-bottom as a sequence of wrappers.

Drill 07

Invisible text in dark mode

Review smelltheming
Compose · review this
@Composable
fun PriceTag(price: String) {
    Surface {                                  // surface follows the theme…
        Text(
            text = price,
            color = Color(0xFF222222),         // …but the text is hardcoded near-black
            modifier = Modifier.padding(12.dp)
        )
    }
}

In review: it looks perfect in the light-theme preview. Why will you still flag it?

Diagnosis

The Surface background flips to near-black in dark mode, but the hardcoded Color(0xFF222222) text does not — so the price becomes near-black on near-black: invisible. Hardcoding hex colors opts a composable out of the Material theme entirely. It also breaks dynamic color, high-contrast accessibility settings, and any future re-brand. It renders fine in one theme, which is exactly why it slips through.

Corrected code

Compose · fixed — read from the colorScheme
@Composable
fun PriceTag(price: String) {
    Surface {
        Text(
            text = price,
            // Semantic role, not a literal: contrasts with the surface in BOTH themes.
            color = MaterialTheme.colorScheme.onSurface,
            modifier = Modifier.padding(12.dp)
        )
    }
}

Principle — Pull colors from MaterialTheme.colorScheme by their semantic role (primary, onSurface, error…), never as raw hex in a composable. The onX roles are guaranteed to contrast with their X background across light, dark, and dynamic themes. Define brand colors once in the theme; consume them by name everywhere else.

Drill 08

Business logic squatting in a composable

Review smellarchitecture
Compose · review this
@Composable
fun InvoiceScreen(rawJson: String) {
    // Parse + tax math + currency formatting, all inside the UI:
    val lines = Json.decodeFromString<List<Line>>(rawJson)   // runs EVERY recompose
    val subtotal = lines.sumOf { it.qty * it.unitCents }
    val tax = (subtotal * 0.07).toInt()
    val total = "$" + ((subtotal + tax) / 100.0)

    Text("Total: $total")
}

In review: it produces the right total. Name two things you'd push back on.

Diagnosis

(1) Wrong layer. JSON parsing, tax rules, and currency math are business logic, not UI. Buried in a composable they can't be unit-tested without a UI test, can't be reused, and tangle presentation with policy. (2) Wrong frequency. Everything in a composable body re-runs on every recomposition, so this re-parses the JSON and re-runs the tax math on each frame that touches this composable — wasted work that can cause jank. The tax rate is also a magic number (0.07) with no name.

Corrected code

Kotlin · fixed — logic in the VM, UI just renders
// --- domain / ViewModel: testable, runs once per data change ---
private const val TAX_RATE = 0.07
data class InvoiceUi(val total: String)

class InvoiceViewModel : ViewModel() {
    fun toUi(rawJson: String): InvoiceUi {
        val lines = Json.decodeFromString<List<Line>>(rawJson)
        val subtotal = lines.sumOf { it.qty * it.unitCents }
        val tax = (subtotal * TAX_RATE).toInt()
        return InvoiceUi(total = "$" + ((subtotal + tax) / 100.0))
    }
}

// --- composable: dumb, just displays the prepared state ---
@Composable
fun InvoiceScreen(ui: InvoiceUi) {
    Text("Total: ${ui.total}")
}

Principle — A composable's job is to describe UI from state, nothing else. Push parsing, calculations, and policy into the ViewModel/domain layer where they run once per data change and can be unit-tested in isolation. If you genuinely must compute a value from state inside a composable, wrap it in remember(key) { … } so it's cached instead of recomputed every recomposition — and give magic numbers like 0.07 a name (TAX_RATE).

Lifecycle & state preservation

The bugs that only appear when you rotate the device, get a phone call, or leave the screen.

Drill 09

Survives recomposition, dies on rotation

Bugconfig change
Compose · the form a user is half-way through filling
@Composable
fun SignUpForm() {
    var email by remember { mutableStateOf("") }   // typed-in email

    OutlinedTextField(
        value = email,
        onValueChange = { email = it },
        label = { Text("Email") }
    )
}

Diagnose: the user types their email, then rotates the phone to landscape. What happens, and why is remember not enough?

Diagnosis

On rotation the field goes blank — the typed email is lost. A configuration change (rotation, dark-mode toggle, language change) destroys and recreates the Activity, which tears down and rebuilds the entire composition. remember only survives recomposition, not Activity recreation, so its stored value is gone. This is the classic “works in testing, infuriates real users” bug, because nobody rotates while clicking through quickly.

Corrected code

Compose · fixed — survive config changes
@Composable
fun SignUpForm() {
    // rememberSaveable persists through Activity recreation via the saved-instance bundle
    var email by rememberSaveable { mutableStateOf("") }

    OutlinedTextField(
        value = email,
        onValueChange = { email = it },
        label = { Text("Email") }
    )
}

Principle — Three tiers of survival: a plain var survives nothing; remember survives recomposition; rememberSaveable survives configuration changes / process-death restoration too (for types the Saver supports — primitives, Strings, Parcelables). Transient UI state that the user would hate to re-enter belongs in rememberSaveable or, better for anything long-lived, in a ViewModel (which also outlives configuration changes).

Drill 10

The ViewModel that holds an Activity

Bugmemory leak
Kotlin · review this ViewModel
class ProfileViewModel(
    private val context: Context   // an Activity is passed in here
) : ViewModel() {

    fun greeting(): String =
        context.getString(R.string.welcome)   // convenient… but dangerous
}

// In the Activity:
val vm = ProfileViewModel(this)   // 'this' == the Activity

Diagnose: it compiles and the greeting works. What's the leak, and why does the ViewModel make it worse?

Diagnosis

A ViewModel deliberately outlives its Activity — that's its whole point: it survives rotation. So storing the Activity Context in it means that when the Activity is destroyed and recreated on rotation, the ViewModel still pins the old Activity in memory. The garbage collector can't reclaim it: a leaked Activity, holding its entire view tree. Rotate a few times and you've leaked several screens' worth of objects.

Corrected code

Kotlin · fixed — use the application context
// Option A: AndroidViewModel gives you a safe, long-lived application context.
class ProfileViewModel(app: Application) : AndroidViewModel(app) {
    fun greeting(): String =
        getApplication<Application>().getString(R.string.welcome)
}

// Option B (cleaner): don't put Android types in the VM at all.
// Resolve the string in the composable and pass plain data in:
//   val welcome = stringResource(R.string.welcome)

Principle — Never store an Activity/View Context in anything that outlives the Activity — ViewModels, singletons, companion objects, static fields. If a long-lived object truly needs a context, use the application context (via AndroidViewModel). Better still, keep framework types out of the ViewModel entirely and resolve resources in the UI layer, passing only plain data down.

Lists & rendering

Lazy lists punish the wrong defaults: missing keys, eager Columns, and unobserved backing lists.

Drill 11

A feed that hitches and eats memory

Review smellLazyColumn
Compose · review this feed
@Composable
fun Feed(posts: List<Post>) {                 // posts can be hundreds long
    Column(Modifier.verticalScroll(rememberScrollState())) {
        for (post in posts) {                  // builds EVERY row up front
            PostCard(post)
        }
    }
}

In review: it scrolls and shows every post. Why is it the wrong tool for a long/unbounded feed?

Diagnosis

A Column with verticalScroll composes and lays out every child immediately, on- or off-screen, and never recycles them. For a few items that's fine; for a feed of hundreds it builds hundreds of PostCards up front, holds them all in memory, and pays layout for all of them on the first frame — slow first paint and dropped frames while scrolling. Even worse: nesting this inside another vertical scroller gives it infinite height and can crash.

Corrected code

Compose · fixed — lazy, recycled, keyed
@Composable
fun Feed(posts: List<Post>) {
    LazyColumn {                               // no verticalScroll — it scrolls itself
        items(posts, key = { it.id }) { post -> // composes only visible rows + buffer
            PostCard(post)
        }
    }
}

Principle — Use LazyColumn/LazyRow for any list that is long, unbounded, or data-driven; reserve a plain Column for a small, fixed set of children (a 5-field form). A lazy list composes only what's visible, recycles as you scroll, and is its own scroll container — so you drop the verticalScroll modifier. (And always pass a stable key — see the next drill.)

Drill 12

The checkbox that jumps to the wrong row

Bugitem key
Compose · a to-do list with per-row state
@Composable
fun TodoList(todos: List<Todo>, onDelete: (Todo) -> Unit) {
    LazyColumn {
        items(todos) { todo ->                 // ← no key
            // Each row remembers its own "expanded" toggle:
            var expanded by remember { mutableStateOf(false) }
            TodoRow(todo, expanded, onToggle = { expanded = !expanded },
                    onDelete = { onDelete(todo) })
        }
    }
}

Diagnose: the user expands row 3, then deletes row 1. What goes wrong, and why?

Diagnosis

Without a key, Compose tracks items by their position in the list. Delete row 1 and everything shifts up: the remembered “expanded” state that belonged to the item at index 3 is now matched to whatever item slides into index 3. The wrong row appears expanded, scroll position can jump, and item animations mis-fire. It's subtle because a static list looks correct — the bug only surfaces on insert/remove/reorder.

Corrected code

Compose · fixed — stable identity via key
@Composable
fun TodoList(todos: List<Todo>, onDelete: (Todo) -> Unit) {
    LazyColumn {
        items(todos, key = { it.id }) { todo ->   // identity follows the data, not the index
            var expanded by rememberSaveable(todo.id) { mutableStateOf(false) }
            TodoRow(todo, expanded, onToggle = { expanded = !expanded },
                    onDelete = { onDelete(todo) })
        }
    }
}

Principle — Always give items(...) a stable, unique key (usually the entity's id) so Compose matches each row's remembered state across data changes. Keys must be unique within the list — duplicating one throws IllegalArgumentException: Key "…" was already used — and you must never key by the loop index, which defeats the entire purpose.

Drill 13

The FAB that adds nothing

Bugobservable state
Compose · tapping the FAB should add a row
@Composable
fun NoteScreen() {
    val notes = remember { mutableListOf("First note") }   // plain MutableList

    Scaffold(
        floatingActionButton = {
            FloatingActionButton(onClick = { notes.add("New note") }) { Text("+") }
        }
    ) { pad ->
        LazyColumn(Modifier.padding(pad)) {
            items(notes) { Text(it) }
        }
    }
}

Diagnose: you tap +, the list doesn't change, no crash. Where does the update get lost?

Diagnosis

remember { mutableListOf(...) } remembers a plain MutableList. Compose does not observe mutations to it, so notes.add(...) updates the backing list silently but never schedules a recomposition — the UI stays frozen at one row. (The new note is in memory; it would even appear if some other state change forced a recompose. That intermittent behavior makes it maddening to debug.) You need a list type Compose watches.

Corrected code

Compose · fixed — an observable list
@Composable
fun NoteScreen() {
    val notes = remember { mutableStateListOf("First note") }   // OBSERVED list

    Scaffold(
        floatingActionButton = {
            FloatingActionButton(onClick = { notes.add("New note") }) { Text("+") }
        }
    ) { pad ->
        LazyColumn(Modifier.padding(pad)) {
            // No key here: the list holds plain Strings and the FAB appends a
            // duplicate "New note" each tap — keying by content would throw
            // "Key … was already used" (see Drill 12). Key by a stable, unique
            // id once the notes become objects.
            items(notes) { Text(it) }
        }
    }
}

Principle — A plain mutableListOf isn't snapshot state, so mutating it never recomposes. Use mutableStateListOf() for an observable list, or hoist the data into a ViewModel as a StateFlow<List<…>> and emit a new immutable list on each change. Either way, the rule holds: recomposition is driven only by reads of observable state. (Don't add a key here — these entries aren't unique, and a duplicate content key would crash exactly as Drill 12 warns; key by a stable id only once the data has one.)

Persistence & networking

The data layer is where “it works on the emulator” meets ANRs, crashes on real devices, and reactive UIs that never refresh.

Drill 14

Room on the main thread

CrashRoomcoroutines
Kotlin · DAO + the call site
@Dao
interface NoteDao {
    @Query("SELECT * FROM notes")
    fun getAll(): List<Note>          // blocking, one-shot
}

// In a composable / on a click:
val notes = db.noteDao().getAll()    // runs on the calling (main) thread

Diagnose: two distinct problems here — one is a crash, one is why the list never live-updates. Name both.

Diagnosis

(1) Main-thread DB access. Calling a blocking, synchronous DAO query on the UI thread freezes the app while it hits disk; on a real device with real data this drops frames and triggers an ANR (“Application Not Responding”). By default Room throws IllegalStateException: Cannot access database on the main thread to stop you — a crash that's actually protecting users. (2) One-shot, not reactive. Returning a plain List<Note> reads the table once; insert a note later and the UI has no idea — it never refreshes until something re-queries manually.

Corrected code

Kotlin · fixed — off-thread + reactive
@Dao
interface NoteDao {
    // Flow: emits a fresh list automatically whenever the table changes.
    @Query("SELECT * FROM notes")
    fun observeAll(): Flow<List<Note>>

    // suspend: Room runs it off the main thread for you.
    @Insert
    suspend fun insert(note: Note)
}

// In the ViewModel — collected as state, never blocking the UI:
val notes: StateFlow<List<Note>> =
    noteDao.observeAll()
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())

Principle — Disk and network are slow and must never touch the main thread — use suspend DAO functions (Room dispatches them off-thread) and run repository work in viewModelScope / on Dispatchers.IO. For anything the UI must stay in sync with, have the DAO return Flow<…>, not a one-shot List: Room re-emits on every table change, and the UI reacts for free. One-shot reads silently go stale.

Drill 15

The entity that won't compile (or shouldn't)

CrashRoom schema
Kotlin · review this entity
@Entity(tableName = "users")
data class User(
    val id: Int,            // intended to be the unique row id…
    val name: String,
    val email: String
)
// build fails: "An entity must have at least 1 field annotated with @PrimaryKey"

Diagnose: the build error names the problem. Fix it — and explain why autoGenerate is usually what you want.

Diagnosis

Every Room @Entity must declare a primary key — the column that uniquely identifies a row — yet id here is just an ordinary field. Room refuses to generate the table and fails the build (a compile-time crash, which is the friendly kind). Even after you tag id, leaving it a caller-supplied Int invites duplicate or zero ids; you almost always want the database to mint the id for you.

Corrected code

Kotlin · fixed — keyed + auto-generated id
@Entity(tableName = "users")
data class User(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,  // DB assigns it
    val name: String,
    val email: String
)
// Insert with id = 0; Room generates and returns the real id.

Principle — A primary key is mandatory and must be unique per row. Prefer @PrimaryKey(autoGenerate = true) on a Long defaulted to 0 so the database owns id generation — no collisions, no “who picks the next id?” logic in your code. Reserve a manual key for cases where a natural, guaranteed-unique business id already exists (and even then, validate it).

Drill 16

A repository that mutates shared state

Review smelldata layer
Kotlin · review this repository
class TaskRepository {
    // A single shared, mutable list handed out to everyone:
    val tasks = mutableListOf<Task>()

    fun all(): MutableList<Task> = tasks            // returns the live list

    fun add(task: Task) { tasks.add(task) }
}

// A caller does this, "just to show completed first":
repo.all().sortByDescending { it.done }            // mutates the repo's own list!

In review: name the smell, the concrete bug it enables, and what you'd change.

Diagnosis

The repository hands out its own internal MutableList. Any caller can sortByDescending, clear(), or reassign elements and silently corrupt the single source of truth — exactly what the call site does when it “just sorts for display.” On a background thread this is also a data race. The mutable, shared, leaky state defeats the whole point of a repository: to be the controlled gateway to data. It compiles and often “works” until two parts of the app fight over the same list.

Corrected code

Kotlin · fixed — immutable boundary, copy to mutate
class TaskRepository {
    private val _tasks = MutableStateFlow<List<Task>>(emptyList())
    val tasks: StateFlow<List<Task>> = _tasks.asStateFlow()   // read-only, observable

    fun add(task: Task) {
        _tasks.update { current -> current + task }   // emit a NEW immutable list
    }
}

// Callers sort a COPY for display; the repo's state is untouched:
val sorted = repo.tasks.value.sortedByDescending { it.done }

Principle — A data layer should expose immutable snapshots (List, StateFlow<List<…>>), keep its mutable state private, and update by emitting a new list rather than editing one in place. This makes the source of truth tamper-proof, recomposition-friendly (a new reference signals change), and free of accidental cross-thread mutation. Same lesson as Drill 03 — val on a mutable collection is not encapsulation — scaled up to the architecture boundary.

Drill 17

The network screen that's silently always stale

BugRetrofitcoroutines
Kotlin · review this ViewModel network call
interface WeatherApi {
    @GET("forecast")
    suspend fun forecast(@Query("city") city: String): Forecast
}

class WeatherViewModel(private val api: WeatherApi) : ViewModel() {
    var ui by mutableStateOf<Forecast?>(null)
        private set

    fun load(city: String) {
        viewModelScope.launch {
            try {
                ui = api.forecast(city)
            } catch (e: Exception) {
                // TODO: handle later
            }                                  // ← every failure vanishes here
        }
    }
}

In review: the happy path works on the emulator. Name the bug the user hits in the real world, and the review smell that hides it.

Diagnosis

The threading is actually fine — a Retrofit suspend function moves itself off the main thread, and viewModelScope ties the coroutine to the ViewModel's lifetime (so it's cancelled when the screen goes away, no leak). The real problem is the empty catch (e: Exception) { }. On a dropped connection, a 500, a timeout, or a JSON parse failure, the exception is caught and thrown away: ui is never updated, no error is surfaced, nothing is logged. The screen just keeps showing whatever stale data it had (or a permanent blank), and the user has no idea anything failed. Swallowing Exception wholesale also hides CancellationException, which a coroutine must rethrow for cancellation to work.

Corrected code

Kotlin · fixed — model loading/error as state, rethrow cancellation
sealed interface WeatherUi {
    data object Loading : WeatherUi
    data class Ready(val forecast: Forecast) : WeatherUi
    data class Error(val message: String) : WeatherUi
}

class WeatherViewModel(private val api: WeatherApi) : ViewModel() {
    var ui by mutableStateOf<WeatherUi>(WeatherUi.Loading)
        private set

    fun load(city: String) {
        viewModelScope.launch {
            ui = WeatherUi.Loading
            ui = try {
                WeatherUi.Ready(api.forecast(city))
            } catch (e: IOException) {          // catch the specific failure…
                WeatherUi.Error("No connection — pull to retry.")
            } catch (e: HttpException) {
                WeatherUi.Error("Server error ${e.code()}.")
            }
            // CancellationException is NOT caught here, so it propagates and
            // cancellation still works. Never `catch (e: Exception)` blindly.
        }
    }
}

Principle — A network call has three outcomes, not one — model them as state (Loading / Ready / Error), never an empty catch. Catch the specific exceptions you can handle (IOException, HttpException) and turn them into user-visible state; an empty catch block is the loudest review smell there is. And in a coroutine, never swallow CancellationException — catching Exception broadly breaks structured-concurrency cancellation, so let it propagate.
