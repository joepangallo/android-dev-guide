The Project Ladder

Five real apps, each one harder than the last, that you build for yourself — not to hand in. Every rung forces the exact skills this course teaches: multi-screen navigation, Compose + state, local data with Room / DataStore, and live data over Retrofit through a repository. Climb them in order and the top rung is a portfolio capstone you can drop into a GitHub profile and talk about in an interview.

🎯 CLO 1–6 Kotlin · Jetpack Compose · Material 3 · Room · Retrofit

🧗 How to climb this ladder

This is your build log, kept on your own machine. Nobody scores it. The only judge that matters is the one in a code review or an interview — so each rung is written to that bar, not to a checklist of boxes.

  • Build in order. Each rung assumes the last one's skills and adds exactly one new in-scope capability — no toy apps, no busywork.
  • Use only what the course teaches. Nothing here needs a library, API, or pattern outside this syllabus. If it isn't in a session, you don't need it to finish.
  • Tick your own "Definition of Done." Each rung has a self-checked acceptance list. It saves on this device. When it's all green, the app is portfolio-ready — that's the whole point of the checklist.
  • Reach for the stretch goals only after Done is green. They're how you stand out, not how you pass.
🎯
What the ladder proves

By the capstone you'll have demonstrated all six outcomes in one codebase: the Android ecosystem & lifecycle CLO 1, Compose UI CLO 2, Kotlin CLO 3, local storage CLO 4, API integration CLO 5, and network protocols CLO 6.

Rung 1 · Foundations & state

FocusTimer — a Pomodoro study timer

Sessions 3–5 ~1 weekend CLO 2CLO 3

A focus timer that runs work/break intervals, survives rotation, and keeps a running count of sessions you've completed today.

Everyone's first "real" app is a timer because it's deceptively hard: it's pure state over time. There's no list to lean on, no network to blame — just a state machine that has to keep ticking correctly when the screen rotates, the app pauses, and the user taps reset mid-countdown. Nail this and remember / state hoisting / lifecycle stop being abstract.

⚙️ What this rung forces you to learn

🧭 Navigation

Two screens: the Timer and a Settings screen (work/break lengths). A NavHost with two routes — the minimum that makes navigation real.

🎨 Compose + state

A countdown driven by a ViewModel + StateFlow, collected with collectAsStateWithLifecycle(). The UI is a pure function of TimerUiState (running / paused / finished).

🔄 Lifecycle

The timer must keep correct time across a rotation. Holding the countdown in the ViewModel (not the Composable) is the whole lesson.

✅ Definition of done (self-checked — your call, not a grade)

↗ Stretch goals (only after Done is green)

  • Persist the work/break lengths with DataStore so Settings survive an app restart (a soft preview of Rung 3).
  • Add a subtle progress ring that animates with the countdown using a Compose animateFloatAsState.
  • Honor prefers-reduced-motion-style respect: a "no animation" toggle in Settings.
🎓 The senior-engineer bar

A junior puts the countdown loop inside the Composable and watches it reset on every rotation. A senior puts the clock in the ViewModel, exposes an immutable TimerUiState, and lets the UI be a dumb projection of it. If your timer survives rotation without a single line of onSaveInstanceState, you've understood why state lives in the ViewModel. Bonus tell of seniority: the "finished" state is its own case, not a special value of the running state.

Rung 2 · Lists, detail & navigation args

TrailLog — a hike journal (list → detail)

Sessions 4–8 ~1 week CLO 2CLO 3

A journal of hikes (or workouts, or books) shown as a scrollable list; tap one to open a full detail screen with everything about it.

The list → detail pattern is the spine of almost every app on your phone. This rung is where navigation grows up: you stop hardcoding screens and start passing an id through a route so the detail screen knows which item to show. Data is still in-memory here (a seeded list) — the point is the UI architecture, not storage. That comes next.

⚙️ What this rung forces you to learn

🧭 Navigation w/ arguments

NavHost with a "detail/{id}" route. Tapping a row navigates by route and passes the id; the detail screen reads it back. This is real navigation, not just two static screens.

🎨 Compose lists + state

A LazyColumn of Material 3 Cards, an empty-state when there are no entries, and an "add entry" form whose input is hoisted state. One ViewModel owns the list.

🏛️ Architecture

A single source of truth for entries in the ViewModel; both list and detail read the same data. No duplicated copies of the list across screens.

✅ Definition of done (self-checked — your call, not a grade)

↗ Stretch goals (only after Done is green)

  • Add search/filter over the list using a derived state in the ViewModel (the query is state; the filtered list is computed from it).
  • Support delete with a confirmation, and a Material 3 swipe-to-dismiss or overflow menu.
  • Add a "sort by" control (date, name, distance) and keep the choice in the ViewModel.
🎓 The senior-engineer bar

The junior tell here is two copies of the list — one the LazyColumn reads and one the detail screen reads — that silently drift apart. A senior keeps one list in the ViewModel and passes only an id through navigation, looking the entry up from the single source of truth. If adding an entry on screen A instantly shows up everywhere with no "refresh," your state lives in exactly one place. Watch the empty state too: seniors design it deliberately; juniors discover it as a bug.

Rung 3 · Local persistence (it survives a restart)

PantryKeeper — an offline inventory with Room

Sessions 9 · 11 ~1 week CLO 4

Track what's in your pantry (or your tools, or your meds): full create / read / update / delete, with everything saved on the device so it's still there after you force-quit.

This is the rung where your app stops being a demo. Until now, kill the app and your data is gone. Here you add a Room database and a DataStore for preferences, and your LazyColumn starts observing a Flow<List> straight off a DAO — so the screen updates itself the instant a row changes. CRUD against a real local DB is the single most common thing Android jobs ask you to have done.

⚙️ What this rung forces you to learn

🗃️ Room (full CRUD)

An @Entity, a @Dao with insert/update/delete and a Flow<List> read query, and a @Database. Writes re-emit the read Flow automatically.

⚙️ DataStore

Store a couple of user preferences (e.g. sort order, "hide out-of-stock") with Preferences DataStore — the right tool for small key/value settings vs. a table of rows.

🏛️ Repository

A repository wraps the DAO so the ViewModel never imports Room. The ViewModel exposes StateFlow<UiState> via stateIn(...).

✅ Definition of done (self-checked — your call, not a grade)

↗ Stretch goals (only after Done is green)

  • Add a Room @Query with a search term and expose a filtered Flow driven by a query in the ViewModel.
  • Undo-able delete: keep the deleted row briefly and offer "Undo" in a snackbar before it's gone for good.
  • Seed the DB on first launch from a bundled JSON/asset so a fresh install isn't empty.
🎓 The senior-engineer bar

The junior returns List from the DAO and bolts on "now re-fetch and setState" calls after every write — and forgets half of them, so the screen goes stale. The senior returns Flow<List> so Room re-emits on every write and the read path and write path decouple completely. If your list updates after an edit and you never wrote a single "refresh" call, you've got it. The other tell: the ViewModel is unaware Room exists — it talks to a repository interface it could swap for a fake in a test.

Rung 4 · Live data from the internet

SkyCast — a weather/feed app over Retrofit

Sessions 10–11 ~1 week CLO 5CLO 6

Pull live data from a public REST API — weather, news headlines, GitHub repos, your pick — and render it with proper loading, success, and error states.

Now the data comes from a server you don't control, which is where the three states become non-negotiable: the network is slow (loading), sometimes it works (success), and sometimes it doesn't (error). This rung is the whole networking pipeline from the visualizer: a tap → a coroutine → an HTTP GET → JSON → Kotlin objects → a StateFlow the UI recomposes from. The fork between IOException (offline) and HttpException (server said no) is what separates a real app from a demo.

⚙️ What this rung forces you to learn

📡 Retrofit + repository

A typed Retrofit interface (@GET), kotlinx.serialization DTOs, a repository that catches and returns Result<T>, and the INTERNET permission in the manifest.

🧵 Coroutines off the main thread

The call runs in viewModelScope.launch on a background dispatcher. No NetworkOnMainThreadException, no frozen UI.

🎨 Three real UI states

A sealed UiState with Loading / Success(data) / Error(msg); a when renders exactly one, and Error has a working Retry.

✅ Definition of done (self-checked — your call, not a grade)

↗ Stretch goals (only after Done is green)

  • Add pull-to-refresh and a "last updated" timestamp so the user knows how fresh the data is.
  • Add an OkHttp logging interceptor at Level.BODY and read your real requests in Logcat.
  • Handle a 404/empty result as its own friendly state, distinct from a connectivity error.
🎓 The senior-engineer bar

The junior wraps the call in one catch (e: Exception) and shows "No internet" — even for a 401 on a full signal. The senior branches the catch: IOException means it never reached a healthy server (retry is fair), HttpException means the server was reached and refused (inspect code(), don't blind-retry). If a user with full bars and an expired token gets a real error message instead of "check your connection," your error taxonomy is correct. And the spinner always ends — even on failure — because the repository always returns a terminal Result.

★

Rung 5 · The portfolio capstone

RouteMate — offline-first, the whole stack in one app

Sessions 11–12 ~2 weeks CLO 1CLO 2CLO 3CLO 4CLO 5CLO 6

One polished app that fetches live data from a REST API, caches it locally in Room, and shows the cached copy instantly even with no signal — list → detail → settings, every state handled, the lot.

This is the rung you actually show people. It's not a new pattern — it's every previous rung fused into one architecture: navigation with arguments (Rung 2), a Compose UI driven by state (Rungs 1–2), Room persistence (Rung 3), and Retrofit networking (Rung 4) — wired into the offline-first pattern where the local cache is the single source of truth and the network's only job is to keep it fresh. Pick a domain you'd actually use: saved hiking routes, a reading list synced from an API, tracked GitHub repos, a recipe box. The domain is yours; the architecture is the deliverable.

🧭
The one rule that makes offline-first work

Data flows in one direction: the network writes into Room, and the UI only ever reads from Room. The Composable never touches the Retrofit response directly. Get this backwards — render the API result into the UI and save it to Room as two parallel paths — and the two copies drift, an empty refresh blanks a full screen, and "why is my list flickering" becomes your weekend. Funnel everything through Room and the offline behavior, the loading flicker, and process-death survival all fall out for free.

⚙️ Every in-scope skill, in one codebase

🧭 Navigation

3+ screens: list, detail (via {id} route), and settings. Clean back-stack behavior throughout.

🎨 Compose + state

A sealed UiState (Loading / Success / Empty / Error); UI is a pure function of state; survives rotation.

🗃️ Room + DataStore

Room is the source of truth, read as a Flow; DataStore holds user settings. Full CRUD where it fits the domain.

📡 Retrofit + repository

Network refresh writes into Room. One repository owns both sources; the ViewModel imports neither Room nor Retrofit.

✅ Definition of done (self-checked — this is your "ship it" bar, not a grade)

↗ Stretch goals — what makes it portfolio-stand-out

  • A short unit test for the repository's state-mapping or the IOException/HttpException branch, using a fake data source (no live server in the test).
  • Pull-to-refresh with a "last synced" timestamp, and a snackbar on a failed refresh that keeps the cached list on screen.
  • An empty-search and a no-results state that are visibly different from each other and from a network error.
  • A 20–40 second screen-recording GIF in the README showing the offline-first behavior (turn off Wi-Fi, relaunch, data's still there).
🎓 The senior-engineer bar

The junior's "final app" works only with a perfect connection on a fresh install and falls over the moment anything is missing. The senior's capstone is boringly robust: it opens instantly to cached data offline, a failed refresh is a quiet snackbar rather than a cleared screen, every state has a deliberate UI, and the architecture is so clean a reviewer can find the data flow in thirty seconds. The interview-grade signal isn't that it's flashy — it's that nothing about it surprises an experienced engineer. That's exactly what makes it worth putting your name on.

📦 Ship it: turn the capstone into a portfolio piece

A finished app on your laptop is invisible. A finished app with a clean GitHub repo, three screenshots, and a 60-second story is a portfolio piece. Here's exactly how to package it.

📝 The README a reviewer actually reads

Most reviewers spend under a minute on a repo before deciding whether to clone it. Front-load the README so that minute lands. Use this skeleton:

markdown · README.md

> One sentence: an offline-first Android app that <does the thing>,
> built in Kotlin + Jetpack Compose with Room caching and a Retrofit API layer.

## Architecture
UI (Compose) → ViewModel (StateFlow<UiState>) → Repository → { Room (source of truth), Retrofit (refresh) }

- **Offline-first:** Room is the single source of truth; the UI observes it via Flow.
  A failed network refresh never clears the cache.
- **State:** sealed UiState = Loading | Success | Empty | Error, each with its own UI.
- **Tech:** Jetpack Compose · Material 3 · Navigation-Compose · Room · DataStore ·
  Retrofit + kotlinx.serialization · Coroutines/Flow.

## Run it
1. Clone, open in Android Studio (latest stable).
2. (If needed) add your API key to `local.properties` as `API_KEY=...`.
3. Run on an emulator or device (min SDK 24).

💡
Three things that quietly disqualify a repo

A committed API key or secret (use local.properties, which is git-ignored), a build that doesn't compile from a clean clone, and a default app/build/ or .idea/ committed. Add a real .gitignore (Android Studio generates one) before your first push.

📸 The three screenshots to capture (and one GIF)

You don't need ten screenshots — you need the three that prove the hard parts, plus a short recording of the offline-first trick. Capture them on an emulator (Android Studio → running device → camera / record icons) and drop them in a docs/ folder.

1 · The listPopulated, scrolling, real data — your best-looking screen.
2 · The detailOne item open via the {id} route — proves navigation.
3 · Offline / errorWi-Fi off, cache still showing or a clean error+Retry.
4 · Demo GIF20–40s: launch → list → detail → turn off Wi-Fi → still works.

🗣️ What to say (the 60-second walkthrough)

Whether it's a demo day, a portfolio review, or an interview, you'll get about a minute. Don't narrate the UI ("here's a button") — narrate the decisions. Rehearse this arc out loud until it's natural:

1 · The one-liner (10s): "RouteMate is an offline-first Android app for <domain>. It pulls live data from <API>, caches it in Room, and works with no signal."

2 · The hard part, demoed (20s): "The interesting bit is the data flow — let me kill the Wi-Fi and relaunch. The cache is the source of truth, so the UI reads from Room and the network just refreshes it in the background. A failed refresh shows a snackbar but never wipes your data."

3 · The architecture (20s): "The ViewModel only knows about one repository interface — it never imports Room or Retrofit, so I can test it with a fake. State flows down as an immutable UiState; events flow up as function calls. Every state — loading, success, empty, error — has its own deliberate UI."

4 · The honest next step (10s): "If I had another week I'd add <X> — I scoped it out to ship something solid first." (Naming what's unfinished reads as judgment, not weakness.)

🔁 Where to sharpen between rungs

🎯
Build the app, drill the patterns

Each rung leans on a pattern you can rehearse in isolation when a build stalls. When you're stuck on a rung, don't grind the whole app — drill the one piece:

  1. Pattern reps: the auto-checked coding challenges — practice the repository cache+network merge, the four-state mapping, and the IOException-vs-HttpException classifier until each goes green from blank.
  2. Find the bug: the debug drills — the "DAO returns List not Flow," "no Loading/Error/Empty," and "main-thread network" seeds are the exact failures you'll hit on Rungs 3 and 4.
  3. Keep it warm: the spaced-repetition deck — so the patterns are still automatic on demo day, not just the day you learned them.
  4. Trace the pipeline: the networking visualizer and the Room CRUD simulator when a Rung 3 or 4 mental model feels fuzzy.
