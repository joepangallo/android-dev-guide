Lecture / Concept Notes

Enough detail to learn each idea from scratch. Every big idea is an accordion — open one at a time and work through it.

In the old View system, UI is imperative: you find a widget and command it to change — findViewById, then textView.setText("Hi"). You are responsible for keeping the screen in sync with the data.

Compose is declarative: you write a function that says "for this data, the screen looks like this." When the data changes, Compose re-runs the function and updates only what changed. That re-running is recomposition.

🧮
Analogy

Imperative is hand-drawing turn-by-turn directions and erasing/redrawing each turn yourself. Declarative is typing the destination into GPS — you describe the goal state and the system figures out the redraw. Today we describe a fixed destination; in Session 5 the destination changes as state changes, and recomposition does the redraw automatically.

⚠️
Misconception to pre-empt

"A composable function is a normal function I call to return a view." No — composables emit UI into the tree; they return Unit. You don't say val t = Text(...). Just call them for their effect.

A composable is a regular Kotlin function annotated with @Composable. Conventions:

  • PascalCase, noun-like name (ProfileCard, not drawProfileCard) — it describes a thing on screen.
  • Returns Unit (no return type written). It emits UI; it doesn't hand one back.
  • First param is usually modifier: Modifier = Modifier so callers can style it — we'll honor this convention all term.
Kotlin · Compose 
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.tooling.preview.Preview

@Composable
fun WelcomeBanner(
    title: String,
    modifier: Modifier = Modifier      // convention: modifier first, with a default
) {
    Text(text = title, modifier = modifier)
}

// @Preview renders this composable in the Design pane — no emulator needed.
@Preview(showBackground = true)
@Composable
fun WelcomeBannerPreview() {
    WelcomeBanner(title = "Welcome to COP4667")
}

Units & imports

Two new Kotlin-isms: 24.sp (scalable pixels, for text) and 16.dp (density-independent pixels, for layout). They come from Compose extension properties — students must import androidx.compose.ui.unit.dp / .sp. Also: TextField needs state (remember { mutableStateOf(...) }) — flag it as a Session-5 topic; today they may use it read-only or skip it.

♿
Accessibility note worth saying out loud

contentDescription is read by TalkBack. Give meaningful descriptions for informative images; pass null only for purely decorative ones. We'll reinforce this all term.

  • Column stacks children vertically (top → bottom).
  • Row places children horizontally (start → end).
  • Box stacks children on top of each other (z-order) — great for badges/overlays.

Control spacing/positioning with arrangement (the main axis) and alignment (the cross axis):

Kotlin · Compose 
// Column: main axis = vertical, cross axis = horizontal
Column(
    verticalArrangement = Arrangement.Center,      // main axis
    horizontalAlignment = Alignment.CenterHorizontally // cross axis
) {
    Text("Top")
    Text("Bottom")
}

// Row: main axis = horizontal, cross axis = vertical
Row(
    horizontalArrangement = Arrangement.SpaceBetween, // push children apart
    verticalAlignment = Alignment.CenterVertically
) {
    Text("Left")
    Text("Right")
}

// Box: overlap children; align each within the box
Box(contentAlignment = Alignment.Center) {
    Image(/* ... */)
    Text("Overlay label")     // sits on top of the image
}
🧮
Memory hook for arrangement vs alignment

"Arrangement moves along the main axis (the direction the layout flows); alignment moves across it." In a Column the main axis is vertical, so verticalArrangement spaces the stack and horizontalAlignment centers it left-right.

A Modifier decorates or augments a composable: size, padding, background, click handling, and more. You build a chain, and Compose applies it left-to-right, top-to-bottom. Order changes the result.

Kotlin · Compose 
// Common modifiers, chained:
Text(
    text = "Hello",
    modifier = Modifier
        .fillMaxWidth()                 // take the full available width
        .padding(16.dp)                 // space INSIDE the width, around content
        .background(Color(0xFF3DDC84))  // paint behind what's left
        .clip(RoundedCornerShape(12.dp))
        .clickable { /* tap */ }
)

Order demo: swapping padding and background changes whether the color sits inside or outside the padding.

.padding(16.dp).background(blue)
content

Padding first → the background paints only the inner region; padding is transparent margin around the color.

.background(green).padding(16.dp)
content

Background first → the color fills the whole area, and padding pushes the content inward on top of the color.

⚖️
weight — the Row/Column divider

Modifier.weight(1f) inside a Row/Column tells a child to take a proportional share of leftover space. Two children both at weight(1f) split 50/50; weight(2f) vs weight(1f) splits 2:1. We use this for the wide-screen lab extension.

The one mapping that matters most

The deepest shift is the last row: in Views you command a widget to change (setText); in Compose you change state and let recomposition redraw. Memorize the layout swaps (Column/Row/Box), but internalize that one.

ℹ️
Why Compose replaces Views

Less boilerplate (no XML + no findViewById + no keeping the two in sync), one language (Kotlin everywhere), UI that updates automatically from state, and powerful reuse via plain functions. The View system still exists and interops, but Compose is the recommended modern toolkit — which is why this course teaches it first.

Goal

Build your own styled profile / business card screen in Compose, using only layout composables and modifiers, that matches the target shown below. The full handout and acceptance checklist are right here in this section.

  1. Set up your file

    Create ProfileCard.kt (or reuse the live-coding file as a base, then make it yours — your name, your role, your bio). Add a @Preview so you can iterate without the emulator.

  2. Build the required pieces

    Your card must include all of these, arranged with a Column (and at least one Row or Box somewhere):

    • A circular avatar — an Image with .clip(CircleShape) and contentScale = ContentScale.Crop (or an Icon stand-in if you have no drawable).
    • A bold name Text (your name) — the largest, heaviest line.
    • A muted role Text (e.g. "Android Developer in training"), visually quieter than the name.
    • A centered bio Text — one or two sentences, textAlign = TextAlign.Center.
    • A Follow Button with a text label in its content slot.
    • Everything inside a Column with horizontalAlignment = Alignment.CenterHorizontally and verticalArrangement = Arrangement.spacedBy(12.dp).
    • At least one Row or Box — e.g. a Row of two small stat chips ("Following 0 · Followers 0") under the bio, or a Box placing a small badge over the avatar corner.
    • A surface look: a .background(color, RoundedCornerShape(16.dp)) with .padding(...) — and you decide the order on purpose (see the modifier-order demo above).

  3. Starter hints

    • Begin from the live-coding Column scaffold; change the content to your own.
    • Arrangement.spacedBy(12.dp) on the Column is the cleanest way to space children.
    • For a card look: .background(color, RoundedCornerShape(16.dp)) then .padding(...) — decide the order on purpose.
    • Let autocomplete add imports; if a symbol is red, hover → Alt+Enter (Option+Return) → Import.

  4. Expected result

    A clean, centered profile card that visually matches the target: photo on top, bold name, muted role, centered bio, and a button — with comfortable spacing and a subtle background.

  5. Push further — make it responsive

    Use the available width to switch between a vertical Column (narrow) and a horizontal Row (wide), with Modifier.weight(1f) giving the text block the remaining space beside the avatar.

    Kotlin · Compose 
    @Composable
fun AdaptiveProfileCard(modifier: Modifier = Modifier) {
    BoxWithConstraints(modifier = modifier.fillMaxWidth()) {
        val wide = maxWidth >= 480.dp
        if (wide) {
            Row(verticalAlignment = Alignment.CenterVertically) {
                Avatar()
                Spacer(Modifier.width(16.dp))
                Column(Modifier.weight(1f)) { ProfileText() }
            }
        } else {
            Column(horizontalAlignment = Alignment.CenterHorizontally) {
                Avatar()
                ProfileText()
            }
        }
    }
}

Troubleshooting
  • Preview is blank / "nothing to preview". Your @Preview function must itself be @Composable and call your card. Click "Build & Refresh" in the pane.
  • Unresolved reference: dp / sp / CircleShape. Missing import. Alt+Enter (Option+Return) on the red symbol → Import. (androidx.compose.ui.unit.dp, ...foundation.shape.CircleShape.)
  • Image is square, not round. Add .clip(CircleShape) and contentScale = ContentScale.Crop; clip must come after size in the chain.
  • Everything overlaps. You haven't wrapped the children in a Column/Row — without a layout, composables draw at the same spot.
  • Drawable not found (R.drawable.x). The image isn't in res/drawable, or the name has illegal characters (must be lowercase, no spaces).

Worked example: reading a modifier chain like the compiler does

The single hardest idea in this session is not "what does padding do" — it is that a Modifier is an ordered pipeline. Each link wraps the previous result, so the chain reads like a stack of nested boxes from the outside in. Master this and modifier-order bugs disappear.

🧠
The mental model that always works

Read a chain top-to-bottom as "then". Each constraint is measured before the next runs. The first size/padding link claims outer space; later links operate on whatever space is left. Drawing (like .background()) paints the region that exists at that point in the chain — so anything after a .padding() draws only the shrunken inner region; anything before it draws the full region.

Trace these two chains by hand before reading the answer. A 100 dp-wide content area:

Kotlin · Compose 
Modifier
    .size(100.dp)          // box is 100×100
    .background(Green)     // paint all 100×100 green
    .padding(16.dp)        // shrink content area to 68×68

Result: a fully green 100 dp box with 16 dp of green showing around the content. Green is painted before padding, so padding eats into the green.

Kotlin · Compose 
Modifier
    .size(100.dp)          // box is 100×100
    .padding(16.dp)        // inner region is now 68×68
    .background(Green)     // paint only the 68×68

Result: a 100 dp box with a 16 dp transparent margin and a smaller green square inside. Green is painted after padding, so it only fills the inner region.

Same two modifiers, opposite visuals — entirely because of order. Now fade the support away and reason about a chain you have not seen. There is no preview here; the answer is a structured prediction, which forces the model into your head.

✍️
Faded task — predict, then check

A composable uses this chain. Predict, in order: (a) the box's outer size, (b) whether the green fills edge-to-edge or sits inset, (c) where the 24 dp of empty space ends up, and (d) whether a tap on the transparent margin registers a click.

Kotlin · Compose 
Box(
    modifier = Modifier
        .size(120.dp)
        .padding(24.dp)
        .clip(RoundedCornerShape(12.dp))
        .background(Color(0xFF3DDC84))
        .clickable { /* TODO */ }
)
  • (a) Outer size: 120×120 dp. .size() is first, so the whole element reserves 120 dp regardless of what follows.
  • (b) Green fill: inset. .padding(24.dp) comes before .background(), so the green only fills the inner 72×72 dp region.
  • (c) Empty space: a 24 dp transparent frame around the green square.
  • (d) Click area: only the inner 72×72 dp green region is clickable — .clickable is after the padding, so the padded margin is dead to taps. (If you wanted the whole 120 dp tappable, put .clickable before .padding.) This last point is the one that bites real apps: an invisible "why won't my button tap near the edge" bug.

Kotlin 
fun rowLabel(name: String, role: String, unread: Int): String {
    val n = name.trim()
    val r = role.trim()
    if (n.isBlank()) return "Unknown contact"
    val base = "$n · $r"
    return when {
        unread <= 0 -> base
        else        -> "$base  ($unread new)"
    }
}

Note how the function is pure: no Compose, no state, no side effects. That is exactly what makes it trivially testable — and why senior engineers pull this kind of logic out of composables.

Common misconceptions & pitfalls

Each one is a real mental model that produces real bugs. Read the symptom first — if you have ever seen it, you had the wrong model.

Wrong model: the chain is a bag of properties; Compose figures out the "right" final look regardless of order.

Why it's wrong: a Modifier is an ordered chain; each link wraps the previous one. size, padding, background, clip, and clickable all measure or draw relative to the chain so far.

Correct model: read it as a pipeline, outside-in. The first link claims space; later links work inside it. .clickable only captures taps on the region that exists at its point in the chain.

Symptom: a button or card that "won't tap near the edges," or a background color that mysteriously fills too much/too little. Move .padding relative to .background/.clickable and the symptom flips.

Wrong model: val card = ProfileCard() gives me a view I can keep, pass around, or add to a list.

Why it's wrong: composables return Unit. Calling one emits nodes into the composition tree as a side effect at that call site; there is no object handed back.

Correct model: call composables for their effect, in the position you want them to appear. To reuse UI, call the function again — not store its "result."

Symptom: code like val x = Text("hi") where x is Unit, or trying to build a List<@Composable> and getting confused why nothing renders.

Wrong model: writing Text(...) three times in a row naturally stacks them vertically, like lines in a document.

Why it's wrong: without a layout composable, siblings have no defined relationship — they are placed at the same top-start origin and overlap. Stacking/spacing is the layout's job, not the children's.

Correct model: a Column stacks (main axis vertical), a Row lays out horizontally, a Box deliberately overlaps for z-order. You choose the relationship explicitly.

Symptom: all your text crammed into the top-left corner, one label drawn on top of another. Wrap the siblings in a Column and it resolves.

Wrong model: "to center my Column's children, I'll set verticalAlignment" (guessing whichever parameter autocomplete offers).

Why it's wrong: a Column has no verticalAlignment — its vertical axis is the main axis, controlled by verticalArrangement; horizontalAlignment handles the cross axis. A Row is the mirror image.

Correct model: Arrangement moves children along the main axis (the direction the layout flows); alignment moves them across it. Column: verticalArrangement + horizontalAlignment. Row: horizontalArrangement + verticalAlignment.

Symptom: "unresolved reference: verticalAlignment" inside a Column, or children that center on the wrong axis.

Wrong model: "it's a function, so it executes one time when the screen opens, and that's it."

Why it's wrong: Compose re-runs a composable whenever the state it reads changes — that is recomposition. It can run many times, in any order relative to siblings, and skip composables whose inputs didn't change. Treat the body as describing UI for the current state, not as a one-shot script.

Correct model: the function is a pure description that may be re-invoked. Never do one-time work (logging, network calls, mutating external lists) directly in the body — it will fire on every recomposition. (You learn the right tools — remember, side-effect APIs — in Session 5.)

Symptom: a log line that prints far more often than expected, a counter that resets, or a network call that fires repeatedly. This is the bridge into Session 5 — keep it in mind.

In production

🏗️
How a senior engineer actually writes a composable
  • Every reusable composable takes modifier: Modifier = Modifier as its first optional parameter, and applies it to the outermost element — once. This is the single most important convention in the ecosystem: it lets a caller size, pad, or position your component without you anticipating every use. A component that hardcodes its own size/padding instead is a code smell.
  • Hoist data in, hoist events out. A production ContactRow takes plain data (name, role) and lambdas (onClick: () -> Unit) — it never reaches out for state or runs business logic. Keep composables "dumb" and you can preview, reuse, and reason about them. (We formalize this as state hoisting in Session 5 and architecture in Session 8.)
  • Pull non-UI logic into pure functions (exactly like the rowLabel helper above). It moves the hard-to-test part out of the hard-to-test place.
  • Multiple @Previews are free documentation — one per meaningful state (empty, one item, many, long text, dark theme via uiMode). Seniors preview the edge cases, not just the happy path, so layout bugs surface before the emulator.

Tasteful gotcha: resist parameter-list bloat. When a composable grows past ~5–6 parameters, that is the signal to group related data into a small data class (a UI-state object) and pass that — not to add a seventh boolean flag. You will feel this pressure on the project; the fix is a data class, learned in Session 2, applied here.
