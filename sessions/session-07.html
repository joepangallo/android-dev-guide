1 · Why lazy lists? Column vs LazyColumn

A plain Column (with Modifier.verticalScroll) composes and lays out every child immediately, whether it’s visible or not. For 10 items that’s fine. For 1,000 — or a feed that never ends — it’s a performance disaster: you build 1,000 composables, hold them all in memory, and pay layout for all of them on the very first frame.

A LazyColumn is lazy: it only composes the items currently on screen (plus a small buffer), and recycles them as you scroll — the same way RecyclerView worked in the old View world, but with far less boilerplate.

💡
Analogy

Column = printing an entire 1,000-page book and stacking every page on the desk before you read one. LazyColumn = an e-reader that renders only the page you’re on and the next one or two. Same book, vastly less work.

Kotlin
// ❌ Fine for a handful of items, BAD for long/unbounded lists.
// Every Text() is composed up front; nothing is recycled.
@Composable
fun NamesColumn(names: List<String>) {
    Column(modifier = Modifier.verticalScroll(rememberScrollState())) {
        for (name in names) {
            Text(text = name, modifier = Modifier.padding(16.dp))
        }
    }
}

Use a plain Column only when the number of children is small and fixed (e.g., a form with 5 fields).

Kotlin
// ✅ Composes only visible rows; recycles as you scroll.
// No manual scroll modifier needed — LazyColumn scrolls itself.
@Composable
fun NamesList(names: List<String>) {
    LazyColumn {
        items(names) { name ->
            Text(text = name, modifier = Modifier.padding(16.dp))
        }
    }
}

Notice: no verticalScroll. A LazyColumn is itself the scroll container.

2 · The items() DSL, keys, content padding & spacing

Inside a LazyColumn you don’t write a normal for loop — you call the lazy list DSL: item { } for a single child, and items(list) { element -> } for a collection. You can mix them (e.g., a header item { } followed by items(data)).

Keys matter. Give each item a stable, unique key (typically the data object’s id). Without keys, Compose tracks items by position; insert/remove/reorder and it may reuse the wrong state, lose scroll position, or mis-animate. With keys, Compose can correctly match items across data changes.

contentPadding pads the scrollable content inside the list (so the first/last items aren’t flush against the edges) without clipping the scroll. verticalArrangement = Arrangement.spacedBy(...) adds gaps between items — cleaner than putting margins on each row.

Kotlin
import androidx.compose.foundation.layout.Arrangement
import androidx.compose.foundation.layout.PaddingValues
import androidx.compose.foundation.lazy.LazyColumn
import androidx.compose.foundation.lazy.items
import androidx.compose.ui.unit.dp

data class Course(val id: Int, val code: String, val title: String)

@Composable
fun CourseList(courses: List<Course>) {
    LazyColumn(
        contentPadding = PaddingValues(16.dp),          // inset top/bottom/sides
        verticalArrangement = Arrangement.spacedBy(12.dp) // gap between rows
    ) {
        item {                                           // a one-off header
            Text("Catalog", style = MaterialTheme.typography.titleLarge)
        }
        items(
            items = courses,
            key = { course -> course.id }                // STABLE, UNIQUE key
        ) { course ->
            Text("${course.code} — ${course.title}")
        }
    }
}
⚠️
Two import gotchas

The items() overload that takes a List lives in androidx.compose.foundation.lazy.items — students often auto-import the wrong thing and get a “type mismatch.” Also, a bare items(5) { i -> } takes an Int count, not a list — different overload. When in doubt, let Studio import after you type items(.

3 · Material 3 chrome: Scaffold, TopAppBar, FAB

Real apps aren’t just a list — they have a top bar, maybe a bottom bar, and a floating action button. Compose Material 3 gives you Scaffold: a layout that slots these standard pieces into the right places and hands your content the leftover space as innerPadding. You must apply that padding or your list will hide behind the app bar.

🧩 Anatomy of a Scaffold
Scaffold(
  topBar      = { TopAppBar }      // title, nav icon, actions
  bottomBar   = { NavigationBar }  // (optional) bottom tabs
  floatingActionButton = { FAB }   // the round + button
) { innerPadding ->            // ← MUST be applied to content
      LazyColumn(Modifier.padding(innerPadding)) { ... }
}
Kotlin
import androidx.compose.material3.*
import androidx.compose.material.icons.Icons
import androidx.compose.material.icons.filled.Add

@OptIn(ExperimentalMaterial3Api::class)   // TopAppBar is still experimental API
@Composable
fun CourseScreen(courses: List<Course>, onAdd: () -> Unit) {
    Scaffold(
        topBar = {
            TopAppBar(title = { Text("My Courses") })
        },
        floatingActionButton = {
            FloatingActionButton(onClick = onAdd) {
                Icon(Icons.Filled.Add, contentDescription = "Add course")
            }
        }
    ) { innerPadding ->
        LazyColumn(
            modifier = Modifier.padding(innerPadding),   // ← apply Scaffold padding
            contentPadding = PaddingValues(16.dp),
            verticalArrangement = Arrangement.spacedBy(12.dp)
        ) {
            items(courses, key = { it.id }) { course ->
                Text("${course.code} — ${course.title}")
            }
        }
    }
}
Kotlin
import androidx.compose.material3.Card
import androidx.compose.material3.CardDefaults

@Composable
fun CourseCard(course: Course, onClick: () -> Unit) {
    Card(
        onClick = onClick,                               // Card has a clickable overload
        modifier = Modifier.fillMaxWidth(),
        elevation = CardDefaults.cardElevation(defaultElevation = 2.dp)
    ) {
        Column(Modifier.padding(16.dp)) {
            Text(course.code, style = MaterialTheme.typography.titleMedium)
            Text(course.title, style = MaterialTheme.typography.bodyMedium)
        }
    }
}

A Card is a Material surface with rounded corners + elevation. The onClick overload gives you ripple feedback for free.

<androidx.recyclerview.widget.RecyclerView

// No notifyDataSetChanged().

If you ever see RecyclerView/Adapter in old tutorials, that’s the legacy View toolkit. You do the same thing with LazyColumn + items.

5 · Click handling with lambda callbacks (state hoisting)

A reusable list item should not know what happens on tap — it just reports “I was tapped.” We pass an onClick: () -> Unit (or onClick: (Course) -> Unit) into the item, and the parent screen decides the behavior (toast, navigate, delete). This is event hoisting: the same principle as lifting state up that we saw in S6, applied to events.

Kotlin
// Child: dumb & reusable — receives data + an event lambda
@Composable
fun CourseCard(course: Course, onCourseClick: (Course) -> Unit) {
    Card(onClick = { onCourseClick(course) }, modifier = Modifier.fillMaxWidth()) {
        Column(Modifier.padding(16.dp)) {
            Text(course.code, style = MaterialTheme.typography.titleMedium)
            Text(course.title)
        }
    }
}

// Parent: decides what the click MEANS
@Composable
fun CourseScreen(courses: List<Course>) {
    val context = LocalContext.current
    LazyColumn {
        items(courses, key = { it.id }) { course ->
            CourseCard(course) { tapped ->
                Toast.makeText(context, "Tapped ${tapped.code}", Toast.LENGTH_SHORT).show()
                Log.d("S7", "Clicked: $tapped")
            }
        }
    }
}
🪝
Why hoist the event?

The same CourseCard can be reused on a “favorites” screen where the tap navigates, and a “delete” screen where it removes. The card stays identical — only the lambda the parent passes changes. Reusable + testable.

6 · Making the list change: state-backed lists (the FAB)

To add an item and have the list update, the list must be observable state. Use remember { mutableStateListOf(...) } (a Compose-aware list) or a mutableStateOf(List) you reassign. Mutating a mutableStateListOf triggers recomposition automatically.

Kotlin
import androidx.compose.runtime.mutableStateListOf
import androidx.compose.runtime.remember

@Composable
fun CourseScreen() {
    // State-backed list: add/remove → automatic recompose
    val courses = remember { mutableStateListOf(*sampleCourses.toTypedArray()) }

    Scaffold(
        floatingActionButton = {
            FloatingActionButton(onClick = {
                val next = courses.size + 1
                courses.add(Course(next, "NEW$next", "New course $next", 3))
            }) { Icon(Icons.Filled.Add, contentDescription = "Add") }
        }
    ) { padding ->
        LazyColumn(Modifier.padding(padding)) {
            items(courses, key = { it.id }) { course ->
                CourseCard(course) { /* onClick */ }
            }
        }
    }
}

    Swap the static list for mutableStateListOf and add a FAB. Tapping + appends an item and the list recomposes instantly.

    Kotlin
    @OptIn(ExperimentalMaterial3Api::class)
@Composable
fun CourseScreen() {
    val context = LocalContext.current
    val courses = remember { mutableStateListOf(*sampleCourses.toTypedArray()) }

    Scaffold(
        topBar = { TopAppBar(title = { Text("My Courses (${courses.size})") }) },
        floatingActionButton = {
            FloatingActionButton(onClick = {
                val next = (courses.maxOfOrNull { it.id } ?: 0) + 1
                courses.add(Course(next, "NEW$next", "New course $next", 3))
            }) {
                Icon(Icons.Filled.Add, contentDescription = "Add a course")
            }
        }
    ) { innerPadding ->
        CourseList(
            courses = courses,
            onCourseClick = { course ->
                Toast.makeText(context, "Tapped ${course.code}", Toast.LENGTH_SHORT).show()
            },
            modifier = Modifier.padding(innerPadding)
        )
    }
}

    See: the title count goes up and a new card appears at the bottom each FAB tap — no manual refresh, no adapter. That’s recomposition.

Troubleshooting (common errors & fixes)
  • “Unresolved reference: items” → wrong import. Use import androidx.compose.foundation.lazy.items.
  • First card hidden under the app bar → you didn’t apply Modifier.padding(innerPadding) from the Scaffold.
  • “TopAppBar … experimental” red squiggle → add @OptIn(ExperimentalMaterial3Api::class) above the composable.
  • Toast won’t compile → you need a Context: val context = LocalContext.current at the top of the composable.
  • List won’t scroll → you used a plain Column, or you wrapped LazyColumn in another scrolling container. LazyColumn scrolls itself.
  • FAB add does nothing → the list is a plain List, not remember { mutableStateListOf(...) }; mutate the state list.
  • Duplicate-key crash → two items share the same id. Keys must be unique.

⌁ Master it — lists that stay correct under change

The deepest idea in this session isn’t “draw a list.” It’s keeping each row’s identity and state correct while the data underneath it changes — inserts, removals, reorders, toggles. That is exactly where production list bugs live, and it’s the skill that separates a working demo from a feed people trust. Work the example below, then complete the faded version yourself.

Worked example: why key + observable state are one idea

Picture a to-do feed. Each row is selectable (a checkbox), and you can delete the top item. The data is a List<Task>; the per-row “is this expanded / is the checkbox bouncing” lives in Compose, keyed by item identity. Reason through what happens when you delete index 0:

  1. Without keys — Compose tracks by position

    Row slot #0 used to render Task A. After deletion, slot #0 now renders Task B — but the remembered state attached to slot #0 (selection highlight, scroll-into-view animation) stays put. Result: B suddenly looks selected, or animates as if it were A moving. The state “bleeds” onto the wrong item.

  2. With key = { it.id } — Compose tracks by identity

    Now Compose knows slot-for-A is gone and B already had its own slot. It disposes A’s remembered state, keeps B’s, and animates the removal correctly. The key is what makes “the same item” a stable concept across recompositions.

  3. The state must be observable, or nothing recomposes at all

    Delete from a plain List and the UI never updates — Compose didn’t observe a change. Use mutableStateListOf (or hoist to a StateFlow you reassign with copy()). Keys decide which row is which; observable state decides whether a redraw happens.

Below is that exact reduce logic as pure Kotlin you can run right here — the same add / remove / toggle operations a mutableStateListOf performs, plus an identity check proving the toggled item is matched by id, not position. Predict each printed line, then press Run to confirm.

Kotlin · runnable
data class Task(val id: Int, val title: String, val done: Boolean = false)

// The three operations every state-backed list supports.
fun List<Task>.added(t: Task): List<Task> = this + t
fun List<Task>.removed(id: Int): List<Task> = filterNot { it.id == id }
fun List<Task>.toggled(id: Int): List<Task> =
    map { if (it.id == id) it.copy(done = !it.done) else it }   // match by IDENTITY, not index

fun main() {
    var tasks = listOf(Task(1, "Wireframe"), Task(2, "Build list"), Task(3, "Ship"))

    tasks = tasks.toggled(2)                 // flip "Build list" by its id
    println(tasks.first { it.id == 2 }.done) // -> true

    tasks = tasks.removed(1)                 // delete "Wireframe" (was index 0)
    println(tasks.map { it.id })             // -> [2, 3]  (item 2 kept its done=true)
    println(tasks.first { it.id == 2 }.done) // -> true   (state did NOT bleed to a neighbor)

    tasks = tasks.added(Task(4, "Polish"))
    println(tasks.size)                      // -> 3
}

Mapping to Compose: added/removed/toggled are what the FAB and a checkbox call; items(tasks, key = { it.id }) is the line that makes the identity-matching above survive on screen.

Kotlin
fun reduce(state: List<Item>, event: ListEvent): List<Item> = when (event) {
    is ListEvent.Add    -> state + event.item
    is ListEvent.Remove -> state.filterNot { it.id == event.id }
    is ListEvent.Toggle -> state.map {
        if (it.id == event.id) it.copy(selected = !it.selected) else it
    }
    ListEvent.Clear     -> emptyList()
}

fun isEmptyState(state: List<Item>): Boolean = state.isEmpty()

Exhaustive when over a sealed type means adding a new event later won’t silently compile — the compiler flags every reducer that forgot it. That’s why production UDF code models events as sealed types.

Common misconceptions & pitfalls

Wrong model: keys are an optional optimization you can skip until things feel slow.

Why it’s wrong: without keys, Compose identifies items by their position in the list. The data shown is always correct, but anything remembered for a row — animation in-flight, a TextField’s focus/cursor, an expanded/selected flag, even scroll anchoring — is tied to the slot, not the item. Insert at the top or reorder, and that remembered state stays on the slot and lands on the wrong item.

Correct model: items(list, key = { it.id }) binds remembered state to item identity, so Compose moves/disposes state with the item, not the slot.

Symptom you’d see: after inserting a new item at the top, the wrong row appears highlighted/expanded, a delete animation plays on a neighbor, or a half-typed text field jumps to a different row.

Wrong model: a Compose UI is “bound” to your list, so changing the list redraws it.

Why it’s wrong: a plain List (even a MutableList) is invisible to Compose. Recomposition only happens when something Compose observes changes. Calling .add() on an ordinary mutableListOf changes data but triggers no recomposition.

Correct model: use observable state — remember { mutableStateListOf(...) } for a local list, or hoist to a StateFlow<List<T>> that you replace with copy()/a new list (Session 8). Mutating the state-backed container is what schedules the redraw.

Symptom you’d see: the FAB “does nothing” — data clearly changed (a log confirms it) but no new card appears until you rotate or navigate away and back.

Wrong model: hardcode something like contentPadding = PaddingValues(top = 64.dp) to clear the bar.

Why it’s wrong: Scaffold already computes the exact space its bars consume and hands it to you as innerPadding. A magic number breaks on different bar heights, large/medium app bars, landscape, gesture insets, and edge-to-edge devices. You’re re-deriving a value the framework already gave you.

Correct model: apply the Scaffold’s lambda parameter: Modifier.padding(innerPadding) on the content (and pass contentPadding separately only for the breathing room inside the scroll).

Symptom you’d see: looks fine on your phone, then on a tablet or in landscape the first card is clipped or there’s a dead gap.

Wrong model: put a header and the list in a Column(Modifier.verticalScroll(...)) together.

Why it’s wrong: nesting a vertically-scrolling list inside a vertically-scrolling parent gives the LazyColumn infinite height to measure, so it composes all items at once — you lose the entire laziness benefit (and Compose may throw “Vertically scrollable component was measured with infinity max height”).

Correct model: let the LazyColumn own the scroll. Put the header as the first item { } (or a stickyHeader) inside the lazy list, not a scrolling wrapper around it.

Symptom you’d see: a runtime crash about infinite height, or silent jank/memory spikes on long lists because nothing recycles.

Wrong model: reuse a non-unique field (a name, a category) as the key, or default to the item’s index.

Why it’s wrong: Compose requires keys to be unique within the list. Duplicate keys throw at runtime, and using the index as a key is the same as having no stable identity at all — it changes the moment you insert/remove.

Correct model: key on a genuinely unique, stable id (a database primary key, a server id, or a generated UUID created when the item is added — never the loop index).

Symptom you’d see: IllegalArgumentException: Key "…" was already used crashing the screen, or with index keys, the same state-bleed bug from the first misconception.

How a senior engineer ships a list screen

The list almost never owns its own data. State is hoisted to a ViewModel that exposes a single immutable StateFlow<UiState>, collected with collectAsStateWithLifecycle(). The screen is a pure function of that state and renders four cases by default — Loading, Success(list), Empty, Error — because “Success with zero items” is a real, common, and very different screen from “still loading.” The list-item composable stays dumb: it takes a data object plus event lambdas (onClick, onToggle) and reports up; the ViewModel decides what those events mean. Keys come from the domain id, and the item composable carries a modifier: Modifier = Modifier parameter so callers control sizing/placement — the convention every reusable Compose component follows.

Tasteful gotcha: if your data class holds a List or other field that doesn’t implement a stable equals, Compose treats every row as “unstable” and may recompose rows that didn’t actually change. Keep item models as data classes of primitives/immutable types (or mark them @Immutable), and you get skipping for free — the difference between a buttery feed and one that drops frames while scrolling.

That isEmptyState predicate you wrote in the challenge isn’t just bookkeeping — it’s the branch that decides whether your LazyColumn shows the list or a friendly placeholder. Here is the four-state idea as the real Compose code you’d type, with the Empty case rendered as a single centered item { } instead of items():

Kotlin
@Composable
fun CourseList(courses: List<Course>, onCourseClick: (Course) -> Unit, modifier: Modifier = Modifier) {
    LazyColumn(
        modifier = modifier.fillMaxSize(),
        contentPadding = PaddingValues(16.dp),
        verticalArrangement = Arrangement.spacedBy(12.dp)
    ) {
        if (courses.isEmpty()) {
            // EMPTY state — one centered placeholder, NOT items()
            item {
                Box(Modifier.fillParentMaxSize(), contentAlignment = Alignment.Center) {
                    Text("No courses yet — tap + to add one.",
                        style = MaterialTheme.typography.bodyLarge)
                }
            }
        } else {
            // SUCCESS state — the real rows, keyed by identity
            items(courses, key = { it.id }) { course ->
                CourseCard(course) { onCourseClick(course) }
            }
        }
    }
}

Note fillParentMaxSize() (a LazyColumn-scope modifier) lets the placeholder fill the whole viewport so the empty message is centered, not hugging the top. The Loading and Error cases pick the same way — swap the body for a CircularProgressIndicator or a retry message — which is why production code models the screen as one when (uiState) over a sealed UiState rather than a bare list.
