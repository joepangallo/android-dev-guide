Big idea 1 — Ecosystem, Kotlin, and the Compose mental model CLO 1CLO 3

Analogy: the Android ecosystem is a city. The OS is the roads and utilities, the SDK is the building code, Android Studio is your workshop, Gradle is the delivery truck, and your app is the house you build to code. Kotlin is the language you give your instructions in.

Kotlin earns its keep with null safety — the type system forces you to handle "this might be missing" instead of crashing at runtime. Remind students this is the same NullPointerException pain they may know from Java, designed out of the language.

Kotlin
// The type system makes "maybe missing" explicit.
val name: String = "Ada"        // can never be null
var nickname: String? = null    // nullable — '?' is required

// Safe-call + Elvis: no NPE possible here.
val length = nickname?.length ?: 0

// Smart cast: inside the null check, the compiler knows it's non-null.
fun greet(who: String?) {
    if (who != null) {
        println("Hi, ${who.uppercase()}") // 'who' is String here
    }
}
Java
// Nullability is a comment, not a contract — the compiler can't help.
String name = "Ada";
String nickname = null;

// You must remember the guard yourself, or it's a runtime crash.
int length = (nickname != null) ? nickname.length() : 0;

void greet(String who) {
    if (who != null) {
        System.out.println("Hi, " + who.toUpperCase());
    }
}

The Compose mental model is the heart of the whole course. Old Android (Views/XML) was imperative: you found a widget by id and mutated it (textView.setText(...)). Compose is declarative: you write a function that describes the UI for a given state, and the framework re-runs (recomposes) that function whenever the state changes.

Kotlin
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    // Describe the UI as a function of 'count'.
    Button(onClick = { count++ }) {
        Text("Clicked $count times")
    }
    // When 'count' changes, Compose recomposes this function for us.
}
Kotlin + XML
// activity_main.xml defines a <Button android:id="@+id/button"/>
class MainActivity : AppCompatActivity() {
    private var count = 0
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        val button = findViewById<Button>(R.id.button)
        button.setOnClickListener {
            count++
            button.text = "Clicked $count times"  // we mutate the widget by hand
        }
    }
}
💡
One-line takeaway

In Compose you never "update the UI." You update the state, and the UI updates itself. Say this sentence at every station today — it is the through-line of the entire course.

Big idea 2 — State, recomposition, and unidirectional data flow CLO 2CLO 3

Recomposition = Compose re-running composable functions whose inputs changed. remember caches a value across recompositions; mutableStateOf makes a value observable so a change triggers recomposition. rememberSaveable additionally survives configuration changes (rotation) and process death of the UI.

State hoisting is the pattern that makes components reusable and testable: move state up to the caller, and pass the value down plus an event up. This produces unidirectional data flow (UDF): state flows down, events flow up.

Kotlin
// HOISTED: this composable owns no state. It is "stateless" and reusable.
@Composable
fun NameField(
    name: String,                       // state flows DOWN
    onNameChange: (String) -> Unit      // events flow UP
) {
    OutlinedTextField(
        value = name,
        onValueChange = onNameChange,
        label = { Text("Your name") }
    )
}

// The caller owns the state and decides what to do with events.
@Composable
fun SignupScreen() {
    var name by rememberSaveable { mutableStateOf("") }
    NameField(name = name, onNameChange = { name = it })
    Text(if (name.isBlank()) "Type your name" else "Hello, $name!")
}

A stateless NameField can be dropped into a signup screen, a profile editor, or a search bar — each caller supplies its own state and handler. A composable that owns its own state can only ever do one thing. Hoisting also makes the component trivial to preview and unit-test because its output is a pure function of its inputs.

  • remember — survives recomposition; lost on rotation. Use for transient UI state (is this menu expanded?).
  • rememberSaveable — also survives configuration changes and UI process death by saving to a Bundle. Use for small UI state you don't want to lose on rotation (a typed-in field).
  • ViewModel — survives configuration changes and is the right home for screen state and business logic, especially anything async (network, DB). It outlives the composable.

Big idea 3 — Navigation and clean architecture CLO 2CLO 3

Navigation-Compose models your app as a graph of routes. A NavHost maps each route string to a screen composable; you navigate by route, and you pass arguments through the route. This is the "≥2 screens with navigation" feature checklist requirement.

Kotlin
@Composable
fun AppNav(navController: NavHostController = rememberNavController()) {
    NavHost(navController, startDestination = "list") {
        composable("list") {
            ListScreen(onItemClick = { id -> navController.navigate("detail/$id") })
        }
        composable(
            route = "detail/{itemId}",
            arguments = listOf(navArgument("itemId") { type = NavType.IntType })
        ) { backStackEntry ->
            val id = backStackEntry.arguments?.getInt("itemId") ?: 0
            DetailScreen(itemId = id, onBack = { navController.popBackStack() })
        }
    }
}

Clean architecture is the "ViewModel + repository" requirement. Each layer has one job, and dependencies point inward (UI depends on ViewModel depends on repository — never the reverse).

LayerResponsibilityDoes NOT
UI (Composable)Render state, emit user eventsTalk to network/DB; hold long-lived state
ViewModelHold screen state (UiState), run use-cases, survive rotationReference Views/Context-heavy Android UI
RepositoryDecide where data comes from (cache vs network), expose a clean APIKnow anything about the UI
Data sourcesRetrofit (network), Room/DataStore (local)Make UI decisions
Kotlin
// A typical screen ViewModel with a single immutable UiState.
data class WeatherUiState(
    val isLoading: Boolean = false,
    val temperatureC: Int? = null,
    val error: String? = null
)

class WeatherViewModel(
    private val repo: WeatherRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow(WeatherUiState())
    val uiState: StateFlow<WeatherUiState> = _uiState.asStateFlow()

    fun load(city: String) {
        _uiState.value = WeatherUiState(isLoading = true)
        viewModelScope.launch {
            try {
                val temp = repo.getTemperature(city)   // suspend call
                _uiState.value = WeatherUiState(temperatureC = temp)
            } catch (e: Exception) {
                _uiState.value = WeatherUiState(error = e.message ?: "Unknown error")
            }
        }
    }
}

Networking (the "data from a web API" requirement) uses Retrofit over OkHttp, with a JSON converter, all on a background coroutine. This is the CLO 5/6 capstone: HTTP, REST, JSON parsing, and asynchronous I/O.

Kotlin
// 1) Model the JSON
@Serializable
data class WeatherDto(val tempC: Int, val description: String)

// 2) Declare the endpoint
interface WeatherApi {
    @GET("weather")
    suspend fun getWeather(@Query("city") city: String): WeatherDto
}

// 3) Build Retrofit (kotlinx.serialization converter)
val retrofit = Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .addConverterFactory(
        Json.asConverterFactory("application/json".toMediaType())
    )
    .build()

val api: WeatherApi = retrofit.create(WeatherApi::class.java)

// 4) The repository hides the source — UI never sees Retrofit.
class WeatherRepository(private val api: WeatherApi) {
    suspend fun getTemperature(city: String): Int = api.getWeather(city).tempC
}
⚠️
The three networking gotchas to name out loud

(1) INTERNET permission in the manifest or every call fails. (2) Never block the main thread — network goes in a coroutine (suspend + viewModelScope.launch). (3) Always handle the error path; phones lose connectivity constantly, so a try/catch that sets uiState.error is not optional.

Worked Exemplar — A Clean Network Screen, End to End

If you ever want a reference for "how should I have wired the network call cleanly?", build this exemplar yourself, step by step: a single screen that fetches data, shows loading/error/success, and survives rotation. It threads CLO 3, 5, and 6 into one small, complete slice you can adapt straight into your own project.

  1. Add the INTERNET permission

    Open AndroidManifest.xml and add the permission above <application>. Nothing visible changes yet — but skipping this is the #1 "why is my app crashing" cause.

    XML
    <uses-permission android:name="android.permission.INTERNET" />

  2. Define the UiState

    One sealed type models every screen state. Keep this in mind as you write it: your UI will be a single when over these cases.

    Kotlin
    sealed interface QuoteUiState {
    data object Loading : QuoteUiState
    data class Success(val text: String, val author: String) : QuoteUiState
    data class Error(val message: String) : QuoteUiState
}

  3. Write the ViewModel

    The ViewModel owns the state and the error handling. Watch the state transition from Loading to Success/Error as the coroutine completes.

    Kotlin
    class QuoteViewModel(private val repo: QuoteRepository) : ViewModel() {
    private val _state = MutableStateFlow<QuoteUiState>(QuoteUiState.Loading)
    val state: StateFlow<QuoteUiState> = _state.asStateFlow()

    init { refresh() }

    fun refresh() {
        _state.value = QuoteUiState.Loading
        viewModelScope.launch {
            _state.value = try {
                val q = repo.randomQuote()
                QuoteUiState.Success(q.text, q.author)
            } catch (e: Exception) {
                QuoteUiState.Error(e.message ?: "Could not load quote")
            }
        }
    }
}

  4. Render state in Compose

    Collect the state lifecycle-aware and branch on it. Run it: you'll see a spinner, then the quote, then — turn off Wi-Fi and press the button — the error state.

    Kotlin
    @Composable
fun QuoteScreen(viewModel: QuoteViewModel) {
    val state by viewModel.state.collectAsStateWithLifecycle()
    Column(Modifier.padding(24.dp)) {
        when (val s = state) {
            QuoteUiState.Loading -> CircularProgressIndicator()
            is QuoteUiState.Success -> {
                Text(text = "“${s.text}”", style = MaterialTheme.typography.headlineSmall)
                Text(text = "— ${s.author}", style = MaterialTheme.typography.bodyMedium)
            }
            is QuoteUiState.Error -> Text("Error: ${s.message}", color = MaterialTheme.colorScheme.error)
        }
        Button(onClick = viewModel::refresh, modifier = Modifier.padding(top = 16.dp)) {
            Text("New quote")
        }
    }
}

  5. Prove it survives rotation

    Run, fetch a quote, then rotate the emulator (Ctrl+F11 / Cmd+←→). The quote stays because the ViewModel outlived the recreated UI. This is the "aha" that ties state + architecture together.

    💡
    The takeaway

    Notice you never wrote any "save on rotate" code. You put the state in the right layer — the ViewModel — and got correctness for free. That is the whole architecture lesson in one breath.

Hands-On — Prepare and Deliver Your Demo

This is your shipping moment. You will present your final project, show your five required features working live, walk through the one piece of code you are proudest of, and answer questions about your architecture. The work below is your self-prep kit: a rehearsal flow you run on your own before demo day, the feedback you should actively seek, and the questions you should be ready to answer.

What a strong demo proves

By the time you present, you should be able to show — by using your app, not just describing it — that you have: ≥2 screens with navigation, a state-driven Compose UI, local persistence (Room or DataStore), data pulled from a web API, and a ViewModel + repository behind it. Your demo is the live evidence that the data path Retrofit/Room → repository → ViewModel → Compose actually runs end-to-end.

Rehearse your presenter flow (run this solo first)

Practice this out loud at least once before demo day — speaking it aloud surfaces the gaps that reading it silently hides. Keep each beat tight so the whole walkthrough stays focused.

  1. Set the stage

    Open with your app name and one sentence: what it does and who it is for. Get it running on the emulator or device on screen before you start talking, so your first impression is a working app, not a build log.

  2. Tour your feature checklist

    Walk your five required features by using the app, narrating as you go: navigate between screens, change state in the UI, save and reload local data, pull data from the API, and name the ViewModel + repository working behind the scenes. Rehearse the exact taps in order so you never hunt for a button live.

  3. Code walkthrough

    Pick the single most interesting file ahead of time — usually your ViewModel's data flow or your repository — and open it. Explain how data travels from network/DB into the screen. Decide now which 10–15 lines you will point at so you are not scrolling during the demo.

  4. Reflect

    Prepare two honest sentences: "The hardest thing I solved was ___." and "If I had another week, I'd add ___." Naming a real obstacle and a real next step shows engineering judgment, not just a finished screen.

  5. Invite feedback and questions

    Close by asking for the feedback you actually want (see below), then field a question or two about your architecture. Have your answers to the question list ready so this feels like a conversation, not a quiz.

Have a backup plan ready

⚠️
Demo-day contingencies to prepare in advance
  • Slow Gradle build? Open the project and build once before you present so the app is already installed and ready to launch.
  • App crashes on launch? The usual cause is a missing INTERNET permission in the manifest — verify it ahead of time, and know how to read the Logcat stack trace if something fails.
  • API returns nothing or the network is flaky? Record a short screen capture of your app working beforehand, so you can fall back to it and still walk your architecture.
  • Emulator won't boot? Have a physical device ready over USB, or use your recording. Don't rely on a single environment working live.
  • Nerves? Rehearsing the flow out loud once or twice is the single best fix — a script you have spoken before is hard to lose.

Optional Course-Synthesis Check

A quick self-check you can run any time to test your grasp of the whole course. It is not the Post-Test — that lives at assessments/post-test.html.

Master it — assemble the whole app from its parts

Every screen you have ever built in this course is the same three moving parts wearing different costumes: a repository that owns where data comes from, a reducer that folds events into immutable state, and a mapper that turns a data result into a four-state UiState the UI can render with one when. The capstone is not a new pattern — it is these three composed. Prove you can wire them together with no UI noise in the way, and you can build any screen in any of these apps.

Worked example — a screen reduced to pure logic

Here is the reasoning a senior engineer runs before touching Compose. We will build a "saved articles" screen end-to-end as data and functions only, because if the logic is right, the UI is a 15-line when. Follow each step; the reasoning is the lesson.

  1. Step 1 — Name every state the screen can be in (sealed model)

    A screen is never just "has data" / "no data". From Session 7 onward the default is four states: Loading, Success, Empty, Error. Modeling them as a sealed interface means the compiler forces you to handle every branch — a forgotten state becomes a compile error, not a blank screen in production.

    Kotlin
    sealed interface ArticlesUiState {
    data object Loading : ArticlesUiState
    data class Success(val titles: List<String>) : ArticlesUiState
    data object Empty : ArticlesUiState          // loaded fine, but zero rows
    data class Error(val message: String) : ArticlesUiState
}

  2. Step 2 — Let the repository own "where data comes from" and return a Result

    The repository is the only layer that knows about cache vs. network. It returns Result<T> — never throws into the ViewModel, never exposes Retrofit or Room. Offline-first means: serve the cache, then try to refresh; a network failure on top of a populated cache is not a user-facing error.

    Kotlin
    // A tiny in-memory stand-in for Room + Retrofit, so the LOGIC is testable.
class ArticleRepository(
    private val cache: MutableList<String> = mutableListOf()
) {
    fun cached(): List<String> = cache.toList()

    // refresh() merges a network result into the cache (offline-first).
    fun refresh(networkResult: Result<List<String>>): Result<List<String>> {
        networkResult.onSuccess { fresh ->
            for (t in fresh) if (t !in cache) cache.add(t)   // de-dup merge
        }
        // If the cache has anything, callers can still render it on failure.
        return networkResult.map { cache.toList() }
    }
}

  3. Step 3 — Map a Result into UiState (the mapper)

    This is the single most reused function in the whole course. It is the only place that decides "success-but-empty → Empty". Notice an IOException (offline) and an HttpException-style failure both become Error here, but the repository already decided whether to surface them at all.

    Kotlin
    fun mapToUiState(result: Result<List<String>>): ArticlesUiState =
    result.fold(
        onSuccess = { list ->
            if (list.isEmpty()) ArticlesUiState.Empty
            else ArticlesUiState.Success(list)
        },
        onFailure = { e -> ArticlesUiState.Error(e.message ?: "Unknown error") }
    )

  4. Step 4 — Reduce events into the next state (UDF)

    The reducer is a pure function (state, event) -> state. It is the heart of unidirectional data flow: the UI emits events, the reducer is the only thing that mutates "state". Because it is pure, you can test the entire screen behavior with no emulator, no Compose, no coroutines.

    Kotlin
    sealed interface ArticlesEvent {
    data object StartedLoading : ArticlesEvent
    data class Loaded(val ui: ArticlesUiState) : ArticlesEvent
}

fun reduce(state: ArticlesUiState, event: ArticlesEvent): ArticlesUiState =
    when (event) {
        ArticlesEvent.StartedLoading -> ArticlesUiState.Loading
        is ArticlesEvent.Loaded      -> event.ui
    }

  5. Step 5 — Compose the three parts into one flow and run it

    Now the payoff: a "load" is just reduce(Loading), then repository.refresh(), then mapToUiState(...), then reduce(Loaded(...)). The harness below runs this for three real scenarios. Predict the printed output for each case, then press Run to confirm.

    Kotlin · runnable
    import java.io.IOException

sealed interface ArticlesUiState {
    data object Loading : ArticlesUiState
    data class Success(val titles: List<String>) : ArticlesUiState
    data object Empty : ArticlesUiState
    data class Error(val message: String) : ArticlesUiState
}

class ArticleRepository(private val cache: MutableList<String> = mutableListOf()) {
    fun cached(): List<String> = cache.toList()
    fun refresh(networkResult: Result<List<String>>): Result<List<String>> {
        networkResult.onSuccess { fresh -> for (t in fresh) if (t !in cache) cache.add(t) }
        return networkResult.map { cache.toList() }
    }
}

fun mapToUiState(result: Result<List<String>>): ArticlesUiState =
    result.fold(
        onSuccess = { if (it.isEmpty()) ArticlesUiState.Empty else ArticlesUiState.Success(it) },
        onFailure = { e -> ArticlesUiState.Error(e.message ?: "Unknown error") }
    )

// Compose the three parts: state in -> loading -> refresh -> map -> state out.
fun load(repo: ArticleRepository, network: Result<List<String>>): ArticlesUiState {
    val loading: ArticlesUiState = ArticlesUiState.Loading      // step: StartedLoading
    println("  during load, UI shows: $loading")
    return mapToUiState(repo.refresh(network))                 // step: Loaded(...)
}

fun main() {
    // Case A: network succeeds with rows -> Success
    val a = load(ArticleRepository(), Result.success(listOf("Compose 101", "Room CRUD")))
    println("A -> $a")

    // Case B: network succeeds but returns nothing -> Empty (NOT Error)
    val b = load(ArticleRepository(), Result.success(emptyList()))
    println("B -> $b")

    // Case C: offline, but cache already had a row -> still Success (offline-first)
    val warm = ArticleRepository(mutableListOf("Cached article"))
    val c = load(warm, Result.failure(IOException("No connection")))
    // refresh() returned failure, so map -> Error; but cache survived:
    println("C result = $c, cache still = ${warm.cached()}")
}
💡
Read the result of Case C carefully

The mapper turned the failed refresh into Error — but the cache still holds "Cached article". That gap is exactly the offline-first decision a senior engineer makes: surface the cache to the user and treat the network failure as a quiet background event, not a red error screen. Your repository decides; your mapper only reflects what it is handed.

Common misconceptions & pitfalls

These are the wrong mental models that survive all the way to a final project demo. Each one has a specific symptom you can recognize.

Wrong model: the final app is some advanced thing beyond the weekly material.

Why it's wrong: the project is composition, not invention. It is Session 8's reduce/StateFlow, Session 9's Room Flow, Session 10's Retrofit + four-state map, and Session 11's repository — wired together. There is no new concept; there is only assembly.

Correct model: draw the data path before coding — Retrofit/Room → Repository (Result) → ViewModel (reduce → UiState) → Compose (when). Every feature is a new instance of that same pipe.

Symptom: students who hold the wrong model freeze at the start and over-build; students who hold the right one ship a thin vertical slice (one screen, one API call, one table) on day one and grow it.

Wrong model: offline ⇒ blank/error screen, every time.

Why it's wrong: if the local database (the source of truth) already holds rows, a failed refresh should not wipe the screen. Conflating "the network call failed" with "there is no data" throws away the whole point of offline-first.

Correct model: the cache is the source of truth; the network only refreshes it. Render Success(cache) and surface the failure quietly (a snackbar), not as a full Error state that hides good data. Reserve Error for "failed and the cache is empty".

Symptom: the app shows a red error screen the instant Wi-Fi drops, even though it displayed a full list a second earlier.

Wrong model: the repository is boilerplate I can skip on a small app.

Why it's wrong: when the ViewModel imports retrofit2 and androidx.room, it now knows three jobs (where data lives, how to merge cache+network, and how to shape UI state). That class becomes impossible to reason about or test, and the cache-vs-network decision leaks into UI logic.

Correct model: the repository owns the data-source decision and returns a clean Result<DomainModel>; the ViewModel only reduces that into UiState. The ViewModel never imports Retrofit or Room.

Symptom: the ViewModel has import retrofit2.* at the top and 200 lines mixing JSON, SQL, and when branches. In a demo, the "where does your data come from?" question gets a tangled answer.

Wrong model: put a mutable var isLoading or a pre-formatted display string on the repository / entity.

Why it's wrong: the data layer must be a pure source of facts. Mutable presentation state living below the ViewModel means two screens observing the same repo fight over it, and Room/Flow emissions stop being trustworthy. Derived values belong where they're shown, computed from state.

Correct model: the data layer exposes immutable domain data; the ViewModel computes UI state (loading flags, formatted strings) when it reduces; Compose computes pure display values inline or with derivedStateOf.

Symptom: a value updates on one screen but is stale on another, or a "loading" flag gets stuck on because two callers toggled the same shared field.

Wrong model: in-memory state held with plain remember (or a local var) lives through a configuration change.

Why it's wrong: rotation recreates the Activity and the whole composition; remember is rebuilt from scratch, so its value is gone. Only rememberSaveable (small UI state via Bundle) or a ViewModel (screen state, async work) outlives recreation.

Correct model: screen state and any in-flight network/DB work live in the ViewModel; the Composable merely collectAsStateWithLifecycle() and renders. You never write "save on rotate" code — you put state in the right layer and get correctness for free.

Symptom: the fetched list, the typed text, or the loading spinner vanishes and re-fetches every time the device rotates.

In production — how a senior engineer ships this

🏭
The idiomatic, current pattern

A senior engineer builds the capstone as a thin vertical slice first: one screen, one endpoint, one Room table, wired end-to-end (network → repo → ViewModel → Compose) before adding a second feature. The repository is the single source of truth — it exposes a Room Flow via stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), initial) so the UI keeps its data across config changes but the upstream collection stops 5 seconds after the screen leaves the foreground. Network calls are wrapped in runCatching { } (or a typed catch (IOException) = retryable vs catch (HttpException) = don't blind-retry), returned as Result<T>, and the ViewModel reduces that into a sealed four-state UiState rendered by a single exhaustive when.

One tasteful gotcha: the 5_000ms in WhileSubscribed is not decoration. With 0, a rotation momentarily unsubscribes and you re-run the whole upstream (re-query, re-fetch) on every rotate; with a too-long timeout you keep collectors alive and leak work after the user navigates away. The 5-second window is the idiomatic balance that survives a rotation without re-fetching. It is the kind of detail that separates "it works on my emulator" from "it's correct in production."
