1 · The repository pattern — one front door for data CLO 4

Until now the ViewModel might call Retrofit directly. That works for one screen, but it couples your UI logic to a specific network library and gives you nowhere to add caching, retries, or a second data source. The repository is a plain Kotlin class that sits between the ViewModel and the data sources. The ViewModel asks the repository for data and does not know or care whether it came from the network or the local database.

🏪
Analogy

Think of the repository as the front desk of a library. You ask for a book; the desk decides whether to hand you one off the local shelf (Room cache) or order it from another branch (network). You never walk into the stockroom yourself. Swap the stockroom and the front desk's interface to you stays identical.

Compose UI — observes uiState, renders Loading/Success/Error/Empty, sends events up.
↑ StateFlow<UiState> ↓ events
ViewModel — holds UI state, calls the repository, maps domain data → UI state.
↑ Flow<List> ↓ refresh()
Repository — single front door. Reads from Room, refreshes from network, writes back to Room.
↕
Local: Room DAO (cache, source of truth)  ·  Remote: Retrofit API (network)

A minimal repository interface plus an implementation that holds the two data sources:

kotlin
// Domain model the rest of the app uses (independent of network/db shapes).
data class Article(val id: Long, val title: String, val body: String)

// The contract the ViewModel depends on — easy to fake in tests.
interface ArticleRepository {
    fun observeArticles(): Flow<List<Article>>   // stream from the local cache
    suspend fun refresh()                          // pull network -> write cache
}

// The real implementation owns BOTH data sources. Constructor injection
// keeps it testable: pass fakes in unit tests, real ones in the app.
class DefaultArticleRepository(
    private val api: ArticleApiService,   // Retrofit (remote)
    private val dao: ArticleDao,          // Room (local cache)
) : ArticleRepository {

    // Read path: UI observes the database. The DB is the single source of truth.
    override fun observeArticles(): Flow<List<Article>> =
        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    // Write path: fetch from network, then overwrite the cache.
    override suspend fun refresh() {
        val networkArticles = api.getArticles()          // suspend GET
        dao.upsertAll(networkArticles.map { it.toEntity() })
    }
}

2 · Offline-first: Room is the source of truth, network is a refresher CLO 4 CLO 6

"Offline-first" means the UI always reads from the local database, never directly from the network. The network's only job is to keep the database fresh. The payoff: open the app with no signal and you still see your last data instantly; when the network returns, the cache updates and — because the UI observes the database via Flow — the screen updates automatically. No manual "now re-fetch and setState."

📰
Analogy

It's a newspaper on your doorstep. You always read the paper that's there (the cache). A delivery person quietly swaps in a fresher edition when one is available (the network refresh). You never stand on the curb waiting for the printing press.

Why Flow is the right pipe between Room and the UI: a Room DAO query that returns Flow re-emits automatically every time the underlying table changes. So our two paths are decoupled — refresh() just writes to the table; the read path notices and pushes new data downstream.

kotlin · don't do this for real apps
// ViewModel calls the network directly and stores the result in memory only.
class FeedViewModel(private val api: ArticleApiService) : ViewModel() {
    var articles by mutableStateOf<List<Article>>(emptyList())
        private set

    fun load() {
        viewModelScope.launch {
            articles = api.getArticles().map { it.toDomain() }  // gone on process death
        }
    }
}
// Problems: blank screen with no network; data lost when the OS kills the
// process; every screen re-implements its own fetch + error handling.
kotlin · offline-first
// UI observes the DB (always something to show). Network only refreshes cache.
class FeedViewModel(private val repo: ArticleRepository) : ViewModel() {

    // Cached stream -> mapped into UI state -> exposed as StateFlow for Compose.
    val uiState: StateFlow<FeedUiState> =
        repo.observeArticles()
            .map { list -> if (list.isEmpty()) FeedUiState.Empty else FeedUiState.Success(list) }
            .catch { e -> emit(FeedUiState.Error(e.message ?: "Unknown error")) }
            .stateIn(
                scope = viewModelScope,
                started = SharingStarted.WhileSubscribed(5_000),
                initialValue = FeedUiState.Loading,
            )

    init { refresh() }

    fun refresh() {
        viewModelScope.launch {
            try { repo.refresh() }              // updates cache -> UI updates itself
            catch (e: IOException) { /* keep showing cache; surface a snackbar */ }
        }
    }
}
ℹ️
Why stateIn(WhileSubscribed)?

stateIn turns a cold Flow into a hot StateFlow with a current value Compose can read immediately. WhileSubscribed(5_000) keeps the upstream alive for 5 seconds after the last collector leaves, so a quick rotation doesn't restart the database query. initialValue = Loading gives the UI something to show on the very first frame.

Read the two error paths carefully. The .catch above guards the cache/DB read path (repo.observeArticles()) — a failure reading the database becomes Error. A failed network refresh is caught separately inside refresh() and deliberately does not replace cached data — that is the offline-first invariant. Don't read .catch as "this handles network errors"; it doesn't.

3 · Loading / Success / Error / Empty — model state, don't guess it CLO 5

A polished app never shows a blank white screen. Every screen that loads data has at least four states. Model them explicitly with a sealed interface so the compiler forces you to handle each branch in your when. This is the single biggest UX upgrade most student projects need before the final demo.

Loading

First load, nothing cached yet. Show a spinner or skeleton.

Success

We have data. Render the list/detail.

Empty

Request succeeded but there's nothing to show. Friendly "no items" message — not a spinner forever.

Error

Request failed. Show the reason + a Retry button. Keep cached data if you have it.
kotlin
// One sealed type the UI can exhaustively switch on.
sealed interface FeedUiState {
    data object Loading : FeedUiState
    data object Empty : FeedUiState
    data class Success(val articles: List<Article>) : FeedUiState
    data class Error(val message: String) : FeedUiState
}

Rendering each branch in Compose — note the when has no else, so adding a new state later is a compile error until you handle it:

kotlin · compose · material3
@Composable
fun FeedScreen(
    uiState: FeedUiState,
    onRetry: () -> Unit,
    modifier: Modifier = Modifier,
) {
    when (uiState) {
        FeedUiState.Loading -> Box(modifier.fillMaxSize(), Alignment.Center) {
            CircularProgressIndicator()
        }
        FeedUiState.Empty -> Box(modifier.fillMaxSize(), Alignment.Center) {
            Text("Nothing here yet.", style = MaterialTheme.typography.bodyLarge)
        }
        is FeedUiState.Error -> Column(
            modifier.fillMaxSize().padding(24.dp),
            verticalArrangement = Arrangement.Center,
            horizontalAlignment = Alignment.CenterHorizontally,
        ) {
            Text("Couldn't load: ${uiState.message}")
            Spacer(Modifier.height(12.dp))
            Button(onClick = onRetry) { Text("Retry") }
        }
        is FeedUiState.Success -> LazyColumn(modifier.fillMaxSize()) {
            items(uiState.articles, key = { it.id }) { article ->
                ListItem(
                    headlineContent = { Text(article.title) },
                    supportingContent = { Text(article.body, maxLines = 2) },
                )
                HorizontalDivider()
            }
        }
    }
}
⚠️
Common misconception

Students conflate "Loading" with "Empty." A finished request that returns zero rows is Empty, not Loading — showing a spinner forever is the #1 "my app is frozen" bug report. Make the distinction explicit: Loading ends the moment the first emission arrives, even if that emission is an empty list.

4 · Final-project integration checklist & common pitfalls CLO 4 CLO 5 CLO 6

The final project must demonstrate all five required features (a)–(e). This is the exact checklist students self-audit against during the studio; it mirrors the Final Project rubric demoed in Session 12. The studio steps below are labelled by the same letters so a student can map "what I just built" straight to "what the rubric grades."

The five required features the final project must demonstrate

a
Multi-screen Compose UI with navigation + a real back stack. At least two destinations (e.g. a list and a detail), wired with a NavHost so the system Back button returns the user the way they came. CLO 5
b
State-driven rendering — Loading / Success / Error / Empty. The screen never shows a blank frame: every data path is one of the four modelled states, rendered from a sealed UiState with an exhaustive when. CLO 5
c
Local persistence (Room or DataStore). At least one piece of data survives a process restart — a cached list in Room, or a key piece of state in DataStore. Prove it: kill and relaunch the app, the data is still there. CLO 4
d
Network / API call via Retrofit. A real suspend GET against a live (or mock) endpoint, parsed into a DTO with a JSON converter, with the INTERNET permission declared. CLO 6
e
A repository tying the data sources together. One repository class is the single front door: the ViewModel depends on it (no Retrofit/Room imports in the ViewModel), and it reads from the cache while refreshing from the network. CLO 4

The five integration pitfalls that eat student time

1
Forgot INTERNET permission. Symptom: SecurityException: Permission denied (missing INTERNET permission?). Fix: add <uses-permission android:name="android.permission.INTERNET" /> to AndroidManifest.xml.
2
Networking on the main thread / NetworkOnMainThreadException. Fix: Retrofit suspend functions called inside viewModelScope.launch (a background dispatcher). Never block the UI thread.
3
JSON field names don't match the data class. Symptom: every field is null/default. Fix: match property names to the JSON, or annotate (@SerialName / @Json(name=...)). Log the raw body once to compare.
4
Room schema changed without a migration. Symptom: IllegalStateException: Room cannot verify the data integrity. For coursework, .fallbackToDestructiveMigration() is acceptable; explain that production needs real migrations.
5
Collecting a Flow wrong in Compose. Fix: val state by viewModel.uiState.collectAsStateWithLifecycle() — lifecycle-aware, stops collecting when the screen is in the background.

⌨️ Guided Live-Coding — Build the Retrofit → Room → Flow repository

Goal: a tiny repo that fetches a list over the network, caches it in Room, and exposes a Flow the ViewModel observes. Build & run after each step. Companion lab: Retrofit + Flow.

  1. Add the dependencies & the INTERNET permission

    In the module build.gradle.kts. Then add the permission to the manifest and Sync.

    kotlin · build.gradle.kts
    dependencies {
    // Networking — Retrofit 3.x is binary-compatible with 2.x and now declares a
    // transitive Kotlin/OkHttp dependency, so most projects just bump the number.
    implementation("com.squareup.retrofit2:retrofit:3.0.0")
    implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.11.0")
    implementation("com.squareup.retrofit2:converter-kotlinx-serialization:3.0.0")

    // Local cache (Room 2.8.x — stay on 2.8 for the course; androidx.room3 isn't stable yet)
    implementation("androidx.room:room-runtime:2.8.4")
    implementation("androidx.room:room-ktx:2.8.4")   // Flow + suspend support
    ksp("androidx.room:room-compiler:2.8.4")

    // Lifecycle / Compose state
    implementation("androidx.lifecycle:lifecycle-runtime-compose:2.8.7")
    implementation("androidx.lifecycle:lifecycle-viewmodel-compose:2.8.7")
}
    xml · AndroidManifest.xml
    <manifest xmlns:android="http://schemas.android.com/apk/res/android">
    <uses-permission android:name="android.permission.INTERNET" />
    <application ... > ... </application>
</manifest>

    You should see: Gradle syncs with no errors. Make sure the KSP plugin is applied at the top of the file (id("com.google.devtools.ksp")) — remember the KSP version prefix must match your Kotlin version (e.g. Kotlin 2.2.20 → KSP 2.2.20-2.0.2).

    Version note (mid-2026)

    We pin Retrofit 3.0.0 (binary-compatible with 2.x — the bump is usually just the number) and Room 2.8.4 with KSP, not kapt. Room is being modernized as androidx.room3 (Kotlin-only, KMP, coroutine-first), but it isn't stable yet, so the course stays on the 2.8.x line — don't migrate your lab to room3.

  2. Define the network DTO + Retrofit service

    The DTO mirrors the JSON exactly. The service exposes a suspend GET.

    kotlin
    @Serializable
data class ArticleDto(
    val id: Long,
    val title: String,
    val body: String,
)

interface ArticleApiService {
    @GET("articles")
    suspend fun getArticles(): List<ArticleDto>
}

    You should see: nothing visual yet — but focus on the suspend keyword. That single word is what keeps the network call off the main thread.

  3. Define the Room entity, DAO (returning Flow), and database

    The DAO's observeAll() returns a Flow — this is the live pipe. @Upsert inserts new rows and updates existing ones.

    kotlin
    @Entity(tableName = "articles")
data class ArticleEntity(
    @PrimaryKey val id: Long,
    val title: String,
    val body: String,
)

@Dao
interface ArticleDao {
    @Query("SELECT * FROM articles ORDER BY title")
    fun observeAll(): Flow<List<ArticleEntity>>     // re-emits on every table change

    @Upsert
    suspend fun upsertAll(items: List<ArticleEntity>)
}

@Database(entities = [ArticleEntity::class], version = 1, exportSchema = false)
abstract class AppDatabase : RoomDatabase() {
    abstract fun articleDao(): ArticleDao
}

    You should see: the project still builds. Open the Database Inspector placeholder so you know where to watch rows appear.

  4. Write the mappers (DTO ↔ Entity ↔ Domain)

    Keep network shapes, DB shapes, and the domain model separate. Mappers are boring on purpose — they're the seams that let each layer change independently.

    kotlin
    fun ArticleDto.toEntity() = ArticleEntity(id = id, title = title, body = body)
fun ArticleEntity.toDomain() = Article(id = id, title = title, body = body)

    You should see: these resolve the toEntity() / toDomain() references used by the repository in the next step.

  5. Assemble the repository

    This is the heart of the lesson. Read from Room as a Flow; refresh by fetching from the network and upserting into Room.

    kotlin
    class DefaultArticleRepository(
    private val api: ArticleApiService,
    private val dao: ArticleDao,
) : ArticleRepository {

    override fun observeArticles(): Flow<List<Article>> =
        dao.observeAll().map { rows -> rows.map { it.toDomain() } }

    override suspend fun refresh() {
        val fromNetwork = api.getArticles()
        dao.upsertAll(fromNetwork.map { it.toEntity() })
        // No UI call needed — observeAll() re-emits because the table changed.
    }
}

    You should see: the comment is the "aha" — writing to the cache is what updates the UI. Say it out loud until it sticks.

  6. ViewModel: turn the cache stream into UI state

    Map the domain list into FeedUiState and expose a StateFlow. Kick off a refresh in init.

    kotlin
    class FeedViewModel(private val repo: ArticleRepository) : ViewModel() {

    val uiState: StateFlow<FeedUiState> =
        repo.observeArticles()
            .map { if (it.isEmpty()) FeedUiState.Empty else FeedUiState.Success(it) }
            .catch { e -> emit(FeedUiState.Error(e.message ?: "Unknown error")) }
            .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), FeedUiState.Loading)

    init { refresh() }

    fun refresh() = viewModelScope.launch {
        try { repo.refresh() }
        catch (e: IOException) { /* offline: keep showing cache */ }
        catch (e: HttpException) { /* non-2xx: log + optional snackbar */ }
    }
}

    You should see: still no UI — but notice that on a cold start the list is empty, so the very first state is Empty until the network call lands, then it flips to Success.

  7. Wire it into Compose & run

    Collect the state lifecycle-aware and feed FeedScreen from the previous section.

    kotlin · compose
    @Composable
fun FeedRoute(viewModel: FeedViewModel) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()
    FeedScreen(uiState = uiState, onRetry = viewModel::refresh)
}

    You should see: a spinner/empty for a beat, then the list fills in. Open the Database Inspector — rows are now in the articles table.

  8. Prove offline-first (the payoff demo)

    With data loaded: toggle airplane mode inside the emulator, kill and relaunch the app. The list is still there — served from Room with no network. Turn airplane mode off, pull to refresh / hit Retry, and watch fresh rows replace the cache automatically.

Troubleshooting (read this before raising your hand)
  • SecurityException / permission denied: missing INTERNET permission in the manifest.

🧠 Master it — the offline-first repository, end to end

This is the keystone of the whole course: a repository that merges a network result into a local cache and exposes the cache as the single source of truth, plus a clean Loading / Success / Error / Empty state machine on top. Room and Retrofit aren't on this in-browser Kotlin runner, so we drill the pure-Kotlin core — the exact reasoning that makes the Android version correct. Master this logic and the framework code is just wiring.

Worked example — reason about it before you build it

Picture the read path running while the user scrolls. refresh() writes new rows into the cache; the UI is observing that cache; the screen updates itself. There is no "now go re-render" call. Walk the logic one decision at a time — this is the mental model the compiler can't give you:

  1. 1 · The cache is the source of truth, not the network

    The UI reads only from the cache. refresh()'s entire job is to mutate the cache. So "did the screen update?" reduces to one question: "did the cache change?" Keep that invariant and offline-first falls out for free — there is nothing on screen that the network owns directly.

  2. 2 · Merge by key, don't blindly replace

    A refresh that returns 3 fresh articles shouldn't wipe 50 cached ones. Upsert means update existing rows by primary key, insert the new ones. In Room that's @Upsert; in pure Kotlin it's "build a map keyed by id, write the network items over it, read the values back sorted." Same idea, no magic.

  3. 3 · A failed refresh must never erase what you already showed

    If the network throws, the cache is untouched, so the user keeps seeing yesterday's list. The error is a signal to the ViewModel ("couldn't refresh"), not a reason to clear data. This is the difference between a resilient app and one that flashes a blank screen the moment a tunnel eats the signal.

  4. 4 · Classify the failure — not every error means "offline"

    IOException = the request never reached a healthy server (no signal, DNS, timeout) → retry is reasonable. HttpException = you reached the server and it said no (404, 401, 500) → blindly retrying just hammers it. Senior code branches these; junior code catches Exception and prints "no internet" for a 401.

  5. 5 · Reduce the result to exactly one UI state

    Given "what the repository returned" you map to one of four states: a non-empty list → Success; a successful-but-empty list → Empty (a friendly message, not a spinner forever); a failure → Error; nothing decided yet → Loading. A sealed type makes the when exhaustive so you physically cannot forget a branch.

Here is that reasoning as complete, runnable Kotlin — the same shape as the Android repository, minus the framework. Predict the printed output, then press Run to confirm.

Kotlin · runnable — the repository core, no Android needed
// ---- Domain + the four-state model (identical to the real app) -------------
data class Article(val id: Long, val title: String)

sealed interface UiState {
    data object Loading : UiState
    data object Empty : UiState
    data class Success(val articles: List<Article>) : UiState
    data class Error(val message: String) : UiState
}

// A network result the repository hands back: success OR a classified failure.
sealed interface NetResult {
    data class Ok(val articles: List<Article>) : NetResult
    data object Offline : NetResult        // IOException-style: never reached server
    data class ServerError(val code: Int) : NetResult  // HttpException-style: reached, said no
}

// ---- The cache = single source of truth (stands in for Room) ---------------
class Cache {
    private val byId = linkedMapOf<Long, Article>()        // keyed by primary key
    fun upsertAll(items: List<Article>) {                  // merge, don't replace
        items.forEach { byId[it.id] = it }
    }
    fun observe(): List<Article> = byId.values.sortedBy { it.title }
}

// ---- The repository: refresh writes the cache; reads come from the cache ----
class ArticleRepository(private val cache: Cache) {
    // Returns whether the refresh succeeded; the CACHE is what actually updates the UI.
    fun refresh(net: NetResult): Result<Unit> = when (net) {
        is NetResult.Ok          -> { cache.upsertAll(net.articles); Result.success(Unit) }
        NetResult.Offline        -> Result.failure(IllegalStateException("offline"))
        is NetResult.ServerError -> Result.failure(IllegalStateException("HTTP ${net.code}"))
        // NOTE: on failure we DO NOT touch the cache — old data stays on screen.
    }
    fun observe(): List<Article> = cache.observe()
}

// ---- The ViewModel's job: turn "cache snapshot" into one UI state ----------
fun mapToUiState(cached: List<Article>, refreshFailed: Boolean): UiState = when {
    cached.isNotEmpty()  -> UiState.Success(cached)   // always prefer showing data
    refreshFailed        -> UiState.Error("Couldn't reach the server")
    else                 -> UiState.Empty             // succeeded, but nothing came back
}

fun main() {
    val repo = ArticleRepository(Cache())

    // 1) Cold start, network OK -> two articles, sorted by title -> Success
    repo.refresh(NetResult.Ok(listOf(Article(2, "Beta"), Article(1, "Alpha"))))
    println(mapToUiState(repo.observe(), refreshFailed = false))

    // 2) A later refresh goes offline -> cache UNTOUCHED -> still Success(Alpha, Beta)
    val offline = repo.refresh(NetResult.Offline)
    println("offline refresh failed? ${offline.isFailure} | still shows: ${mapToUiState(repo.observe(), refreshFailed = true)}")

    // 3) Merge a fresh item by key -> 3 articles, no data lost
    repo.refresh(NetResult.Ok(listOf(Article(3, "Gamma"))))
    println(mapToUiState(repo.observe(), refreshFailed = false))

    // 4) Empty cache + a successful-but-empty response -> Empty, NOT Loading forever
    val fresh = ArticleRepository(Cache())
    fresh.refresh(NetResult.Ok(emptyList()))
    println(mapToUiState(fresh.observe(), refreshFailed = false))
}

Four println lines, in order. The list shapes are Kotlin's real data class toString() — note the full Article(id=…, title=…) form, and that the Empty object prints just its name:

Common misconceptions & pitfalls

Each one is a real mental model students bring in — and the exact symptom it produces in a running app.

Why it's wrong: if the UI reads from the network, then every screen needs its own fetch + loading + error handling, and the moment the signal drops you have nothing to show. You also can't render anything until the round-trip completes.

Correct model: the cache (Room) is the source of truth; the network's only job is to keep the cache fresh. The UI observes the cache via Flow and updates itself whenever rows change.

Symptom you'd see: a blank screen on cold start with no signal; data that vanishes the instant the user walks into an elevator; duplicated fetch/error code on every screen.

Why it's wrong: a suspend fun getAll(): List<…> reads the table once. After refresh() writes new rows, the screen has a stale list and no idea anything changed, so you bolt on manual "now re-fetch and setState" calls everywhere — and forget half of them.

Correct model: return Flow<List<…>> from the DAO. Room re-emits the query result automatically on every write to that table, so the write path (refresh) and the read path are fully decoupled.

Symptom you'd see: "I refreshed but the list didn't change until I left the screen and came back."

Why it's wrong: it conflates two opposite situations. IOException means the request never reached a healthy server (offline, timeout) — retrying makes sense. HttpException means you did reach the server and it returned a 4xx/5xx — retrying a 401 or 404 just hammers a broken request and lies to the user.

Correct model: branch the catch. catch (e: IOException) → offline/retryable; catch (e: HttpException) → reached server, inspect e.code(), don't blind-retry; a generic catch last for everything else. The repository surfaces a Result<T>; the ViewModel maps it to a state.

Symptom you'd see: a user with full bars getting "Check your internet connection" because their auth token expired (a 401).

Why it's wrong: Loading and Empty are different facts. Loading = "I don't have an answer yet." Empty = "I have an answer and it's zero rows." Treating empty as loading produces the #1 "my app is frozen" bug report — a spinner that never stops.

Correct model: the moment the first emission arrives — even if it's an empty list — you leave Loading. A non-empty list → Success; an empty one after a successful load → Empty with a friendly message and (usually) a Retry.

Symptom you'd see: a perpetual spinner on a brand-new account that legitimately has no data yet.

Why it's wrong: the ViewModel becomes glued to a specific network library, can't be tested without a real server, and has nowhere to add caching, retries, or a second data source. You also end up holding mutable cache state in the data layer of every screen.

Correct model: the ViewModel depends on a single repository interface (no Retrofit/Room imports in the ViewModel at all). The repository owns both data sources via constructor injection, so you can pass fakes in a test and the real ones in the app. State flows down as an immutable StateFlow<UiState>; events flow up as function calls.

Symptom you'd see: a ViewModel with import retrofit2.* at the top, untestable without a live backend, and copy-pasted fetch logic across screens.

How a senior engineer actually wires this

The repository is constructor-injected with its data sources and is the only place that touches Retrofit and Room. The read path is a Flow straight off the DAO, lifted into the ViewModel with stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), UiState.Loading) so it survives configuration changes but tears down a few seconds after the screen goes away. The write path (refresh) runs in viewModelScope.launch, returns a Result<Unit>, and the ViewModel maps failures to a state — it never mutates UI state by hand. Heavy work (the network call, the Room write) runs on Dispatchers.IO; Retrofit's suspend functions and Room's suspend/Flow APIs are already main-safe, so you rarely switch dispatchers yourself. Compose collects with collectAsStateWithLifecycle() — never a bare collectAsState() — so it stops collecting in the background.

The tasteful gotcha: pin one JSON converter coordinate and reuse it everywhere — com.squareup.retrofit2:converter-kotlinx-serialization (shipped with Retrofit 3.x) with a single Json { ignoreUnknownKeys = true } instance. Set ignoreUnknownKeys = true so a backend adding a field next week doesn't crash your parser — but keep explicitNulls sane and prefer non-nullable fields with defaults so a missing value is a compile-time decision, not a 2 a.m. NullPointerException. And remember the unglamorous one: a @GET("photos") path with a BASE_URL that's missing its trailing slash silently resolves to the wrong URL. Versions drift; accept Android Studio's stable bump when it offers one.
