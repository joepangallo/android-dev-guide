1 · Why persist — and the storage spectrum

A running app holds state in memory (your ViewModel, Compose state). The moment Android reclaims the process — the user swipes the app away, the OS kills it for memory, or the device reboots — memory is wiped. Persistence means writing data somewhere durable on the device so it's still there next launch.

🧠
Analogy

Memory (RAM) is your desk — fast, but everything gets swept off when you go home. Persistence is the filing cabinet — slower to open, but your papers are there tomorrow. DataStore is a sticky note on the cabinet ("theme = dark"); Room/SQLite is the organized drawers of labeled folders you can search.

Android gives several options. Pick by the shape of the data:

Key-value

A handful of small, simple values: flags, prefs, last-opened id.

Preferences DataStore SharedPreferences (legacy)

Structured / relational

Many records you query, sort, filter, relate. The bulk of app data.

Room (over SQLite)

Files

Blobs: images, audio, exported documents, downloaded JSON caches.

Internal/external storage

Typed objects

A whole structured config object, schema-validated.

Proto DataStore

✅
Decision rule

"A few simple settings → DataStore. A collection of records you query → Room. Big binary blobs → files." Say it out loud until it's automatic — this is the single most testable idea today.

2 · Key-value: SharedPreferences → Preferences DataStore

SharedPreferences is the original key-value store. It works, but it has real problems: its apply()/commit() API can block the UI thread, it has no built-in error signaling, and reads are synchronous. DataStore is the modern replacement: it's asynchronous, uses Kotlin coroutines + Flow, and is transactional and safe.

Declare a single top-level DataStore, define typed keys, then expose a read Flow and a suspend write. Reads emit automatically whenever the value changes.

🧩
One-sentence definition

by preferencesDataStore(name = "settings") is a delegated extension property on Context that hands the whole app one shared DataStore — a coroutine-based key-value store you read as a Flow (an auto-updating stream) and write with a suspend function.

Kotlin 
// SettingsRepository.kt
import android.content.Context
import androidx.datastore.preferences.core.booleanPreferencesKey
import androidx.datastore.preferences.core.edit
import androidx.datastore.preferences.core.emptyPreferences
import androidx.datastore.preferences.preferencesDataStore
import androidx.datastore.preferences.core.Preferences
import kotlinx.coroutines.flow.Flow
import kotlinx.coroutines.flow.catch
import kotlinx.coroutines.flow.map
import java.io.IOException

// One DataStore per file name, declared at the top level (one instance per process).
private val Context.dataStore by preferencesDataStore(name = "settings")

class SettingsRepository(private val context: Context) {

    private object Keys {
        val DARK_MODE = booleanPreferencesKey("dark_mode")
    }

    // READ: a Flow that emits the current value and every future change.
    val darkMode: Flow<Boolean> = context.dataStore.data
        .catch { e ->
            // DataStore surfaces read errors as IOException — emit a safe default.
            if (e is IOException) emit(emptyPreferences()) else throw e
        }
        .map { prefs -> prefs[Keys.DARK_MODE] ?: false }

    // WRITE: suspend function, runs off the main thread, transactional.
    suspend fun setDarkMode(enabled: Boolean) {
        context.dataStore.edit { prefs ->
            prefs[Keys.DARK_MODE] = enabled
        }
    }
}

Notice that darkMode is a stream. The UI doesn't "ask once" — it observes, so flipping the switch updates everywhere instantly.

Reading the DataStore code — chunk by chunk

  • private val Context.dataStore by preferencesDataStore(name = "settings") — a top-level delegated extension property. by preferencesDataStore(...) is the delegate that lazily creates exactly one DataStore instance (backed by the file settings.preferences_pb) for the entire process. Because it extends Context, you reach it as context.dataStore from anywhere — but it's still the same instance every time.
  • val DARK_MODE = booleanPreferencesKey("dark_mode") — a typed key. The boolean… prefix locks this key to Boolean; reading or writing it with the wrong type won't compile. The string "dark_mode" is the actual name stored on disk.
  • context.dataStore.data — this is a Flow<Preferences>: a stream that emits the whole current preferences snapshot now, and re-emits a fresh snapshot every time any value is written.
  • .catch { e -> if (e is IOException) emit(emptyPreferences()) else throw e } — DataStore surfaces a read/disk failure as an IOException inside the Flow. This operator recovers from that one case by emitting an empty snapshot (so the UI falls back to defaults instead of crashing); any other error is rethrown.
  • .map { prefs -> prefs[Keys.DARK_MODE] ?: false } — transforms each Preferences snapshot into the one value you care about. prefs[KEY] returns null if nothing was ever stored, and the elvis ?: false supplies the default. Result: darkMode is a clean Flow<Boolean>.
  • suspend fun setDarkMode(...) + dataStore.edit { settings -> settings[KEY] = value } — writing is a suspend function (it must be called from a coroutine, off the main thread). edit { } runs your mutation as one atomic transaction; the moment it commits, the read Flow above re-emits the new value.

Strip away the repository and the new idea — a Context extension property whose value comes from a delegate — is tiny on its own:

Kotlin · the delegated extension property in isolation 
// Declared once, at the top level of a file — NOT inside a class.
val Context.dataStore by preferencesDataStore(name = "settings")

// Anywhere you have a Context, you read the SAME instance:
val a = applicationContext.dataStore   // ┐ both names point at
val b = someActivity.dataStore         // ┘ the one process-wide store

// Reading is a stream; writing is a coroutine call:
val flow: Flow<Preferences> = context.dataStore.data        // observe
suspend fun save() { context.dataStore.edit { it[KEY] = 1 } } // mutate
⚠️
Common confusion

This is not SharedPreferences. There is no blocking getString("dark_mode", …) you call on the main thread — a read is a Flow you collect, and a write is a suspend edit { } you call from a coroutine. Also, declare by preferencesDataStore(...) once at the top level: creating a second DataStore over the same file name throws at runtime, which is exactly why it's a single delegated property and not a new-ed object.

This is here only for contrast. It's what you may have seen online; the point is to understand why you don't reach for it in new code.

// write

// read (synchronous — happens on whatever thread you call it)

3 · Structured data: SQLite under the hood, Room on top

Every Android device ships with SQLite — a tiny, self-contained relational database stored as a single file in your app's private storage. You can talk to it with raw SQL and the SQLiteOpenHelper API, but that means hand-writing string SQL, manually mapping cursor columns to objects, and zero compile-time checking.

Room is Google's recommended abstraction layer over SQLite. You describe your data with annotations; Room generates the SQL and the boilerplate, and validates your queries at compile time. A typo in a column name fails the build, not the app at runtime.

🏗️
Three pieces of Room

@Entity = a table (one class = one table, fields = columns). @Dao = the queries (an interface of read/write methods). @Database = the connection that ties entities + DAOs together and hands you the database instance.

Kotlin 
import androidx.room.Entity
import androidx.room.PrimaryKey

@Entity(tableName = "notes")
data class NoteEntity(
    @PrimaryKey(autoGenerate = true) val id: Int = 0,
    val title: String,
    val body: String,
    val createdAt: Long = System.currentTimeMillis()
)

Each property becomes a column. autoGenerate = true lets SQLite assign the id. A data class is ideal — it's just the row.

Kotlin 
import androidx.room.*
import kotlinx.coroutines.flow.Flow

@Dao
interface NoteDao {

    // READ — returns a Flow: emits the whole list, re-emits on any change.
    @Query("SELECT * FROM notes ORDER BY createdAt DESC")
    fun observeAll(): Flow<List<NoteEntity>>

    // READ one (suspend = one-shot, no live updates)
    @Query("SELECT * FROM notes WHERE id = :id")
    suspend fun getById(id: Int): NoteEntity?

    // CREATE
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insert(note: NoteEntity): Long

    // UPDATE (matches on @PrimaryKey)
    @Update
    suspend fun update(note: NoteEntity)

    // DELETE
    @Delete
    suspend fun delete(note: NoteEntity)

    @Query("DELETE FROM notes")
    suspend fun clearAll()
}

Key point: write methods are suspend (one-shot, off-main-thread); the live read returns Flow (a stream that keeps emitting). Room enforces this — a non-suspend write that hits the DB on the main thread will crash.

🧩
One-sentence definition

This companion-object block is the thread-safe "build it once" singleton (the double-checked locking idiom): the first caller builds the expensive database, every later caller gets that same shared instance back, and two threads racing at startup can never build it twice.

Kotlin 
import android.content.Context
import androidx.room.Database
import androidx.room.Room
import androidx.room.RoomDatabase

@Database(entities = [NoteEntity::class], version = 1, exportSchema = false)
abstract class AppDatabase : RoomDatabase() {

    abstract fun noteDao(): NoteDao

    companion object {
        @Volatile private var INSTANCE: AppDatabase? = null

        fun get(context: Context): AppDatabase =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: Room.databaseBuilder(
                    context.applicationContext,
                    AppDatabase::class.java,
                    "app.db"
                ).build().also { INSTANCE = it }
            }
    }
}

The double-checked @Volatile singleton ensures one database instance per process — opening multiple is wasteful and can corrupt. applicationContext avoids leaking an Activity.

The singleton, line by line

  • @Volatile private var INSTANCE: AppDatabase? = null — the one cached database (or null before it's built). @Volatile guarantees that when one thread writes INSTANCE, every other thread immediately sees the new value instead of a stale cached copy — essential for the check that follows.
  • INSTANCE ?: synchronized(this) { … } — the outer elvis check. If INSTANCE is already built, return it instantly with no locking (the fast, common path). Only when it's still null do we pay the cost of entering the synchronized block.
  • synchronized(this) { … } — a lock that lets only one thread at a time run the block. Threads that arrive while one is building wait here rather than building their own copy.
  • INSTANCE ?: Room.databaseBuilder(...) — the inner elvis check, re-tested after acquiring the lock. A thread that was waiting may find another thread already built the DB; this re-check hands back that instance instead of building a second one. The outer + inner pair is the double-checked locking idiom.
  • .build().also { INSTANCE = it } — actually construct the database, then .also { } caches it into INSTANCE (for all future callers) and returns that same instance as the value of the expression.

The idiom isn't Room-specific — here it is on a plain expensive object, with the four moving parts isolated:

Kotlin · double-checked locking, on any expensive object 
@Volatile private var INSTANCE: Engine? = null      // 1. visible across threads

fun get(): Engine =
    INSTANCE ?: synchronized(this) {                // 2. fast path, else lock
        INSTANCE ?: Engine()                        // 3. re-check inside the lock
            .also { INSTANCE = it }                 // 4. cache + return it
    }
⚠️
Common confusion

You never call AppDatabase() yourself — it's abstract, and Room builds it via databaseBuilder(...).build(). Building a database is expensive, so it must be a single shared instance for the whole process, not one per screen. And in a real app you would almost never hand-write this boilerplate: a dependency-injection framework like Hilt creates the database once and provides it for you. We write it by hand here only so the mechanism — and why it has to be thread-safe — is no longer a mystery.

Gradle (Kotlin DSL) 
// Project-level build.gradle.kts — plugins { } (alias/version catalog style)
plugins {
    id("com.google.devtools.ksp") version "2.2.20-2.0.2" apply false
}

// app/build.gradle.kts
plugins {
    id("com.google.devtools.ksp")
}

dependencies {
    val room = "2.8.4"
    implementation("androidx.room:room-runtime:$room")
    implementation("androidx.room:room-ktx:$room")   // Flow + coroutines support
    ksp("androidx.room:room-compiler:$room")          // generates the code
}
⚠️
Version pitfall

The KSP version's prefix must match your Kotlin plugin version (e.g. Kotlin 2.2.20 → KSP 2.2.20-2.0.2). A mismatch is the most common build failure today. Use room-ktx (not just room-runtime) so Flow works.

4 · CRUD & observing with Flow through the architecture

The UI must never touch Room directly. The clean chain is: Compose UI → ViewModel → Repository → DAO → SQLite. The repository hides "where the data comes from," the ViewModel exposes UI-friendly state, and Compose just renders it.

Kotlin 
class NoteRepository(private val dao: NoteDao) {
    val notes: Flow<List<NoteEntity>> = dao.observeAll()
    suspend fun add(title: String, body: String) =
        dao.insert(NoteEntity(title = title, body = body))
    suspend fun remove(note: NoteEntity) = dao.delete(note)
    suspend fun edit(note: NoteEntity) = dao.update(note)
}
Kotlin 
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import kotlinx.coroutines.flow.SharingStarted
import kotlinx.coroutines.flow.stateIn
import kotlinx.coroutines.launch

class NotesViewModel(private val repo: NoteRepository) : ViewModel() {

    // Convert the cold Flow into UI state the UI can collect with lifecycle awareness.
    val notes = repo.notes.stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000),
        initialValue = emptyList()
    )

    fun addNote(title: String, body: String) = viewModelScope.launch {
        if (title.isNotBlank()) repo.add(title, body)
    }
    fun deleteNote(note: NoteEntity) = viewModelScope.launch { repo.remove(note) }
}

Writes launch in viewModelScope so the suspend DAO calls run off the main thread automatically. The read is a single source of truth — after an insert, the Flow re-emits and the list updates with no manual refresh.

Kotlin · Compose 
import androidx.compose.runtime.*
import androidx.lifecycle.compose.collectAsStateWithLifecycle
import androidx.compose.foundation.lazy.LazyColumn
import androidx.compose.foundation.lazy.items
import androidx.compose.material3.*

@Composable
fun NotesScreen(viewModel: NotesViewModel) {
    val notes by viewModel.notes.collectAsStateWithLifecycle()
    var title by remember { mutableStateOf("") }

    Column {
        OutlinedTextField(value = title, onValueChange = { title = it }, label = { Text("Title") })
        Button(onClick = { viewModel.addNote(title, ""); title = "" }) { Text("Add") }

        LazyColumn {
            items(notes, key = { it.id }) { note ->
                ListItem(
                    headlineContent = { Text(note.title) },
                    trailingContent = {
                        IconButton(onClick = { viewModel.deleteNote(note) }) {
                            Icon(Icons.Default.Delete, contentDescription = "Delete")
                        }
                    }
                )
            }
        }
    }
}

collectAsStateWithLifecycle() stops collecting when the screen isn't visible — battery- and lifecycle-safe. The key = { it.id } helps Compose animate list changes correctly.

🔁
The "single source of truth" payoff

Because the UI observes a Flow straight from the database, you never manually update the list after a write. Insert → table changes → Room re-emits → list redraws. Delete the same way. This is the aha moment: the database is the single source of truth, and the UI just follows it.

⚠️
Troubleshooting — the usual suspects
  • Build fails after adding Room: KSP version prefix ≠ Kotlin version. Align them. Confirm ksp(...) (not implementation(...)) is used for room-compiler.
  • "Cannot access database on the main thread": a write method is missing suspend, or you're calling it outside a coroutine. Wrap in viewModelScope.launch { }.
  • List never updates after Add: your read returns List, not Flow, or you forgot collectAsStateWithLifecycle(). The read must be a Flow you collect.
  • Data gone after restart: you're probably reading an in-memory list, not the DB. Make sure the UI is collecting the DAO's Flow, and that you used databaseBuilder (persistent), not inMemoryDatabaseBuilder.
  • Crash on second launch / "no such column": you changed the entity without bumping version or providing a migration. For class, bump the version and uninstall/reinstall to wipe the DB.
  • DataStore switch always false: you read .first() once instead of collecting the Flow, or your default in .map overrides the saved value.

🧠 Master it — local data that actually sticks

Today's deepest idea isn't "Room saves rows." It's that a Room read is a live Flow — the database becomes the single source of truth, and a suspend write that mutates the table makes the read re-emit on its own. No manual refresh. Below you reason through that mechanism on a tiny pure-Kotlin model of a DAO, then drive it to "all tests green" yourself.

Worked example — why an insert "appears by itself"

On a real Room DAO, observeAll(): Flow<List<NoteEntity>> and suspend fun insert(...) look unrelated — two methods on an interface. Walk the reasoning that links them:

  1. The read is a stream, not a snapshot. observeAll() returns a Flow. The UI collects it and keeps the latest list. It never "asks again."
  2. Room watches the table. Room registers an invalidation tracker on every table your query touches (here, notes).
  3. A write mutates that table. suspend fun insert(...) runs the SQL INSERT. Room's tracker fires "table notes changed."
  4. The Flow re-queries and re-emits. Room re-runs SELECT * FROM notes … and pushes the new list down the same Flow. Your collectAsStateWithLifecycle() swaps state → Compose recomposes the LazyColumn.

So the "magic" is just: one observable read + writes that touch the same table. Model that with an in-memory store and a hand-rolled observer, and the whole reactive loop fits in 30 lines:

Kotlin · runnable — predict the output, then Run to confirm
// A tiny model of a reactive DAO: an in-memory table + an "observable" read.
data class Note(val id: Int, val title: String)

class FakeNoteDao {
    private val rows = mutableListOf<Note>()
    private var nextId = 1
    private val observers = mutableListOf<(List<Note>) -> Unit>()

    // READ as a stream: register a listener, immediately emit the current list,
    // and re-emit on every future change. (Room does this for you via Flow.)
    fun observeAll(onChange: (List<Note>) -> Unit) {
        observers += onChange
        onChange(rows.toList())          // initial emission
    }
    private fun emit() { val snap = rows.toList(); observers.forEach { it(snap) } }

    // WRITE (suspend in real Room): mutate the table, then notify observers.
    fun insert(title: String): Int {
        val note = Note(nextId++, title); rows += note; emit(); return note.id
    }
    fun delete(id: Int) { rows.removeAll { it.id == id }; emit() }
}

fun main() {
    val dao = FakeNoteDao()
    val seen = mutableListOf<List<Note>>()
    dao.observeAll { list -> seen += list }   // UI starts collecting

    dao.insert("Buy milk")                    // table changes -> re-emit
    dao.insert("Walk dog")                    // table changes -> re-emit

    // The UI never re-asked. It only saw what the stream pushed:
    println("emissions: " + seen.size)        // 1 initial + 2 inserts = 3
    println("latest:    " + seen.last().map { it.title })
}

The collector recorded 3 emissions — the empty start plus one per insert — and never called a "refresh." That is exactly the contract Flow<List<…>> gives you for free in Room.

fun insert(text: String): Int {

    rows += t
    emit()
    return t.id
}

    val i = rows.indexOfFirst { it.id == task.id }
    if (i >= 0) rows[i] = task
    emit()
}

    emit()
}

Common misconceptions & pitfalls

Each is a wrong mental model people actually hold, why it bites, the correct model, and the symptom you'd see in a running app.

Why it's wrong: a suspend fun getAll(): List<Note> is a one-shot snapshot. It runs the query once, hands you a frozen list, and forgets you. Nothing re-runs it when the table changes.

Correct model: for a live list, the read must return Flow<List<Note>>. The Flow is wired to Room's invalidation tracker, so any write to that table re-emits a fresh list. Use one-shot suspend reads only for "fetch once" cases (e.g. getById to pre-fill an edit screen).

Symptom: you add an item, the database genuinely has the new row (you can see it in the Database Inspector), but the on-screen list doesn't change until you navigate away and back. The two-line fix is List → Flow on the read and collect it.

Why it's wrong: a non-suspend DAO write executed on the main thread crashes with IllegalStateException: Cannot access database on the main thread…, and even a suspend call from a composable body runs on every recomposition and outside any safe scope.

Correct model: writes are suspend and launched from a coroutine scope you own — almost always viewModelScope.launch { repo.add(...) }. Room dispatches the actual SQL onto a background executor for you. The composable only renders state and calls viewModel.addNote(...) in an onClick.

Symptom: instant crash on the first insert (main-thread access), or — subtler — a write fired repeatedly because you called it from the composable's body instead of a callback.

Why it's wrong: DataStore is asynchronous and reactive. .first() grabs a single value once and stops collecting. If the setting changes later (the user flips the switch), your UI never hears about it — and calling .first() from the wrong thread/scope can block or throw.

Correct model: expose the read as a Flow (val darkMode: Flow<Boolean>), turn it into UI state with stateIn(...), and collect with collectAsStateWithLifecycle(). Now the switch and any other screen stay in sync automatically. Reserve .first() for a true one-time read (e.g. a "has the user seen onboarding?" check at startup).

Symptom: the toggle works the first time but a second screen showing the same setting is stale, or the switch snaps back to its default because your .map { prefs[KEY] ?: false } default overran a value that hadn't loaded yet.

Why it's wrong: a mutable list held in the data layer is a second source of truth that drifts from the database. Mutating it in place (cache.add(...)) doesn't notify Compose (no new reference, no state change), and it doesn't survive process death because it's just memory.

Correct model: the database (its Flow) is the single source of truth. The repository exposes dao.observeAll() directly (or maps it), and writes go through the DAO, so the same Flow re-emits. No hand-maintained cache list.

Symptom: the list looks right while the app is open but is empty or wrong after a restart, and edits sometimes don't show because you mutated the list contents without producing a new list instance.

Why it's wrong: the on-device SQLite file still has the old schema. Room compares the compiled schema to the file's schema at open time; a mismatch with no migration throws IllegalStateException: … you must either provide a Migration … or call fallbackToDestructiveMigration().

Correct model: any schema change requires bumping version and supplying a Migration (production) — or, while prototyping in this course, fallbackToDestructiveMigration() / uninstalling the app to wipe the DB. Migrations are real production work; for now, understand why the crash happens.

Symptom: the app crashes on launch the run after you changed the entity — never the run you changed it on — with a "Room cannot verify the data integrity" / "no such column" style message.

In production — how a senior engineer wires this

🏭
The idiomatic, current pattern

A senior Android engineer treats the database as the single source of truth and lets the UI observe it, never poll it. Concretely:

  • One DAO read Flow → stateIn → collectAsStateWithLifecycle(). The read is Flow<List<Entity>>; the ViewModel converts it once with stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList()). WhileSubscribed(5_000) keeps the query alive through a rotation but stops it 5s after the screen goes away — no wasted DB work in the background.
  • Writes are suspend, launched in viewModelScope. The UI calls a plain function (vm.addNote(...)); the ViewModel does viewModelScope.launch { repo.add(...) }. Room moves the SQL off the main thread automatically — you don't hand-pick Dispatchers.IO for Room calls.
  • Map entities to domain/UI models at the repository seam. The composable should receive a clean UI model, not a raw @Entity. This is the same seam you'll reuse in Session 11 to merge network + DB.
  • A @Volatile double-checked singleton for the @Database built with applicationContext — one instance per process, no Activity leak.

One tasteful gotcha: don't expose a MutableStateFlow or the DAO's raw Flow straight to Compose and then also keep a local list — that's the "two sources of truth" trap. Expose one read-only StateFlow/observed list and route every change through a write. If the list ever looks stale after an insert, your instinct should be "I have a second source of truth somewhere," not "I need a manual refresh."
