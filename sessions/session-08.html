1 · Navigation in Compose: NavController, NavHost, routes & arguments

In Jetpack Compose, navigation is built from three pieces:

  • NavController — the brain. It owns the back stack (the history of where you've been) and exposes navigate(...) / popBackStack(). You create one with rememberNavController().
  • NavHost — the window. It's a composable bound to a NavController that swaps in whichever destination's content matches the current route.
  • Destinations / routes — the places. Each is a composable("route") { ... } entry. A route is just a string key (e.g. "home", "detail/{itemId}").
🧠
Analogy — a stack of cards

The back stack is a literal stack of index cards. Navigating pushes a new card on top; pressing Back pops the top card off and you see the one beneath. The NavHost only ever shows the top card. This is why "Back" from Detail returns you to the list: the list card was underneath the whole time.

A minimal three-destination graph:

Kotlin · Compose 
@Composable
fun AppNavHost(navController: NavHostController = rememberNavController()) {
    NavHost(navController = navController, startDestination = "home") {

        composable("home") {
            HomeScreen(
                onItemClick = { id -> navController.navigate("detail/$id") },
                onAboutClick = { navController.navigate("about") }
            )
        }

        // Route with a typed argument: detail/{itemId}
        composable(
            route = "detail/{itemId}",
            arguments = listOf(navArgument("itemId") { type = NavType.IntType })
        ) { backStackEntry ->
            val itemId = backStackEntry.arguments?.getInt("itemId") ?: 0
            DetailScreen(
                itemId = itemId,
                onBack = { navController.popBackStack() }
            )
        }

        composable("about") {
            AboutScreen(onBack = { navController.popBackStack() })
        }
    }
}
⚠️
Pass IDs, not objects

Navigation arguments are encoded into the route string, so they must be primitives (Int, String, Boolean…). Pass an id and let the destination look the data up from the ViewModel/repository. Never try to shove a whole data class into a route — it doesn't fit the model and breaks deep links and process death.

🛡️
Avoid duplicate destinations

Tapping a button twice quickly can push the same destination twice. Use launchSingleTop = true for tabs, and for "go home" actions use popUpTo(...) to trim the stack. We demo this in the live-coding bottom-nav extension.

2 · Single-Activity architecture & the Fragment context

Modern Compose apps are usually single-Activity: one MainActivity hosts one NavHost, and "screens" are just composables swapped in and out. Contrast that with how Android apps were built before Compose:

One Activity, one NavHost, destinations are @Composable functions. State lives in ViewModels. This is what the course, the project, and Unit 4 use.

Kotlin · Compose 
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            MyAppTheme {
                AppNavHost()   // the single NavHost for the whole app
            }
        }
    }
}

Pre-Compose, a screen was usually a Fragment (a reusable UI chunk hosted by an Activity), with its UI defined in an XML layout and screen-to-screen movement handled by the Navigation Component + a nav_graph.xml. You will meet this in older codebases and it's in the syllabus reading.

Kotlin · legacy Fragment 
class HomeFragment : Fragment(R.layout.fragment_home) {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        view.findViewById<Button>(R.id.detailButton).setOnClickListener {
            // Navigation Component, defined in res/navigation/nav_graph.xml
            findNavController().navigate(R.id.action_home_to_detail)
        }
    }
}

Note the parallels: Fragment ≈ a Compose screen composable, nav_graph.xml ≈ your NavHost block, findNavController() ≈ your NavController. The concepts transfer; only the syntax and the toolkit changed.

3 · App architecture: UI layer vs data layer

Android's recommended architecture splits the app into layers so each has one job:

UI layerComposables (draw pixels, emit events) + ViewModel (holds & exposes screen state)
▲ state down ▼ events up
Data layerRepository (the single source of truth API) → data sources: Room / DataStore (local) & network (web API)

Rules of thumb we'll repeat all term:

  • Composables are dumb and stateless where possible — they render state and fire events. No business logic, no database calls.
  • ViewModel is the state holder for one screen. It prepares UI state and handles events.
  • Repository is the source of truth for data. The ViewModel asks the repository; it never talks to a database or the network directly. (We build the repository in the data-persistence + networking sessions; today the ViewModel can hold an in-memory list.)
🍽️
Analogy — a restaurant

The composable is the dining room (what the customer sees). The ViewModel is the waiter (takes orders/events, brings out plated state). The repository is the kitchen (the source of the food/data). The dining room never walks into the kitchen — it asks the waiter. This separation is exactly project feature (e): clean architecture, ViewModel + repository.

4 · ViewModel as state holder & lifecycle awareness

A ViewModel is a class scoped to a screen (a navigation destination or the Activity) that survives configuration changes like rotation. That's the killer feature: when the screen is recreated, the same ViewModel instance is handed back, with its state intact.

Compare the two ways to "hold state" and when each dies:

  • remember { mutableStateOf(...) } survives recomposition but is lost on rotation (configuration change recreates the composition).
  • rememberSaveable survives rotation and process death for small, simple values — but it serializes through a Bundle, so it's only for lightweight UI state (a toggle, a text field).
Kotlin · Compose 
// Fine for trivial, local UI: an expanded/collapsed toggle.
var expanded by rememberSaveable { mutableStateOf(false) }

// NOT where your screen's real data should live.
// Rotate the phone and a plain remember{} resets to its initial value.

Screen data (lists, selections, loading flags) belongs in a ViewModel exposed as an immutable StateFlow<UiState>. Before the code, pin down the one new tool it uses:

🧩
One-sentence definition

A StateFlow is an observable value-holder that always has one current value and pushes the latest value to anyone collecting it — so the ViewModel keeps the screen's state in it, and the UI reads that state and redraws whenever it changes.

The idiomatic shape is a backing-property pair: a private, writable MutableStateFlow the ViewModel mutates, and a public, read-only StateFlow the UI collects. Watch for that pair (_uiState and uiState) in the code below:

Kotlin · ViewModel 
// A single immutable class describing everything the screen needs to draw.
data class HomeUiState(
    val items: List<Item> = emptyList(),
    val selectedId: Int? = null,
    val isLoading: Boolean = false
)

class HomeViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(HomeUiState())
    val uiState: StateFlow<HomeUiState> = _uiState.asStateFlow()  // read-only outside

    init { load() }

    private fun load() {
        // In-memory for today; later this comes from a repository.
        _uiState.update { it.copy(items = sampleItems(), isLoading = false) }
    }

    fun onItemSelected(id: Int) {
        _uiState.update { it.copy(selectedId = id) }
    }
}

The StateFlow pattern — every line, explained

  • private val _uiState = MutableStateFlow(HomeUiState()) — the private, writable holder. MutableStateFlow is created with an initial value (HomeUiState()), so it is never empty — it always has a current state to hand out. The leading underscore is just a naming convention for "this is the internal, mutable backing field; don't touch it from outside."
  • val uiState: StateFlow<HomeUiState> = _uiState.asStateFlow() — the public, read-only view of the same holder. .asStateFlow() wraps _uiState in a plain StateFlow that can be read but not written. The UI gets this one; only the ViewModel can reach _uiState to change it. (This is the encapsulation that makes unidirectional data flow enforceable, not just a suggestion.)
  • _uiState.update { it.copy(items = sampleItems(), isLoading = false) } — the only way state changes. update { current -> ... } takes the current state (here named it) and atomically replaces it with a new value. Inside, it.copy(...) uses the data class's generated copy() to make a fresh HomeUiState that is identical except for the named fields — items and isLoading change; selectedId is carried over untouched. You never mutate the old state object; you swap in a new immutable one.
  • fun onItemSelected(id: Int) { _uiState.update { it.copy(selectedId = id) } } — a named event handler. The UI calls this; the ViewModel does the actual write. Same update { it.copy(...) } move, this time changing only selectedId.

Strip away Compose and the pattern is tiny — a private writable flow, a public read-only one over the top, and update { it.copy(...) } to advance it:

Kotlin · the backing-property pair, isolated
private val _count = MutableStateFlow(0)        // writable — ViewModel only
val count: StateFlow<Int> = _count.asStateFlow() // read-only — what the UI sees

fun increment() {
    _count.update { current -> current + 1 }     // atomically replace with a new value
}
// _count.value is always readable: it starts at 0 and is never "empty".
// Outside code can read count.value, but it CANNOT write to count — only increment() can.

And the composable collects that state in a lifecycle-aware way:

Kotlin · Compose 
@Composable
fun HomeScreen(
    onItemClick: (Int) -> Unit,
    viewModel: HomeViewModel = viewModel()   // scoped + survives rotation
) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()

    ItemList(
        items = uiState.items,
        onItemClick = { id ->
            viewModel.onItemSelected(id)
            onItemClick(id)
        }
    )
}
  • val uiState by viewModel.uiState.collectAsStateWithLifecycle() — the UI reads the public, read-only uiState (never _uiState). collectAsStateWithLifecycle() subscribes to the flow and turns its current value into Compose state, so the screen recomposes every time the ViewModel pushes a new state — and it pauses collecting while the screen is backgrounded, which is the "lifecycle-aware" part.
  • onItemClick = { id -> viewModel.onItemSelected(id); ... } — the UI does not change state itself; it sends an event up by calling the ViewModel's named function. State flows down (via uiState), events flow up (via onItemSelected). One direction each.
⚠️
Common confusion

The public type is StateFlow, not MutableStateFlow, on purpose: the UI never assigns to uiState. There is no viewModel.uiState.value = ... from a composable — that line won't even compile, because .asStateFlow() hands out a read-only view. Only the ViewModel mutates the private _uiState. If you exposed the MutableStateFlow directly, two places could write state, the screen could force the ViewModel into an "impossible" state, and unidirectional data flow would quietly fall apart.

⚠️
Common misconception

"I'll just keep the data in a var inside the composable." It works until you rotate the phone — then it's gone, because the composition is rebuilt. The ViewModel is the fix: it lives outside the composition's lifecycle. We prove this live by rotating the emulator.

5 · State holders + unidirectional data flow (UDF)

Unidirectional data flow is a one-way loop: state flows down from the ViewModel into composables; events flow up from composables back to the ViewModel; the ViewModel updates state; the new state flows down again.

ViewModel state — state down → Composable UI — event up → ViewModel updates ↺ loop

Practically, this means:

  • Hoist state out of low-level composables. Pass values down and callbacks up (onItemClick: (Int) -> Unit), so leaf composables are reusable and previewable.
  • The UI state class (e.g. HomeUiState) is a single immutable snapshot of everything the screen needs — easy to reason about and test.
  • A composable never mutates ViewModel internals directly; it calls a named function (onItemSelected). One direction in, one direction out.
💡
Stateless leaf, stateful parent

A StatelessCounter(count, onIncrement) can be dropped anywhere and tested with fake values. Its parent (or the ViewModel) owns the real count. This "state hoisting" pattern is the heart of UDF — and it's the same idea whether the state lives in a parent composable or a ViewModel.

Hands-On Lab — Multi-Destination Nav App

🎯 Goal

Build an app with a list screen → detail screen (passing an id argument) → back, where a ViewModel owns the list and the current selection. Once that works, push further by adding a 3-tab bottom navigation.

⏱ 70 minutes 👥 Individual (pair if an emulator is down)

  1. Model your data and put it in a ViewModel

    Create a small data class and a ViewModel that exposes the list and a selected id via StateFlow. Use any theme you like (recipes, songs, planets…).

    Kotlin · ViewModel 
    data class Planet(val id: Int, val name: String, val summary: String)

data class CatalogUiState(
    val planets: List<Planet> = emptyList(),
    val selectedId: Int? = null
)

class CatalogViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(
        CatalogUiState(planets = seed())
    )
    val uiState: StateFlow<CatalogUiState> = _uiState.asStateFlow()

    fun select(id: Int) = _uiState.update { it.copy(selectedId = id) }
    fun planet(id: Int): Planet? = _uiState.value.planets.find { it.id == id }

    private fun seed() = listOf(
        Planet(1, "Mercury", "Smallest planet, closest to the Sun."),
        Planet(2, "Venus", "Hottest planet; thick CO2 atmosphere."),
        Planet(3, "Earth", "The only known planet with life."),
        Planet(4, "Mars", "The red planet; target of rovers.")
    )
}

    Starter hint: import androidx.lifecycle.ViewModel, kotlinx.coroutines.flow.*.

  2. Build the list screen (stateless leaf + stateful host)

    Hoist state: the list row takes a planet and an onClick callback; the screen reads the ViewModel. Expected: a scrollable list of names.

    Kotlin · Compose 
    @Composable
fun CatalogScreen(onPlanetClick: (Int) -> Unit, vm: CatalogViewModel = viewModel()) {
    val state by vm.uiState.collectAsStateWithLifecycle()
    LazyColumn {
        items(state.planets, key = { it.id }) { planet ->
            ListItem(
                headlineContent = { Text(planet.name) },
                modifier = Modifier.clickable {
                    vm.select(planet.id)
                    onPlanetClick(planet.id)
                }
            )
            HorizontalDivider()
        }
    }
}

  3. Add the NavHost with a detail argument

    Two destinations: "catalog" and "detail/{planetId}". Navigate with the tapped id; popBackStack() for Back. Expected: tapping a row opens the detail route.

    Kotlin · Compose 
    // navController is created once, at the top of your single-Activity app
// (e.g. inside setContent { … }), exactly like AppNavHost() in section 1.
val navController = rememberNavController()

NavHost(navController, startDestination = "catalog") {
    composable("catalog") {
        CatalogScreen(onPlanetClick = { id -> navController.navigate("detail/$id") })
    }
    composable(
        "detail/{planetId}",
        arguments = listOf(navArgument("planetId") { type = NavType.IntType })
    ) { entry ->
        val id = entry.arguments?.getInt("planetId") ?: return@composable
        PlanetDetail(planetId = id, onBack = { navController.popBackStack() })
    }
}

  4. Detail reads its item from the ViewModel by id

    The detail composable receives only the id and looks the planet up — proving "pass the id, not the object." Expected: the detail shows the correct name + summary; Back returns to the list.

    Kotlin · Compose 
    @Composable
fun PlanetDetail(planetId: Int, onBack: () -> Unit, vm: CatalogViewModel = viewModel()) {
    val planet = vm.planet(planetId)
    Column(Modifier.padding(24.dp)) {
        Text(planet?.name ?: "Unknown", style = MaterialTheme.typography.headlineMedium)
        Spacer(Modifier.height(8.dp))
        Text(planet?.summary ?: "No data.")
        Spacer(Modifier.height(24.dp))
        Button(onClick = onBack) { Text("Back to list") }
    }
}

  5. Rotate to verify state survival

    Rotate the device on the list and on the detail screen. Expected: scroll position and selection are sensible; nothing crashes or resets to a blank screen. If you stored data in a plain remember{}, fix it by moving it into the ViewModel.

🧯
Troubleshooting — the usual suspects
  • "Navigation destination … cannot be found" → the route string you navigate(...) to doesn't match the composable(...) pattern. Compare them character-for-character (the {planetId} placeholder is replaced by the actual number when navigating).
  • Argument is always 0 / null → you forgot the arguments = listOf(navArgument(...) { type = NavType.IntType }) declaration, or you read the wrong key name.
  • Two different ViewModel instances → calling viewModel() inside both screens scopes a ViewModel per destination. For shared state across destinations, scope it to the nav graph / Activity (we cover graph-scoped ViewModels later; for this lab, looking up by id from a per-screen seed is fine).
  • Unresolved collectAsStateWithLifecycle → add androidx.lifecycle:lifecycle-runtime-compose and import it.
  • List doesn't scroll / recomposes oddly → make sure items(list, key = { it.id }) has a stable key.
🚀
Push further — 3-tab bottom navigation

Wrap the NavHost in a Scaffold with a NavigationBar of three tabs (e.g. Catalog, Favorites, About). Use navController.currentBackStackEntryAsState() to highlight the selected tab and launchSingleTop = true + popUpTo so tabs don't stack endlessly.

Kotlin · Compose 
// Each tab carries its OWN icon so the three destinations look distinct.
data class Tab(val route: String, val label: String, val icon: ImageVector)
val tabs = listOf(
    Tab("catalog",   "Catalog",   Icons.Default.List),
    Tab("favorites", "Favorites", Icons.Default.Favorite),
    Tab("about",     "About",     Icons.Default.Info)
)
val backStack by navController.currentBackStackEntryAsState()
val current = backStack?.destination?.route

NavigationBar {
    tabs.forEach { tab ->
        NavigationBarItem(
            selected = current == tab.route,
            onClick = {
                navController.navigate(tab.route) {
                    launchSingleTop = true
                    popUpTo(navController.graph.startDestinationId) { saveState = true }
                    restoreState = true
                }
            },
            icon = { Icon(tab.icon, contentDescription = null) },  // label already names it
            label = { Text(tab.label) }
        )
    }
}

The built-in Icons.Default.* set comes from androidx.compose.material:material-icons-core (pulled in by Material 3). Reach for the larger material-icons-extended artifact only if you need an icon that isn't in the core set.

1 · Worked example — the reduce(state, event) at the heart of a ViewModel

Strip away Compose, StateFlow, and the emulator and a ViewModel is doing exactly one thing: it is a pure function from (current state, an event) → new state. That function is the whole of unidirectional data flow. If you can write it as plain Kotlin, the Android wiring (_uiState.update { reduce(it, event) }) is mechanical. Master the core and the framework is just plumbing.

Reason through it the way a senior engineer does, one decision at a time:

Step 1 — Name the stateOne immutable data class snapshot of everything the screen draws. Here: a counter app with count, a step, and a derived parity label. State is the single source of truth.
▼
Step 2 — Enumerate the eventsA sealed interface of everything the user can do: Increment, Decrement, SetStep(n), Reset. Sealed = the compiler forces you to handle every case in when — no silent gaps.
▼
Step 3 — Write the reducerA pure fun reduce(state, event): State that returns a new state via copy(...). No mutation, no I/O, no clock, no randomness — same inputs always give the same output, so it is trivially testable.
▼
Step 4 — Derive, don't storeparity is computed from count inside reduce every time. Never keep a second field you have to remember to update — that is how UI states drift out of sync.

Here is the complete, runnable reference. Predict the printed output first (walk the events through your head), then press Run to confirm. Notice that the harness feeds events one-by-one through reduce exactly as a ViewModel would feed them through _uiState.update { }.

Kotlin · runnable — the reducer, worked
// ----- Step 1: the immutable UI state snapshot -----
data class CounterState(
    val count: Int = 0,
    val step: Int = 1,
    val parity: String = "even"   // DERIVED from count — see reduce()
)

// ----- Step 2: every event the screen can emit -----
sealed interface CounterEvent {
    data object Increment : CounterEvent
    data object Decrement : CounterEvent
    data class  SetStep(val n: Int) : CounterEvent
    data object Reset : CounterEvent
}

// ----- Step 3 + 4: pure reducer; parity is derived, never stored stale -----
fun reduce(state: CounterState, event: CounterEvent): CounterState {
    val next = when (event) {
        CounterEvent.Increment   -> state.copy(count = state.count + state.step)
        CounterEvent.Decrement   -> state.copy(count = state.count - state.step)
        is CounterEvent.SetStep  -> state.copy(step = event.n)
        CounterEvent.Reset       -> CounterState(step = state.step) // keep the step, zero the count
    }
    // derive parity from the NEW count, so it can never drift out of sync
    return next.copy(parity = if (next.count % 2 == 0) "even" else "odd")
}

fun main() {
    // This loop is what a ViewModel does: fold events over state.
    val events = listOf(
        CounterEvent.Increment,        // 0 -> 1
        CounterEvent.SetStep(5),       // step becomes 5
        CounterEvent.Increment,        // 1 -> 6
        CounterEvent.Decrement,        // 6 -> 1
        CounterEvent.Reset             // count -> 0, step stays 5
    )
    val finalState = events.fold(CounterState()) { state, event -> reduce(state, event) }
    println(finalState)   // CounterState(count=0, step=5, parity=even)
}
🔑
The insight to carry away

events.fold(initial) { s, e -> reduce(s, e) } is unidirectional data flow in one line. A real ViewModel just spreads that fold across time: each user action calls _uiState.update { reduce(it, event) }, and Compose re-reads the new immutable state. Pure reducer in the middle, framework on the outside.

Kotlin · one correct reducer 
fun reduce(state: CartState, event: CartEvent): CartState {
    val lines: Map<String, Int> = when (event) {
        is CartEvent.Add -> {
            val q = state.lines[event.product] ?: 0
            state.lines + (event.product to q + 1)
        }
        is CartEvent.Remove -> {
            val q = (state.lines[event.product] ?: 0) - 1
            if (q <= 0) state.lines - event.product
            else state.lines + (event.product to q)
        }
        CartEvent.Clear -> emptyMap()
    }
    val itemCount = lines.values.sum()
    val subtotal = lines.entries.sumOf { (p, q) -> q * (PRICES[p] ?: 0) }
    return CartState(lines = lines, itemCount = itemCount, subtotalCents = subtotal)
}

2 · Common misconceptions & pitfalls

Each one is a real wrong mental model that bites people shipping navigation + ViewModel code. Read the symptom — it's how you'll recognize it in your own app.

Wrong mental model: a route argument is like a function parameter, so I can hand navigate my Planet data class directly.

Why it's wrong: a route is a string ("detail/3"). Arguments are serialized into that string, so they must be primitives (Int, String, Boolean…). There is no slot for a live object reference. Even where libraries appear to allow it, the object can't survive a deep link or process death — the OS only restored a string.

Correct model: pass the id in the route; the destination looks the object up from the ViewModel/repository (the single source of truth).

Symptom you'd see: compile errors trying to put a class in navArgument; or, after relaunch-from-recents, the detail screen shows blank/garbage because the "object" was never really there.

Wrong mental model: the composable is my screen, so state I keep inside it is the screen's state.

Why it's wrong: a plain var resets on every recomposition; remember{} survives recomposition but is destroyed and rebuilt on a configuration change (rotation, dark-mode switch, font-size change, language change). The composition is not a stable home for real data.

Correct model: screen data lives in a ViewModel (survives config changes), exposed as StateFlow<UiState> and read with collectAsStateWithLifecycle(). Use rememberSaveable only for tiny throwaway UI bits (an expanded toggle, an in-progress text field).

Symptom you'd see: rotate the phone and your list, scroll, or selection vanishes / resets to the initial value. The classic "it worked until I turned the phone sideways" bug.

Wrong mental model: it's fewer lines if the composable just sets viewModel.uiState.value = ... when something happens.

Why it's wrong: that breaks unidirectional data flow. Now two places mutate state, ordering becomes unpredictable, and the screen can put the ViewModel into a state the ViewModel's own logic would never allow. State changes scatter across the codebase and become impossible to reason about or reproduce.

Correct model: keep the mutable flow private and expose a read-only one (val uiState = _uiState.asStateFlow()). The composable sends events up (onItemClick(id) → viewModel.onItemSelected(id)); only the ViewModel writes state. State down, events up — one direction each.

Symptom you'd see: "impossible" UI states (loading and error at once), heisenbugs you can't reproduce, and merge conflicts because state writes are sprinkled everywhere.

Wrong mental model: val vm: NotesViewModel = viewModel() always works, so I'll add a constructor parameter and keep calling viewModel().

Why it's wrong: the default viewModel() can only construct a ViewModel with a no-argument constructor. The moment your ViewModel needs NotesViewModel(repo), the default factory has no idea how to supply repo and crashes at runtime.

Correct model: provide a factory. The in-scope, current pattern is a viewModelFactory { initializer { ... } } passed to viewModel(factory = ...):

Kotlin · supplying a constructor dependency 
// In the ViewModel's companion, or a top-level val:
val Factory = viewModelFactory {
    initializer { NotesViewModel(NotesRepository) }   // hand it what it needs
}

// At the call site (e.g. inside a composable destination):
val vm: NotesViewModel = viewModel(factory = NotesViewModel.Factory)

Symptom you'd see: RuntimeException: Cannot create an instance of class NotesViewModel / "no zero-argument constructor" the instant the screen opens.

Wrong mental model: viewModel() is a singleton, so the list screen and the detail screen see the same instance and the same selection.

Why it's wrong: viewModel() is scoped to the nearest ViewModelStoreOwner — and with Navigation-Compose, each destination is its own owner. Calling viewModel() in both the list and the detail composable gives you two different instances, each with its own (re-seeded) state.

Correct model: for genuinely shared state, scope the ViewModel to the nav graph / parent entry, or (simpler and in-scope here) pass the id as a route argument and have the detail look the item up. Don't rely on a "shared" instance that isn't shared.

Symptom you'd see: you set a selection on the list, navigate to detail, and the detail's ViewModel reports selectedId == null — because it's a brand-new, freshly-seeded instance.

3 · In production — how a senior engineer wires this

🏭
The idiomatic, current pattern (2024+)

On a real team, a screen's ViewModel almost always looks like this skeleton, and code review will push you toward it:

  • One immutable UiState data class per screen, exposed as a read-only StateFlow built from a private val _uiState = MutableStateFlow(...) via asStateFlow(). Reads in the UI use collectAsStateWithLifecycle() so collection pauses when the screen is backgrounded — saving battery and avoiding work on an invisible screen.
  • State is updated with _uiState.update { it.copy(...) }, never by reassigning .value in two places. The update lambda is atomic, which matters once coroutines update the same flow concurrently.
  • The ViewModel talks only to a repository, never to Retrofit/Room directly. That seam is what lets you swap a fake repository in for the screen during development and keeps networking/database concerns out of the UI layer (you'll build that repository in the data + networking sessions).
  • Dependencies arrive through a factory — viewModel(factory = ...) with viewModelFactory { initializer { ... } } — so the ViewModel is constructed with its repository instead of reaching out to grab one. (Bigger apps reach for a DI framework here, but that's beyond this course; the factory is the in-scope, fully sufficient tool.)
  • Navigation passes ids, not objects, and "go home / switch tab" actions use launchSingleTop = true + popUpTo(...) so the back stack stays sane instead of growing a duplicate destination on every tap.

One tasteful gotcha: when a ViewModel collects an upstream flow (e.g. from the database) and re-exposes it with stateIn, use SharingStarted.WhileSubscribed(5_000), not Eagerly or Lazily. The 5-second grace period keeps the upstream alive across a rotation (so you don't re-query the DB on every screen flip) but still tears it down shortly after the user truly leaves — the difference between a snappy app and one that quietly drains the battery or thrashes the database. You'll use this exact idiom in the Room session.
