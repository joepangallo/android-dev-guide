Lecture / Concept Notes

Learn-from-here notes — everything you need to understand Concept Block A is in this section.

Analogy first. HTTP is like ordering at a restaurant. Your request is the order ("a GET for table 5's menu"); the server prepares it and sends back a response (the food + a receipt). The status code is the waiter's reply: 200 = "here you go," 404 = "we don't have that dish," 500 = "the kitchen caught fire."

An HTTP request has a method (verb), a URL, headers, and an optional body. The response has a status code, headers, and a body (often JSON). REST (Representational State Transfer) is a convention for designing those URLs around resources:

VerbMeansExample (REST)
GETRead data (no side effects)GET /quotes?limit=20
POSTCreate a new resourcePOST /quotes (+ JSON body)
PUT/PATCHUpdate an existing resourcePUT /quotes/42
DELETERemove a resourceDELETE /quotes/42

Status code families (memorize the first digit)

RangeFamilyWhat your app does
2xxSuccess (200 OK, 201 Created, 204 No Content)Parse the body, show data.
3xxRedirectUsually handled by the HTTP client.
4xxClient error (400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests)Your fault: bad URL, missing auth, wrong params. Show a useful error.
5xxServer error (500, 503 Service Unavailable)Their fault: offer retry.
💡
CLO 6 framing

HTTP rides on top of TCP; HTTPS adds a TLS encryption layer (this is why endpoints are https:// — Android blocks plain http:// cleartext by default since API 28). We'll mention WebSockets later for real-time, but REST-over-HTTPS is the workhorse for "backend communication" in this course.

Open the Retrofit-flow visualizer to show the request → response cycle interactively before moving on.

Two non-negotiables before any network code runs.

(a) Declare the INTERNET permission

Without this line your request throws SecurityException / silently fails. It's a normal permission — no runtime prompt, just the manifest entry.

XML 
<!-- AndroidManifest.xml — child of <manifest>, BEFORE <application> -->
<uses-permission android:name="android.permission.INTERNET" />

(b) Never block the main thread

Android draws the UI on a single main (UI) thread. A network call can take seconds. If you do it on the main thread, the screen freezes; after ~5 seconds Android shows the dreaded ANR ("Application Not Responding"). Modern Android forbids it outright: a synchronous network call on the main thread throws NetworkOnMainThreadException.

The Kotlin answer is coroutines. A suspend function can pause and resume without blocking a thread. We launch it from viewModelScope and let Retrofit move the actual I/O onto a background dispatcher.

Kotlin · anti-pattern 
// DON'T: runs on the calling (main) thread → freeze + crash
fun loadQuotes() {
    val response = api.getQuotes()   // synchronous I/O on main thread
    _quotes.value = response.results // ANR / NetworkOnMainThreadException
}
Kotlin · correct 
// DO: suspend fn launched on viewModelScope; Retrofit handles the dispatcher
fun loadQuotes() {
    viewModelScope.launch {            // coroutine tied to ViewModel lifecycle
        val response = api.getQuotes() // suspends here, does NOT block the thread
        _uiState.value = QuoteUiState.Success(response.results)
    }
}
ⓘ
Why viewModelScope?

A coroutine launched in viewModelScope is automatically cancelled when the ViewModel is cleared (screen destroyed). That prevents leaking work or updating state for a screen that no longer exists. You get this scope free from androidx.lifecycle:lifecycle-viewmodel-compose.

JSON (JavaScript Object Notation) is the text format almost every web API speaks. Objects are { "key": value }; arrays are [ ... ]. Our job is to turn that text into typed Kotlin objects so the compiler can help us.

Sample response from our demo endpoint:

JSON 
{
  "count": 2,
  "results": [
    { "_id": "abc123", "content": "Stay hungry, stay foolish.", "author": "Steve Jobs", "tags": ["wisdom"] },
    { "_id": "def456", "content": "Simplicity is the soul of efficiency.", "author": "Austin Freeman", "tags": ["technology"] }
  ]
}

Each JSON object becomes a Kotlin data class. Field names should match the JSON keys (or be mapped with an annotation). A converter library does the heavy lifting at runtime.

Kotlin · kotlinx.serialization (recommended) 
import kotlinx.serialization.SerialName
import kotlinx.serialization.Serializable

@Serializable
data class QuoteResponse(
    val count: Int,
    val results: List<Quote>
)

@Serializable
data class Quote(
    @SerialName("_id") val id: String,     // JSON "_id" → Kotlin id
    val content: String,
    val author: String,
    val tags: List<String> = emptyList()  // default makes it null-safe
)

Pairs with the kotlinx-serialization-converter. Compile-time safe, no reflection, Kotlin-first. This is the modern default.

Kotlin · Moshi 
import com.squareup.moshi.Json
import com.squareup.moshi.JsonClass

@JsonClass(generateAdapter = true)
data class Quote(
    @Json(name = "_id") val id: String,
    val content: String,
    val author: String,
    val tags: List<String> = emptyList()
)

Pairs with converter-moshi + the Moshi Kotlin codegen. Fast, widely used.

Kotlin · Gson 
import com.google.gson.annotations.SerializedName

data class Quote(
    @SerializedName("_id") val id: String,
    val content: String,
    val author: String,
    val tags: List<String> = emptyList()
)

Pairs with converter-gson. Oldest/most familiar to Java devs; uses reflection and is less Kotlin-aware (watch out for non-null fields the parser can leave null).

⚠️
Common JSON-mapping bugs

(1) A key in the JSON that's missing from your data class is simply ignored — you don't need to model every field. (2) But a non-null Kotlin field that's absent/null in the JSON will crash at parse time. Give optional fields a default or a nullable type. (3) Names are case-sensitive: JSON author ≠ Kotlin Author.

Retrofit (by Square, built on OkHttp) turns an HTTP API into a Kotlin interface. You annotate methods with the verb + path; Retrofit generates the implementation, performs the call, and uses a converter to deserialize the JSON into your data classes.

Step 1 — the API interface

Kotlin · Retrofit interface 
import retrofit2.http.GET
import retrofit2.http.Query

interface QuoteApiService {

    // GET https://api.quotable.io/quotes?limit=20
    @GET("quotes")
    suspend fun getQuotes(
        @Query("limit") limit: Int = 20
    ): QuoteResponse        // Retrofit + converter return the parsed object
}

Note suspend — Retrofit natively supports coroutines, so the call neatly fits inside viewModelScope.launch { } with no callbacks.

Step 2 — build the Retrofit instance (singleton)

🧩
One-sentence definition — by lazy

val api by lazy { … } means "don't build this object yet — run the { … } block the first time someone reads api, then cache that result and hand back the same instance forever after," which is exactly what you want for an expensive object you might not need right away.

Kotlin · Retrofit builder (kotlinx.serialization) 
import retrofit2.converter.kotlinx.serialization.asConverterFactory
import kotlinx.serialization.json.Json
import okhttp3.MediaType.Companion.toMediaType
import retrofit2.Retrofit

private const val BASE_URL = "https://api.quotable.io/"

private val json = Json { ignoreUnknownKeys = true }  // tolerate extra fields

object QuoteApi {
    val retrofitService: QuoteApiService by lazy {
        Retrofit.Builder()
            .baseUrl(BASE_URL)
            .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
            .build()
            .create(QuoteApiService::class.java)
    }
}
💡
Two rules students always trip on

(1) BASE_URL must end with a trailing slash /, and the @GET path must not start with one, or Retrofit throws IllegalArgumentException. (2) Use by lazy so the service is built once, not on every call.

Read that builder block chunk by chunk — it is one long method chain where each call returns something the next call hangs off of:

  • val retrofitService: QuoteApiService by lazy { … } — declares the service, but the { … } body does not run now. The first time any code reads QuoteApi.retrofitService, the block runs once, builds the service, caches it, and every later read returns that same cached instance. It is thread-safe by default, so two coroutines asking at once still get one shared instance. (This is the same by delegate idea you met in Session 5 — not a new keyword, just a different delegate behind it.)
  • Retrofit.Builder() — starts a fresh, empty builder. You configure it step by step, then ask it to assemble the finished object (the builder pattern).
  • .baseUrl(BASE_URL) — sets the root URL every request is resolved against. It must end in a slash ("https://api.quotable.io/"); the @GET("quotes") path gets appended to it.
  • .addConverterFactory(json.asConverterFactory(…)) — plugs in the piece that does JSON ↔ Kotlin-object conversion. Here it's kotlinx.serialization (Moshi and Gson ship their own factories). Without a converter, Retrofit wouldn't know how to turn the response body into a QuoteResponse.
  • .build() — takes everything you configured and produces the actual Retrofit instance. The builder's job is finished here.
  • .create(QuoteApiService::class.java) — the magic step: you hand Retrofit your interface and it generates a concrete implementation of it at runtime, returning an object that satisfies QuoteApiService. Calling getQuotes() on that returned object is what actually fires the HTTP request.

To see by lazy on its own — stripped of all the Retrofit noise — here is the whole idea in one tiny example:

Kotlin · by lazy in isolation 
val greeting: String by lazy {
    println("...building it now...")   // runs ONCE, on first read
    "Hello"
}

fun main() {
    println("before")        // the block has NOT run yet
    println(greeting)        // first read → prints "...building it now..." then "Hello"
    println(greeting)        // second read → just "Hello" (cached; block skipped)
}
// Output:
// before
// ...building it now...
// Hello
// Hello
⚠️
Common confusion

You never write the body of QuoteApiService and you never new it up — there is no QuoteApiServiceImpl class for you to author. The interface is just a description of the endpoints; .create(QuoteApiService::class.java) is what builds the working implementation behind the scenes and returns it. And by lazy is not a brand-new keyword — it's the same property-delegate (by) mechanism from Session 5, just using Kotlin's built-in lazy { } delegate to defer and cache the value.

Compare: Retrofit vs Volley

  • Type-safe interface; JSON → data class automatically via a converter.
  • First-class suspend / coroutine support — clean, sequential code.
  • Industry standard for REST clients today; pairs with OkHttp interceptors.
Kotlin · Volley (for awareness only) 
val queue = Volley.newRequestQueue(context)
val request = JsonObjectRequest(Request.Method.GET, url, null,
    { json -> /* manually pull fields out of json */ },
    { error -> /* handle */ })
queue.add(request)

Volley is callback-based and you parse JSON by hand. Fine for tiny apps, but no type-safety and awkward with coroutines. We mention it so students recognize it; we build with Retrofit.

A network screen is always in one of three states. Model them explicitly with a sealed interface so the UI is exhaustive and you can never forget the spinner or the error message.

Loading
Show a spinner. The request is in flight.
Success(data)
Render the list in a LazyColumn.
Error(msg)
Show a message + Retry button.
Kotlin · sealed UI state 
sealed interface QuoteUiState {
    data object Loading : QuoteUiState
    data class Success(val quotes: List<Quote>) : QuoteUiState
    data class Error(val message: String) : QuoteUiState
}
Kotlin · Compose renders the state 
@Composable
fun QuoteScreen(
    uiState: QuoteUiState,
    onRetry: () -> Unit,
    modifier: Modifier = Modifier
) {
    when (uiState) {
        is QuoteUiState.Loading -> LoadingView(modifier)
        is QuoteUiState.Error   -> ErrorView(uiState.message, onRetry, modifier)
        is QuoteUiState.Success -> QuoteList(uiState.quotes, modifier)
    }
}

@Composable
fun LoadingView(modifier: Modifier = Modifier) {
    Box(modifier.fillMaxSize(), contentAlignment = Alignment.Center) {
        CircularProgressIndicator()
    }
}

@Composable
fun ErrorView(message: String, onRetry: () -> Unit, modifier: Modifier = Modifier) {
    Column(
        modifier.fillMaxSize().padding(24.dp),
        horizontalAlignment = Alignment.CenterHorizontally,
        verticalArrangement = Arrangement.Center
    ) {
        Text("⚠️ $message", style = MaterialTheme.typography.bodyLarge)
        Spacer(Modifier.height(12.dp))
        Button(onClick = onRetry) { Text("Retry") }
    }
}

@Composable
fun QuoteList(quotes: List<Quote>, modifier: Modifier = Modifier) {
    LazyColumn(
        modifier = modifier.fillMaxSize(),
        contentPadding = PaddingValues(16.dp),
        verticalArrangement = Arrangement.spacedBy(12.dp)
    ) {
        items(quotes, key = { it.id }) { quote ->
            QuoteCard(quote)
        }
    }
}
💡
Why a key in items{}

Passing key = { it.id } lets Compose track items across data changes — better recomposition performance and correct animations when the list updates (e.g., after a refresh).

Retrofit is one kind of API integration. CLO5 is broader: Android apps integrate many platform and cloud APIs. Two you'll meet again:

  • Location services — the Fused Location Provider (com.google.android.gms:play-services-location) gives device location. It requires a runtime permission (ACCESS_FINE_LOCATION), unlike INTERNET. A common pattern: get the user's lat/long, then call a weather REST API with it — exactly the Retrofit skill from today, plus a permission prompt.
  • Cloud-based storage — services like Firebase / cloud object stores let an app read/write data remotely. Conceptually it's the same request/response model; the SDK wraps the HTTP for you.

    Kotlin · network/QuoteApi.kt

import okhttp3.logging.HttpLoggingInterceptor

  6. Create the sealed UI state + ViewModel

    Kotlin · ui/QuoteViewModel.kt 
    import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow
import kotlinx.coroutines.launch
import java.io.IOException

sealed interface QuoteUiState {
    data object Loading : QuoteUiState
    data class Success(val quotes: List<Quote>) : QuoteUiState
    data class Error(val message: String) : QuoteUiState
}

class QuoteViewModel(
    private val repository: QuoteRepository = NetworkQuoteRepository()
) : ViewModel() {

    private val _uiState = MutableStateFlow<QuoteUiState>(QuoteUiState.Loading)
    val uiState: StateFlow<QuoteUiState> = _uiState.asStateFlow()

    init { loadQuotes() }

    fun loadQuotes() {
        _uiState.value = QuoteUiState.Loading
        viewModelScope.launch {
            _uiState.value = try {
                QuoteUiState.Success(repository.getQuotes())
            } catch (e: IOException) {
                QuoteUiState.Error("No internet connection. Check your network and retry.")
            } catch (e: Exception) {
                QuoteUiState.Error("Couldn't load quotes: ${e.message}")
            }
        }
    }
}

Read the network stack, every line explained

After the app runs once, slow down and read the code as a data path. The goal is not to memorize Retrofit syntax; the goal is to know what every line is responsible for, where errors can happen, and which layer is allowed to know about HTTP.

1. Gradle + manifest: make networking possible

  • kotlin("plugin.serialization") version "2.2.20" adds compile-time support for Kotlin serialization annotations. Keep this version aligned with the Kotlin version, because the compiler plugin is tied to the compiler.
  • implementation("com.squareup.retrofit2:retrofit:3.0.0") adds Retrofit, the library that turns annotated Kotlin interfaces into HTTP calls.
  • implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.11.0") adds the JSON parser that knows how to create Kotlin data classes from a JSON string.
  • implementation("com.squareup.retrofit2:converter-kotlinx-serialization:3.0.0") connects those two worlds: Retrofit downloads bytes, the converter asks kotlinx.serialization to decode them.
  • logging-interceptor is a teaching and debugging tool. It prints the request and response in Logcat so you can see whether the failure is "no request happened," "the server answered with an error," or "JSON parsing failed."
  • lifecycle-viewmodel-compose gives Compose-friendly lifecycle tools such as viewModel() and lifecycle-aware collection. Networking results should flow through the ViewModel, not through a composable-local variable.
  • <uses-permission android:name="android.permission.INTERNET" /> is the OS permission gate. Without it, the app is not allowed to open the network socket, even if every Kotlin file is correct.

2. Model file: teach Kotlin the shape of the JSON

  • import kotlinx.serialization.Serializable brings in the annotation that tells the compiler to generate JSON encode/decode code for this class.
  • @Serializable goes on each class that should be created from JSON. If you forget it, Retrofit can download the response but the converter cannot build your object.
  • data class QuoteResponse(...) models the top-level object returned by the quotes endpoint. The API does not return a bare list; it returns an object with fields such as count and results.
  • val count: Int says the JSON field named count must decode into an integer. If the server sends text there, decoding fails, which is good: the model no longer matches reality.
  • val results: List<Quote> says the interesting part of the response is a list of quote objects. The repository later unwraps this so the UI does not care about the response envelope.
  • @SerialName("_id") val id: String maps the JSON key _id to the nicer Kotlin property name id. Use this whenever a JSON key is ugly, reserved, or not idiomatic Kotlin.
  • val tags: List<String> = emptyList() gives a safe default. If the API omits tags, the app still gets an empty list instead of a crash.

3. Retrofit service: describe the request, do not perform it yet

  • interface QuoteApiService is only a contract. There is no hand-written class body; Retrofit generates the runtime implementation for you.
  • @GET("quotes") says this function performs an HTTP GET request to the relative path quotes. Combined with BASE_URL, the final URL starts as https://api.quotable.io/quotes.
  • suspend fun getQuotes(...) means this call can pause without blocking the main thread. You must call it from a coroutine, which is why the ViewModel uses viewModelScope.launch.
  • @Query("limit") limit: Int = 20 appends ?limit=20 by default. The parameter stays typed as an Int in Kotlin; Retrofit handles turning it into URL text.
  • : QuoteResponse is the decoded return type. By the time this function returns, the HTTP body has already been parsed into your Kotlin model or an exception has been thrown.

4. Retrofit singleton: build the client once

  • private const val BASE_URL = "https://api.quotable.io/" must end with a slash. Retrofit joins this base with the relative path in @GET("quotes").
  • Json { ignoreUnknownKeys = true } tells the decoder to ignore JSON fields your model does not list. That makes the app resilient when the server adds extra fields later.
  • OkHttpClient.Builder() creates the underlying HTTP client Retrofit will use for real network traffic.
  • .addInterceptor(HttpLoggingInterceptor().apply { ... }) attaches request/response logging. apply configures the interceptor object and then returns that same object to addInterceptor.
  • object QuoteApi makes one process-wide holder for the Retrofit service. This prevents rebuilding the client every time a screen appears.
  • val service: QuoteApiService by lazy { ... } creates the service the first time someone asks for it, then reuses it. by lazy is Kotlin's "build once, cache forever" pattern.
  • Retrofit.Builder().baseUrl(...).client(...).addConverterFactory(...).build() assembles the three required pieces: where requests go, how HTTP is performed, and how JSON becomes Kotlin.
  • .create(QuoteApiService::class.java) asks Retrofit to generate an implementation of your interface. This is the moment the annotation-only interface becomes callable code.

5. Repository: hide Retrofit from the rest of the app

  • interface QuoteRepository describes what the app needs: "give me quotes." It does not mention HTTP, Retrofit, URLs, or JSON.
  • suspend fun getQuotes(limit: Int = 20): List<Quote> keeps the repository asynchronous because the real implementation may hit the network, disk, or both.
  • class NetworkQuoteRepository(...) is the production implementation. The name admits where it gets data, but callers only depend on the interface.
  • private val service: QuoteApiService = QuoteApi.service supplies the real Retrofit service by default while still allowing a fake service in a test or classroom demo.
  • service.getQuotes(limit).results is the translation boundary: Retrofit returns the API envelope, and the repository returns only the list the screen actually needs.

6. ViewModel: turn work into screen state

  • private val repository: QuoteRepository = NetworkQuoteRepository() gives the ViewModel a data source. In a larger app, dependency injection would provide it; here the default keeps the sample readable.
  • private val _uiState = MutableStateFlow<QuoteUiState>(QuoteUiState.Loading) creates the private writable state holder. The screen starts in Loading because no network response has arrived yet.
  • val uiState: StateFlow<QuoteUiState> = _uiState.asStateFlow() exposes a read-only view. Compose can observe the state, but only the ViewModel can write it.
  • init { loadQuotes() } kicks off the first load when the ViewModel is created, so the screen has data without the user pressing a button.
  • _uiState.value = QuoteUiState.Loading resets the UI before each attempt. Retry should visibly return to the loading state, not leave stale content pretending nothing is happening.
  • viewModelScope.launch { ... } starts a coroutine tied to the ViewModel lifecycle. When the ViewModel is cleared, this work is cancelled automatically.
  • try { QuoteUiState.Success(repository.getQuotes()) } is the happy path: await the suspend repository call, then wrap the data in a success state.
  • catch (e: IOException) handles connectivity and transport problems separately, which lets you show a useful "check your network" message.
  • catch (e: Exception) catches the remaining unexpected failures, such as decoding issues. Do not stop at this generic branch only; students need to learn that offline is a normal, expected mobile state.

7. Compose route and list: state becomes pixels

  • fun QuoteRoute(viewModel: QuoteViewModel = viewModel()) is a screen-level entry point. It obtains the ViewModel and connects it to stateless UI.
  • val uiState by viewModel.uiState.collectAsStateWithLifecycle() collects the public StateFlow and turns it into Compose state. When the flow changes, this composable recomposes; when the screen is stopped, collection pauses.
  • QuoteScreen(uiState = uiState, onRetry = viewModel::loadQuotes) passes a value down and an event callback up. viewModel::loadQuotes is a function reference: "call this ViewModel function when retry is clicked."
  • LazyColumn(...) renders only the visible rows plus a small buffer, which is why it is the right list container for API results.
  • items(quotes, key = { it.id }) gives each row a stable identity. Stable keys help Compose keep row state attached to the correct item when the list changes.
  • QuoteCard(quote) keeps each row small and readable. The list decides which items appear; the card decides how one item appears.

One request, traced from tap to pixel

  1. The user opens the screen or taps Retry.
  2. QuoteRoute calls or references viewModel.loadQuotes().
  3. The ViewModel sets Loading, so Compose shows progress.
  4. viewModelScope launches the suspend work off the direct UI path.
  5. The repository asks the Retrofit service for quotes.
  6. Retrofit + OkHttp perform GET /quotes?limit=20.
  7. The converter decodes JSON into QuoteResponse.
  8. The repository unwraps results into List<Quote>.
  9. The ViewModel stores Success(list) or Error(message).
  10. Compose observes the new state and re-renders the spinner, list, or error view.

If a student can narrate this path without looking, they understand the whole networking lesson.

🧠 Master it — turn the network layer into a reflex

The hardest part of networking isn't the happy path — it's the edge: the request that fails, the body that's empty, the exception that escapes. Below you'll build the one piece of logic that decides everything downstream — the function that turns a Result into a UI state — first by watching it worked out, then by writing it yourself with the playground checking every case.

Worked example: the Result<T> → UiState reducer

Notice the seam in the live-coding ViewModel: the repository did the network call, and the ViewModel turned the outcome into Loading / Success / Error. A senior engineer pulls that decision into one pure function — no Android, no Retrofit, no coroutine — so it is trivial to reason about and to test. Let's reason through it step by step.

  1. Model the four states explicitly (add Empty)

    A real network screen has four states, not three. A request can succeed and still hand back zero rows — "200 OK, here are your zero photos." Rendering an empty LazyColumn looks identical to a bug ("did it even load?"). Give emptiness its own state so the UI can say "Nothing here yet."

    Kotlin · the four-state model 
    sealed interface UiState {
    data object Loading : UiState
    data object Empty : UiState                      // 200 OK, but zero rows
    data class Success(val photos: List<Photo>) : UiState
    data class Error(val message: String) : UiState
}

  2. Decide what each branch of Result becomes

    The repository returns Result<List<Photo>> (the idiomatic Kotlin wrapper). Reducing it is a four-line decision: success-with-rows → Success; success-with-no-rows → Empty; failure → Error with a message that matches the cause. That last point is the whole game: an IOException (you're offline) deserves a different message and a retry, while an HttpException (the server answered, badly) usually should not be blindly retried.

    Kotlin · the reducer (read it, predict its output) 
    fun reduce(result: Result<List<Photo>>): UiState =
    result.fold(
        onSuccess = { photos ->
            if (photos.isEmpty()) UiState.Empty
            else UiState.Success(photos)
        },
        onFailure = { e ->
            UiState.Error(
                when (e) {
                    is IOException   -> "You appear to be offline. Tap Retry."
                    is HttpException -> "Server error (HTTP ${'$'}{e.code()}). Try again later."
                    else             -> "Something went wrong: ${'$'}{e.message}"
                }
            )
        }
    )

    This mirrors the canonical taxonomy used in the Retrofit-flow lab: IOException = never reached the server (retryable); HttpException = reached it, got a non-2xx (don't blindly retry); generic last.

Common misconceptions & pitfalls

Each of these has bitten real Android apps in production. Recognize the wrong mental model so you never ship it.

Wrong mental model: a single catch (e: Exception) with the message "No internet connection."

Why it's wrong: "no internet" is only the IOException case — the request never reached the server. A 404 (wrong URL), 401 (bad/expired auth), or 500 (their bug) all reached the server and come back as a Retrofit HttpException, not an IOException.

Correct model: catch IOException first (offline → show Retry), then HttpException (read e.code(); a 4xx is usually your bug to fix, not something to spam-retry), then a generic last resort.

Symptom you'd see: a tester on perfect Wi-Fi reports "it says I'm offline" — because your code mislabeled a 500 as a connectivity problem, and your Retry button just re-hits the same broken endpoint forever.

Wrong mental model: "I marked it suspend, so it can't block the UI."

Why it's wrong: suspend only means the function can suspend; it does not by itself move work to a background thread. viewModelScope launches on Dispatchers.Main. Retrofit's generated suspend functions are main-safe (they internally switch to a background dispatcher), which is why the live-coding code is correct. But if you do your own CPU/IO work — parsing a huge string, reading a file — inside that coroutine, it runs on Main and freezes the UI.

Correct model: wrap your own blocking work in withContext(Dispatchers.IO) { … }. Trust Retrofit/Room to be main-safe; don't trust your own loops.

Symptom you'd see: the spinner appears, the screen jank-freezes for a beat on a big response, and on slower devices you get an ANR — even though "everything is suspend."

Wrong mental model: data class Photo(val title: String) — "the docs say title is always there."

Why it's wrong: JSON is dynamically typed and APIs change. The day the server omits title or sends null, kotlinx.serialization throws a MissingFieldException at parse time — which surfaces as a thrown exception from your Retrofit call, crashing your "Success" path you assumed was safe.

Correct model: model truly-optional fields as nullable (val title: String?) or give them a default (val tags: List<String> = emptyList()), and set Json { ignoreUnknownKeys = true } so extra fields are skipped (extra keys are fine; missing required keys are the killer).

Symptom you'd see: the app works for weeks, then crashes for a subset of users whose data happens to be missing one field — a "works on my machine" bug that's brutal to reproduce.

Wrong mental model: inject QuoteApiService straight into the ViewModel and call service.getQuotes() from viewModelScope.

Why it's wrong: the ViewModel now knows about HTTP, DTOs, and exceptions. You can't swap in a fake for offline work, you can't add caching (Session 11's offline-first), and the ViewModel grows two jobs: networking and state. That seam is exactly where Session 11 inserts the local-database source of truth.

Correct model: the repository owns "where data comes from" and returns clean domain types (or Result<T>); the ViewModel owns "what state the screen is in." Keep network DTOs out of the ViewModel entirely.

Symptom you'd see: when you try to add caching or write a quick fake for class, you have to rewrite the ViewModel — and the test/offline path becomes "just trust the network."

Wrong mental model: set state to Loading, call the network, set Success — and assume a failure "doesn't matter, it'll just not update."

Why it's wrong: if the call throws and you never catch it, the line that sets Success never runs, so the state is stuck on Loading forever. Worse, an uncaught exception in viewModelScope can crash the app.

Correct model: always reach a terminal state. Wrap the call in try/catch (or runCatching → Result) and map every outcome — success, empty, and each failure — to a concrete UiState. The reduce function above guarantees this: there is no path out of it that isn't a real state.

Symptom you'd see: a spinner that never stops. Users force-quit; you can't reproduce it because your network never fails on your desk.

How a senior engineer actually wires this

The repository never lets a raw exception leak to the ViewModel. It wraps the call in runCatching { … } and returns a Result<T> (or a sealed NetworkResult); the ViewModel folds that into the sealed UiState — the exact reduce pattern you just wrote. One OkHttpClient / one Retrofit instance is built once and shared (a singleton via by lazy), because creating one per call leaks connection pools and thread pools. The current converter coordinate is com.squareup.retrofit2:converter-kotlinx-serialization, now first-party in Retrofit 3 (the same one used in the Retrofit-flow lab); the older com.jakewharton… artifact still works on 2.x but Square folded it into Retrofit itself. And Json { ignoreUnknownKeys = true } is non-negotiable — servers add fields you didn't model all the time, and without it your app crashes the day they ship a new feature.

Tasteful gotcha: a Retrofit suspend fun that returns List<Photo> directly throws HttpException on a non-2xx; but if you declare it as Response<List<Photo>> instead, a 404 does not throw — you get a Response with isSuccessful == false, and forgetting to check that is how a "successful" call silently delivers null/empty data. Pick one style and be deliberate: bare body (exceptions) is simplest for this course.
