1 · Material 3: color scheme, typography, shape & MaterialTheme

Analogy: Think of MaterialTheme as a building's electrical system. You wire up the outlets (color, type, shape) once at the top, and every composable downstream just "plugs in" with MaterialTheme.colorScheme.primary instead of running its own generator. Change the source, and the whole app updates — no rewiring each room.

Material 3 theming has three pillars, each a system you provide to MaterialTheme(...):

  • Color — a ColorScheme of ~30 named roles: primary, onPrimary, primaryContainer, secondary, surface, background, error, etc. The on* roles are the content color that sits on that background (this is how M3 guarantees contrast).
  • Typography — a Typography object of named text styles: displayLarge, headlineMedium, titleLarge, bodyLarge, labelSmall, etc.
  • Shape — a Shapes object mapping size buckets (extraSmall … extraLarge) to corner radii used by buttons, cards, sheets.

Each chip shows a base color role with its paired on* content color drawn on top — exactly how M3 guarantees readable contrast (e.g. onPrimary text sits on primary). The on* role is never a separate background; it's the foreground that goes on the matching surface.

The golden rule: never hard-code colors, fonts, or corner radii in a composable. Read them from the theme. Below, the same Card + Text, done wrong then right:

Kotlin
// Anti-pattern: literal colors & sizes everywhere.
// Breaks in dark mode, ignores dynamic color, hard to restyle.
Card(
    colors = CardDefaults.cardColors(containerColor = Color(0xFFFFFFFF)),
    shape = RoundedCornerShape(4.dp)
) {
    Text(
        text = "Workout summary",
        color = Color(0xFF000000),
        fontSize = 22.sp,
        fontWeight = FontWeight.Bold
    )
}
Kotlin
// Theme-aware: colors, type & shape come from MaterialTheme.
// Dark mode + dynamic color "just work"; one theme edit restyles all.
Card(
    // Card already defaults to surface colors & the medium shape,
    // so often you don't pass them at all:
) {
    Text(
        text = "Workout summary",
        style = MaterialTheme.typography.titleLarge,
        color = MaterialTheme.colorScheme.onSurface
    )
}

Here is a complete, realistic theme file. This is what you'll paste into your own project in the lab. One brand-new piece of syntax appears at the bottom — the content: @Composable () -> Unit parameter — so read the definition first, then the file:

🧩
One-sentence definition

content: @Composable () -> Unit is a slot: a parameter whose type is a function that emits UI, so the caller hands you a chunk of UI and you drop it into your layout by calling content() wherever you want it to appear.

Kotlin · ui/theme/Theme.kt
import androidx.compose.foundation.isSystemInDarkTheme
import androidx.compose.material3.MaterialTheme
import androidx.compose.material3.darkColorScheme
import androidx.compose.material3.dynamicDarkColorScheme
import androidx.compose.material3.dynamicLightColorScheme
import androidx.compose.material3.lightColorScheme
import androidx.compose.runtime.Composable
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.platform.LocalContext
import android.os.Build

// 1) Define brand colors (generate these with the Material Theme Builder).
private val BrandGreen      = Color(0xFF3DDC84)
private val BrandGreenDark  = Color(0xFF20B265)
private val BrandTeal       = Color(0xFF073042)
private val BrandAmber      = Color(0xFFFFB300)

// 2) Two color schemes — one per theme.
private val LightColors = lightColorScheme(
    primary = BrandGreenDark,
    onPrimary = Color.White,
    primaryContainer = Color(0xFFB8F2D4),
    secondary = BrandTeal,
    tertiary = BrandAmber,
    background = Color(0xFFF4F7F6),
    surface = Color(0xFFFFFFFF),
    onSurface = Color(0xFF0E1A1F)
)

private val DarkColors = darkColorScheme(
    primary = BrandGreen,
    onPrimary = BrandTeal,
    primaryContainer = Color(0xFF0A4A5A),
    secondary = Color(0xFF7FD7EA),
    tertiary = Color(0xFFFFC233),
    background = Color(0xFF06181E),
    surface = Color(0xFF0C2630),
    onSurface = Color(0xFFE6F1EE)
)

// 3) The theme composable wraps your app's content.
@Composable
fun KeiserFitTheme(
    darkTheme: Boolean = isSystemInDarkTheme(),
    dynamicColor: Boolean = true,          // Material You on Android 12+
    content: @Composable () -> Unit
) {
    val context = LocalContext.current
    val colorScheme = when {
        dynamicColor && Build.VERSION.SDK_INT >= Build.VERSION_CODES.S ->
            if (darkTheme) dynamicDarkColorScheme(context)
            else dynamicLightColorScheme(context)
        darkTheme -> DarkColors
        else      -> LightColors
    }

    MaterialTheme(
        colorScheme = colorScheme,
        typography = KeiserTypography,   // defined in Type.kt
        shapes = KeiserShapes,           // defined in Shape.kt
        content = content
    )
}

The slot lines are what make this a reusable wrapper. Reading them in order:

  • content: @Composable () -> Unit — a parameter named content whose type is @Composable () -> Unit: "a composable function that takes no arguments and emits UI." You're not receiving a String or a number — you're receiving a piece of UI-emitting code the caller will hand you.
  • content = content — you forward that received slot into MaterialTheme(...)'s own content slot, so whatever the caller passed ends up inside the themed scope. (You could also write MaterialTheme(...) { content() } — same result; the named-argument form is what Studio generates.)
  • content() — calling the slot is how you place the caller's UI at this exact spot in the tree. Here it happens indirectly via MaterialTheme; in your own slot you'd call content() yourself (see below).

The caller supplies the slot as a trailing lambda — that's the { … } block you've been writing after Surface, Column, and Scaffold all along. So this call passes Greeting() into the theme's slot:

Kotlin · the caller fills the slot
KeiserFitTheme {        // everything in these braces becomes `content`
    Greeting()          // …and gets placed where KeiserFitTheme calls content()
}

You can build your own slot composable with the exact same pattern — accept a content parameter, then call it where you want the caller's UI to land:

Kotlin · make your own slot
@Composable
fun Labeled(label: String, content: @Composable () -> Unit) {
    Column {
        Text(label)     // your own UI first…
        content()       // …then drop in whatever the caller passed
    }
}

// Caller fills the slot with a trailing lambda:
Labeled(label = "Heart rate") {
    Text("72 bpm")      // this Text becomes `content`
}

This is the slot API pattern, and it's everywhere in Compose: Scaffold has topBar + content slots, Button has a content slot (the label), Surface and Column take their children as a slot. Learn it once here and you can read all of them.

⚠️
Common confusion

content is not data you compute or return — it's UI-emitting code you were handed. You never write return content or treat it like a String; you invoke content() to place it. And forgetting the parentheses (content instead of content()) means the slot never gets called, so the caller's UI silently never appears.

💡
Where the theme is applied

Wrap your top-level content in setContent { KeiserFitTheme { /* app */ } } inside MainActivity.onCreate. Everything inside the lambda can now read MaterialTheme.colorScheme, .typography, and .shapes.

Kotlin · ui/theme/Type.kt & Shape.kt
import androidx.compose.material3.Typography
import androidx.compose.material3.Shapes
import androidx.compose.foundation.shape.RoundedCornerShape
import androidx.compose.ui.text.TextStyle
import androidx.compose.ui.text.font.FontWeight
import androidx.compose.ui.unit.dp
import androidx.compose.ui.unit.sp

val KeiserTypography = Typography(
    titleLarge = TextStyle(fontWeight = FontWeight.Bold,   fontSize = 22.sp, lineHeight = 28.sp),
    bodyLarge  = TextStyle(fontWeight = FontWeight.Normal, fontSize = 16.sp, lineHeight = 24.sp),
    labelSmall = TextStyle(fontWeight = FontWeight.Medium, fontSize = 11.sp, lineHeight = 16.sp)
    // Unspecified styles fall back to Material 3 defaults — that's fine.
)

val KeiserShapes = Shapes(
    extraSmall = RoundedCornerShape(4.dp),
    small      = RoundedCornerShape(8.dp),
    medium     = RoundedCornerShape(14.dp),   // matches our card radius
    large      = RoundedCornerShape(20.dp)
)

2 · Light / dark theme & dynamic color (Material You)

Because every composable reads from the theme, supporting dark mode is almost free: you supply a second ColorScheme and pick it based on isSystemInDarkTheme() (already wired in the theme composable above). The OS flips darkTheme when the user toggles system dark mode.

Dynamic color (Material You, Android 12 / API 31+) goes further: the OS derives a color scheme from the user's wallpaper. dynamicLightColorScheme(context) / dynamicDarkColorScheme(context) return those schemes. You must guard with a version check (Build.VERSION.SDK_INT >= S) because older devices don't have it.

To preview dark mode without changing system settings, add a dark @Preview (see idea 4) or wrap a preview in your theme with darkTheme = true.

Kotlin
// A simple in-app theme switch backed by state (no system change needed).
// Useful for a settings screen and for demoing light/dark live.
var userPrefersDark by rememberSaveable { mutableStateOf(false) }

KeiserFitTheme(darkTheme = userPrefersDark, dynamicColor = false) {
    Scaffold { padding ->
        Column(Modifier.padding(padding)) {
            Switch(
                checked = userPrefersDark,
                onCheckedChange = { userPrefersDark = it }
            )
            Text("Dark theme", style = MaterialTheme.typography.bodyLarge)
        }
    }
}

3 · Responsive & adaptive layout

Analogy: A responsive layout is like water — it fills whatever container you pour it into. We achieve that in Compose with fillMaxWidth()/fillMaxHeight() (stretch to the parent) and Modifier.weight(...) (split remaining space proportionally), instead of fixed .dp widths.

There are two related ideas:

  • Responsive — the same layout stretches/shrinks fluidly (weights, fillMax, Arrangement.spacedBy).
  • Adaptive — the layout changes shape at breakpoints (e.g., a list-only phone screen becomes a list+detail two-pane on a tablet). Decide with WindowSizeClass or BoxWithConstraints.
Compact width → single column (list only)
Expanded width → two-pane (list + detail)

Weights & fillMax — the responsive workhorses:

Kotlin
// Two cards that always split the row 1:2, regardless of screen width.
Row(
    modifier = Modifier
        .fillMaxWidth()
        .padding(16.dp),
    horizontalArrangement = Arrangement.spacedBy(16.dp)
) {
    StatCard(modifier = Modifier.weight(1f))   // takes 1 share
    ChartCard(modifier = Modifier.weight(2f))  // takes 2 shares (twice as wide)
}

Adaptive via WindowSizeClass — the recommended way to choose a layout. Width buckets: Compact (< 600dp, phones), Medium (600–840dp, large phones / small tablets), Expanded (≥ 840dp, tablets / desktop).

Kotlin · MainActivity.onCreate
// build.gradle: implementation("androidx.compose.material3:material3-window-size-class")
import androidx.compose.material3.windowsizeclass.WindowWidthSizeClass
import androidx.compose.material3.windowsizeclass.calculateWindowSizeClass

setContent {
    KeiserFitTheme {
        val windowSize = calculateWindowSizeClass(this)
        when (windowSize.widthSizeClass) {
            WindowWidthSizeClass.Compact -> WorkoutListScreen()            // phone: list only
            else -> WorkoutListDetailScreen()                              // tablet: two-pane
        }
    }
}

Or contrast it with BoxWithConstraints, which lets a single composable branch on the space it was given (great for reusable components):

Kotlin
// Best for top-level navigation decisions (whole-screen layout).
when (windowSize.widthSizeClass) {
    WindowWidthSizeClass.Compact  -> NavigationBar { /* bottom bar */ }
    WindowWidthSizeClass.Medium   -> NavigationRail { /* side rail */ }
    WindowWidthSizeClass.Expanded -> PermanentNavigationDrawer { /* drawer */ }
}
Kotlin
// Best for a single reusable component reacting to ITS OWN width.
@Composable
fun ProfileBlock() {
    BoxWithConstraints {
        if (maxWidth < 480.dp) {
            Column { Avatar(); ProfileDetails() }   // stacked when narrow
        } else {
            Row { Avatar(); ProfileDetails() }      // side-by-side when wide
        }
    }
}
⚠️
Don't reach for absolute pixels

Avoid Modifier.size(412.dp) to "fit the Pixel 7." That breaks on every other device. Use fillMaxWidth(), weight(), and size classes. Fixed .dp is fine for spacing and icons, not for whole-screen widths.

4 · Spacing system & consistent components (UI polish)

Polish comes from consistency, not cleverness. Pick a spacing scale (commonly multiples of 4dp: 4, 8, 12, 16, 24, 32) and use it everywhere. Centralize values so the whole app shares one rhythm — a tiny custom object works well, or define them in the theme.

Kotlin · ui/theme/Spacing.kt
import androidx.compose.ui.unit.dp

// One source of truth for spacing — reference everywhere.
object Spacing {
    val xs = 4.dp
    val sm = 8.dp
    val md = 16.dp
    val lg = 24.dp
    val xl = 32.dp
}

// Usage:
Column(
    modifier = Modifier.padding(Spacing.md),
    verticalArrangement = Arrangement.spacedBy(Spacing.sm)
) { /* ... */ }

Pair the spacing system with consistent component use: prefer Scaffold for screen structure (it handles top/bottom bars and insets), reuse small composables (StatCard, SectionHeader) rather than copy-pasting layouts, and lean on Material defaults (Button, Card, OutlinedTextField) which already honor the theme.

  1. Add the window-size-class dependency

            if (size.widthSizeClass == WindowWidthSizeClass.Expanded) {
                WorkoutListDetail()     // two-pane

                WorkoutList()           // single column
            }

    Kotlin
    @Composable
fun WorkoutListDetail() {
    Row(Modifier.fillMaxSize()) {
        WorkoutList(
            modifier = Modifier.weight(1f)        // left third-ish
        )
        WorkoutDetail(
            modifier = Modifier
                .weight(2f)                        // right: twice as wide
                .fillMaxHeight()
        )
    }
}

🧠 Master it — Material 3 theming & adaptive layout

The four ideas above are easy to read and easy to get subtly wrong in real code. This section makes you build the hardest one from first principles, names the mistakes that actually ship to users, shows the senior-engineer pattern, and then sends you to deliberate practice. Work it top to bottom — predict before you run, and don't peek at a solution until you've made the cases go green yourself.

1 · Worked example: deriving the width breakpoints (then you finish it)

The single highest-leverage skill today is turning a raw available width into the correct adaptive decision. Studio gives you WindowSizeClass, but the logic underneath is just bucketing a number into Compact / Medium / Expanded. If you can reproduce that bucketing exactly — including the boundaries — you understand the abstraction instead of memorizing an API. Let's reason it out.

🔎
The reasoning, step by step
  1. What are the official cut points? Material's width classes are: Compact = width < 600dp, Medium = 600dp ≤ width < 840dp, Expanded = width ≥ 840dp. Two numbers, 600 and 840, define everything.
  2. Which side owns the boundary? The boundaries are inclusive on the lower end of the larger bucket. Exactly 600dp is Medium, not Compact. Exactly 840dp is Expanded. This off-by-one is the #1 bug in hand-rolled breakpoint code, so encode it deliberately with >=, not >.
  3. What about garbage input? A width can't be negative. Decide the contract: treat < 0 as invalid and map it to the smallest sane bucket (Compact) rather than crashing — defensive layout code never throws on a measurement.
  4. Order the checks largest-first or guard explicitly. If you write if (w < 600) Compact else if (w < 840) Medium else Expanded, the inclusive boundaries fall out for free. That's the cleanest shape.

2 · Common misconceptions & pitfalls

Each of these is a real mistake that compiles, runs, and then misbehaves on a user's device. Open one, find the symptom you'd actually see.

Wrong mental model: a literal color is “safe” because you can see it works in the preview you're looking at.

Why it's wrong: the preview you're looking at is light mode. A literal color is frozen — it can't follow the theme. In dark mode your surface flips to near-black, but your literal text stays black, so it vanishes. Dynamic color (Material You) can also recolor surfaces; literals ignore that entirely.

Correct model: read the role, not the value. Text on a surface uses MaterialTheme.colorScheme.onSurface; the framework guarantees the on* pairing has contrast in every scheme. Reserve literals for your theme definition file only.

Symptom: “Dark text on a dark background” / invisible text after toggling system dark mode — the exact bug listed in the lab troubleshooting.

Wrong mental model: modifiers are a bag of properties, so .padding().background() and .background().padding() produce the same thing.

Why it's wrong: a Modifier chain is an ordered pipeline applied outside-in. Modifier.padding(16.dp).background(Color.Red) paints red inside the padding (red box with a transparent margin). Modifier.background(Color.Red).padding(16.dp) paints red first, then insets the content — red fills the whole area with 16dp of red showing around the content. Same with .clickable: putting padding before it shrinks the tap target.

Correct model: read the chain left-to-right as “each modifier wraps the next.” Decide deliberately whether color/clip/click should be inside or outside the spacing.

Symptom: backgrounds that don't reach the edges, ripples that don't cover the visible card, or tap targets that feel “dead” near the edges.

Wrong mental model: you tested on one device, it fit, so a fixed width is “correct.”

Why it's wrong: 412dp is one device's portrait width. On a narrower phone it overflows and clips; on a tablet or in split-screen it leaves a tiny island in a sea of empty space; rotate the Pixel itself and it's wrong. Absolute pixel layouts are device-coupled by construction.

Correct model: let the layout flow — fillMaxWidth(), Modifier.weight(), and Arrangement.spacedBy() — and only change shape at real breakpoints via WindowSizeClass / BoxWithConstraints. Keep fixed .dp for spacing, icon sizes, and minimum touch targets, never whole-screen widths.

Symptom: horizontal scrollbars / clipped content on small phones, and an absurd amount of dead space on tablets and foldables.

Wrong mental model: the boring gray preview means your composable is ugly, so you start tweaking colors inside the composable.

Why it's wrong: a @Preview renders in isolation. If you don't wrap its content in YourAppTheme { }, it falls back to default Material colors/type — it ignores your custom scheme entirely. The composable is fine; the preview just isn't seeing your theme.

Correct model: every preview body is YourAppTheme { TheComposable() }. If you preview light and dark, wrap each accordingly (or pass uiMode = UI_MODE_NIGHT_YES). The running app is always themed because setContent wraps it — previews must do the same wrapping manually.

Symptom: “my preview looks broken/plain but the app looks right” — a missing theme wrapper, not a styling problem.

Wrong mental model: turning on dynamic color enhances your palette while keeping your brand identity.

Why it's wrong: dynamic color (Android 12+) derives the whole scheme from the user's wallpaper and overrides your hand-picked brand colors. With dynamicColor = true, two users with different wallpapers see two different-colored versions of your app — and your green-fitness brand might come out lavender.

Correct model: decide intentionally. If brand consistency matters, set dynamicColor = false and ship your scheme. If you want the “feels native, matches my phone” effect, keep it on — but guard with Build.VERSION.SDK_INT >= S and provide your brand scheme as the fallback for older devices.

Symptom: “my brand colors disappeared on a Pixel” / inconsistent app color across users — expected behavior when dynamic color is left on.

3 · In production

🏭
How a senior engineer actually ships this

A production theme is generated, not hand-typed. You design the seed color in the Material Theme Builder, export the full Color.kt + Theme.kt, and never touch the 30 roles by hand — that keeps tonal contrast correct. The theme composable is wired exactly once, at the top of setContent, with dynamic color off by default unless the product explicitly wants Material-You theming (brand teams usually keep it off). Adaptive decisions live in a single place near the top of the screen — typically a when on WindowSizeClass.widthSizeClass choosing NavigationBar (Compact) → NavigationRail (Medium) → PermanentNavigationDrawer (Expanded) — so the rest of the tree stays size-agnostic and reusable. Spacing comes from one constants object (or a theme CompositionLocal), never scattered magic numbers. And every screen ships with a small multi-@Preview stack (Phone, Tablet, Dark, fontScale = 1.5f) so reviewers catch dark-mode and large-font breakage in code review, before it reaches a device.

Tasteful gotcha: calculateWindowSizeClass() measures the window, not the screen. In multi-window / split-screen / foldable modes the window is smaller than the physical display — so size class is recomputed automatically and your “tablet” can legitimately become Compact when the user splits the screen in half. That's the feature, not a bug: branch on the window, and your app does the right thing in split-screen for free. Never branch on raw screen size or hardware model.
