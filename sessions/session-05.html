Today students learn the single most important idea in Jetpack Compose: UI is a function of state. When state changes, Compose recomposes. We build a counter, then a live Tip Calculator, and practice state hoisting so composables stay stateless, reusable, and testable.

1 · State drives the UI: recomposition

A Composable function does not "draw" the screen the way println writes text. It describes what the UI should look like for the current state. When that state changes, Compose re-runs the affected composables to produce a fresh description and updates only what changed. That re-run is called recomposition.

💡
Analogy

Think of a spreadsheet. Cell C1 = A1 + B1. You never tell C1 to update — you change A1 and the cell recalculates itself. In Compose, your UI is C1 and your state is A1. Change the state; the UI recomputes.

state ──▶ Composable(state) ──▶ UI on screen ▲ │ │ user event (tap, type) │ └────────── update state ◀───────────┘ (this triggers recomposition)

The loop: state → UI → user event → new state → recomposition → new UI. This is "unidirectional data flow." Your job is to (a) hold the state and (b) provide event callbacks that change it.

You don't need this to build apps — but if you like knowing why the rules exist, here is the real machine behind "change state → the UI updates." It is one loop with automatic dependency tracking:

setContent { } → run composables → build UI tree → layout → draw ↑ │ └─── state changes ◀── onClick / input writes state ◀──────────────┘ (only the composables that read that state re-run)

Five pieces make that work:

  1. A composable emits into a tree. When your @Composables run, each Text/Column/… emits a node into an in-memory tree Compose keeps (the slot table) — "what the screen should be right now."
  2. The compiler instruments every composable. The Compose compiler plugin rewrites each function to carry a hidden $composer and inserts positional markers, so Compose knows which slot in the tree each call belongs to across re-runs. That is what lets remember find its stored value again.
  3. State reads are tracked; writes invalidate (the snapshot system). Reading state.value during composition quietly records "this composable depends on this state." Writing state.value = … notifies every composable that read it and flags them invalid. You never register a listener — reading the state IS the subscription.
  4. Recomposition re-runs only the invalid ones — and skips the rest. On the next frame Compose re-executes just the flagged composables (one whose inputs did not change is skipped), diffs the new nodes against the tree, and applies the minimal change. That is why bumping one counter doesn't redraw the whole screen.
  5. remember + the slot table = persistence. remember { … } stashes a value in that position's slot; on recomposition Compose reads it back instead of recomputing. No remember → no slot → the value resets every re-run (the misconception above, mechanically explained).

Every frame (~16 ms), the invalidated work flows through three phases:

  1. Composition — run/recompose → update the tree of what to show.
  2. Layout — measure & place each node (this is where the modifier chain does its work — size, padding, position).
  3. Drawing — paint the pixels.

Compose skips phases it doesn't need (a color-only change can re-draw without re-composing), and many state writes in one frame coalesce into a single recomposition.

The analogy that makes it click — a spreadsheet:

  • A cell's formula ↔ a @Composable function
  • A formula reads other cells ↔ a composable reads State
  • Change a cell → only its dependents recalc ↔ write State → only its readers recompose
  • The sheet tracks the dependency graph for you ↔ the snapshot system tracks reads for you

You never tell a spreadsheet "when A1 changes, update B1" — you declare B1 = A1 * 2 and it works out the graph. Compose is that, for UI.

🎯
Why this session's rules fall straight out of the machine

Keep composables side-effect-free — Compose decides when, whether, in what order, and on which thread to run them (it can skip, reorder, even parallelize). Always remember state — without a slot it can't survive a re-run. Events only change state — a write is the one thing that triggers invalidate → recompose; the click never touches the tree directly.

2 · mutableStateOf and remember

Two separate problems, two tools:

  • mutableStateOf(...) creates an observable value. When you write to it, Compose knows to recompose anything that read it.
  • remember { ... } stores a value across recompositions so it isn't re-created every time the function re-runs.

You almost always need both together. First, the classic mistake — a plain variable that never updates the UI:

Kotlin · BROKEN 
@Composable
fun BrokenCounter() {
    // ❌ A plain var is NOT observable. Tapping changes the number in memory,
    //    but Compose never recomposes, so the screen never updates.
    var count = 0

    Button(onClick = { count++ }) {
        Text("Clicked $count times")
    }
}

Now make the value observable with mutableStateOf, and keep it across recompositions with remember:

Kotlin · WORKING 
import androidx.compose.runtime.*

@Composable
fun Counter() {
    // remember -> survives recomposition; mutableStateOf -> observable.
    // 'by' delegates so we read/write `count` directly (not count.value).
    var count by remember { mutableStateOf(0) }

    Button(onClick = { count++ }) {
        Text("Clicked $count times")
    }
}

First time you've seen by — what that keyword actually does

That little by in var count by remember { … } is doing real work, and it is not a Compose feature — it's a plain Kotlin language feature called a delegated property. You'll meet it again in later sessions (by viewModels(), by lazy), so it's worth nailing now.

🧩
One-sentence definition

by is Kotlin's property-delegation keyword: it tells the compiler to route every read of the property through the delegate's getValue and every write through its setValue, instead of storing the value in the property itself.

So var count by remember { mutableStateOf(0) } reads like this, piece by piece:

  • remember { mutableStateOf(0) } — produces a MutableState<Int>, a little box that holds 0 and is observable. This box is the delegate.
  • var count by … — the by says "count doesn't store its own value; delegate to that box." From here on, count is just a friendly name pointing at the box.
  • reading count (e.g. "Clicked $count times") — the compiler rewrites this into state.getValue(...), which hands back the box's current value and records that this composable read it (so Compose knows to recompose it later).
  • writing count++ — the compiler rewrites this into state.setValue(...), which updates the box and notifies Compose to recompose every reader.

Because reads and writes are rewritten for you, you get to treat count as if it were an ordinary var Int — no .value anywhere. That convenience is the only thing by buys you here.

Side-by-side: the exact same counter, written with by vs. with .value. These compile to the same thing — by is just sugar for the .value version:

Kotlin · `by` (delegated — no .value)
import androidx.compose.runtime.getValue   // needed for read
import androidx.compose.runtime.setValue   // needed for write

var count by remember { mutableStateOf(0) }

count++              // write  — looks like a plain Int
Text("$count")       // read   — looks like a plain Int
Kotlin · `=` (the State holder itself — uses .value)
// No getValue/setValue imports needed here.

val count = remember { mutableStateOf(0) }

count.value++        // write — you touch the box's .value
Text("${count.value}")  // read — you touch the box's .value

Same behavior, same recomposition. The only difference is ergonomics: by hides the .value; = exposes it. Notice the left side too — by needs var (the property is read/write), while the = form is a val because the box never changes, only its .value does.

⚠️
Common confusion

by is not magic and not Compose-specific — it's a compiler rewrite into getValue/setValue calls (the same mechanism behind by lazy and by viewModels()). Two traps follow from that: (1) by needs the imports androidx.compose.runtime.getValue and androidx.compose.runtime.setValue — forget them and you get a red "unresolved by" error. (2) Don't mix the two forms: with by you write count (never count.value); with = you write count.value everywhere. The by-vs-= choice is the whole reason one version needs .value and the other doesn't.

⚠️
Common misconception

"If I drop remember but keep mutableStateOf, it should still work." It won't — without remember the state is re-created as 0 on every recomposition, so the count resets. Demo this live; it's the lightbulb moment.

The three ways to declare Compose state — show all three so students recognize each in docs and code samples:

Kotlin 
// Most common, cleanest. Requires: import androidx.compose.runtime.getValue / setValue
var name by remember { mutableStateOf("") }
name = "Ada"          // write
Text(name)            // read
Kotlin 
// Explicit .value — what `by` is sugar for.
val name = remember { mutableStateOf("") }
name.value = "Ada"    // write
Text(name.value)      // read
Kotlin 
// Destructured into a value + setter (handy when hoisting).
val (name, setName) = remember { mutableStateOf("") }
setName("Ada")        // write
Text(name)            // read

3 · Event handling: lambdas, onClick, onValueChange

Compose connects UI to code through lambdas (anonymous functions) passed as parameters. Since many students are new to Kotlin, anchor the syntax explicitly.

Kotlin lambda cheat-sheet
  • { } is a lambda — a block of code you can pass around.
  • onClick = { count++ } runs when tapped.
  • onValueChange = { newText -> ... } receives the new value as a parameter.
  • If a lambda is the last argument, Kotlin lets you move it outside the parentheses: Button(onClick = {}) { Text("Hi") } — the content block is a trailing lambda.
  • it is the implicit name of a single lambda parameter: { println(it) }.
Java vs Kotlin: same idea

For students coming from Java listeners:

Kotlin 
Button(onClick = { count++ }) {
    Text("Add one")
}
Java 
button.setOnClickListener(new View.OnClickListener() {
    @Override public void onClick(View v) {
        count++;
        textView.setText("Add one");
    }
});

Same concept — "run this when tapped" — but Compose has no manual setText: change state, and the UI recomposes itself.

A TextField in Compose is controlled: it shows the value you give it and reports edits via onValueChange. You must store the value in state and feed it back, or typing appears to "do nothing."

Kotlin · Material 3 
import androidx.compose.material3.OutlinedTextField
import androidx.compose.material3.Text

@Composable
fun NameField() {
    var name by remember { mutableStateOf("") }

    OutlinedTextField(
        value = name,                       // what the field displays
        onValueChange = { name = it },      // store every keystroke back into state
        label = { Text("Your name") },
        singleLine = true
    )
    Text("Hello, $name")                    // recomposes live as you type
}
⚠️
"My TextField won't type!"

The #1 beginner bug: passing a constant to value (e.g. value = "") and never wiring onValueChange back to state. The field renders, you press keys, nothing appears — because the field always re-displays the unchanged state. Fix: value = name + onValueChange = { name = it }.

4 · State hoisting: stateless composables

State hoisting means moving state up to the caller and making the child composable stateless — it receives a value and an onValueChange callback instead of owning the state itself. The pattern is so common it has a name: a state holder passes value down and events up.

💡
Analogy

A stateless composable is like a thermostat display that just shows a number and reports button presses. The actual temperature setting lives in the house's control system (the parent). The display is dumb, reusable, and easy to test — you can show it any number.

Kotlin 
// Owns its own state. Hard to reuse, hard to test,
// the parent can't read or control the text.
@Composable
fun EditNameField() {
    var name by remember { mutableStateOf("") }
    OutlinedTextField(
        value = name,
        onValueChange = { name = it },
        label = { Text("Name") }
    )
}
Kotlin 
// Stateless: value comes IN, events go OUT. Reusable + testable.
@Composable
fun EditNameField(
    name: String,
    onNameChange: (String) -> Unit,
    modifier: Modifier = Modifier
) {
    OutlinedTextField(
        value = name,
        onValueChange = onNameChange,
        label = { Text("Name") },
        modifier = modifier
    )
}

// The PARENT now owns the state and can read it:
@Composable
fun SignupScreen() {
    var name by remember { mutableStateOf("") }
    Column {
        EditNameField(name = name, onNameChange = { name = it })
        Text("Welcome, $name")              // parent can use the value
    }
}

5 · rememberSaveable: surviving rotation

remember survives recomposition but not a configuration change like screen rotation, which recreates the Activity (callback to the Activity lifecycle from Session 1). Rotate the device and a remember counter resets to 0. Use rememberSaveable to persist simple state across these recreations.

Kotlin 
// Rotate the phone -> Activity recreated -> count is 0 again.
var count by remember { mutableStateOf(0) }
Kotlin 
import androidx.compose.runtime.saveable.rememberSaveable

// Survives rotation / process-death restore for primitive + Parcelable types.
var count by rememberSaveable { mutableStateOf(0) }
ℹ️
Rule of thumb

Use rememberSaveable for UI state the user would be annoyed to lose on rotation (form text, a counter, a selected tab). Use remember for transient/derived things. Persistent app data (survives the app fully closing) is a later topic — DataStore/Room in Unit 6.

6 · Deriving UI from state

Don't store what you can compute. If the total tip is "bill × percent," store the bill and percent in state and compute the tip in the composable body. It recomputes automatically on each recomposition. Storing a duplicate "tip" state invites bugs where the two get out of sync.

Kotlin 
var amountInput by remember { mutableStateOf("") }
var tipPercent by remember { mutableStateOf("15") }

// DERIVED — recomputed every recomposition, never stored separately:
val amount = amountInput.toDoubleOrNull() ?: 0.0
val percent = tipPercent.toDoubleOrNull() ?: 0.0
val tip = amount * percent / 100

Text("Tip: " + NumberFormat.getCurrencyInstance().format(tip))

This also answers the worry "won't the whole screen redraw on every keystroke?" — no. Compose tracks which composable read which state, and re-runs only those recomposition scopes. Trace one keystroke in the bill field through the Tip screen:

You type "5" in the bill field │ ▼ amountInput state changes ("5" → "50") tipInput unchanged │ ├─▶ bill OutlinedTextField reads amountInput → RE-RUNS (shows "50") │ ├─▶ val amount / tip / total read amountInput → RECOMPUTED │ └─▶ "Tip" + "Total" Text read those vals → RE-RUN │ ├─▶ tip-% OutlinedTextField reads only tipInput → SKIPPED (no change) │ └─▶ round-up Switch + its label read only roundUp → SKIPPED

Only the composables that read the changed state re-run; everything else is skipped. That's why "derive, don't store" is cheap — the derived vals recompute, but Compose still repaints just the few Text rows that depend on them. (Reserve derivedStateOf for the rare expensive derivation read by frequently-changing state — see Master it.)

  2. Fix it with remember + mutableStateOf

import androidx.compose.runtime.setValue

@Composable

  3. Tip Calculator v1: a controlled TextField

    Start a TipTimeScreen. Add an OutlinedTextField for the bill and a derived 15% tip.

    Kotlin · Material 3 
    import androidx.compose.foundation.layout.*
import androidx.compose.material3.*
import androidx.compose.runtime.*
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp
import java.text.NumberFormat

@Composable
fun TipTimeScreen() {
    var amountInput by remember { mutableStateOf("") }

    val amount = amountInput.toDoubleOrNull() ?: 0.0
    val tip = amount * 0.15
    val tipFormatted = NumberFormat.getCurrencyInstance().format(tip)

    Column(
        modifier = Modifier.padding(32.dp),
        verticalArrangement = Arrangement.spacedBy(16.dp)
    ) {
        Text("Calculate Tip", style = MaterialTheme.typography.titleLarge)
        OutlinedTextField(
            value = amountInput,
            onValueChange = { amountInput = it },
            label = { Text("Bill amount") },
            singleLine = true
        )
        Text("Tip: $tipFormatted", style = MaterialTheme.typography.headlineSmall)
    }
}

    You should see: typing a bill amount updates the tip line live as each digit lands.

  4. Make the percent editable + add a numeric keyboard

    Add a second field for the tip percent and a derived total. Set keyboardOptions so the numeric keypad appears.

    Kotlin · Material 3 
    import androidx.compose.foundation.text.KeyboardOptions
import androidx.compose.ui.text.input.KeyboardType

@Composable
fun TipTimeScreen() {
    var amountInput by remember { mutableStateOf("") }
    var tipInput by remember { mutableStateOf("18") }

    val amount = amountInput.toDoubleOrNull() ?: 0.0
    val tipPercent = tipInput.toDoubleOrNull() ?: 0.0
    val tip = amount * tipPercent / 100
    val total = amount + tip
    val currency = NumberFormat.getCurrencyInstance()

    Column(
        modifier = Modifier.padding(32.dp),
        verticalArrangement = Arrangement.spacedBy(16.dp)
    ) {
        OutlinedTextField(
            value = amountInput,
            onValueChange = { amountInput = it },
            label = { Text("Bill amount") },
            singleLine = true,
            keyboardOptions = KeyboardOptions(keyboardType = KeyboardType.Number)
        )
        OutlinedTextField(
            value = tipInput,
            onValueChange = { tipInput = it },
            label = { Text("Tip %") },
            singleLine = true,
            keyboardOptions = KeyboardOptions(keyboardType = KeyboardType.Number)
        )
        Text("Tip: ${currency.format(tip)}")
        Text("Total: ${currency.format(total)}",
             style = MaterialTheme.typography.headlineSmall)
    }
}

    You should see: editing either field recomputes tip and total instantly.

  5. Hoist the bill field into a stateless composable

    Extract EditNumberField as a reusable, stateless composable. The screen now owns the state and passes value + onValueChange down. Run again — behavior is identical, structure is cleaner.

    Kotlin · Material 3 
    @Composable
fun EditNumberField(
    label: String,
    value: String,
    onValueChange: (String) -> Unit,
    modifier: Modifier = Modifier
) {
    OutlinedTextField(
        value = value,
        onValueChange = onValueChange,        // event flows UP
        label = { Text(label) },
        singleLine = true,
        keyboardOptions = KeyboardOptions(keyboardType = KeyboardType.Number),
        modifier = modifier
    )
}

@Composable
fun TipTimeScreen() {
    var amountInput by remember { mutableStateOf("") }   // state lives HERE now
    var tipInput by remember { mutableStateOf("18") }
    // ... derive amount/tip/total exactly as before ...

    Column(/* ... */) {
        EditNumberField(
            label = "Bill amount",
            value = amountInput,
            onValueChange = { amountInput = it }          // value flows DOWN
        )
        EditNumberField(
            label = "Tip %",
            value = tipInput,
            onValueChange = { tipInput = it }

Build steps

  1. Sketch your state first

    On paper or in a comment, list the state: amountInput: String, tipInput: String, roundUp: Boolean. Everything else (tip, total) is derived, not stored.

  2. Add the round-up toggle

    Kotlin · Material 3 
    var roundUp by remember { mutableStateOf(false) }

Row(verticalAlignment = Alignment.CenterVertically) {
    Text("Round up tip", modifier = Modifier.weight(1f))
    Switch(checked = roundUp, onCheckedChange = { roundUp = it })
}

  3. Compute the tip with validation + round-up

    Kotlin 
    import kotlin.math.ceil
import java.text.NumberFormat

// Return the raw Double so we can both display it AND add it to the total.
private fun calculateTip(amount: Double, percent: Double, roundUp: Boolean): Double {
    var tip = percent / 100 * amount
    if (roundUp) tip = ceil(tip)        // always round the tip UP
    return tip
}

// In the composable:
val currency = NumberFormat.getCurrencyInstance()
val amount = amountInput.toDoubleOrNull() ?: 0.0    // validation: bad input -> 0.0
val percent = tipInput.toDoubleOrNull() ?: 0.0
val tip = calculateTip(amount, percent, roundUp)    // a Double
val total = amount + tip

Text("Tip: ${currency.format(tip)}")
Text("Total: ${currency.format(total)}")

    Keep calculateTip returning a Double so you can both display it and add it to the total, then format once with NumberFormat at the point of display.

  4. Wire it into your hoisted fields + run

    Reuse the EditNumberField from live-coding for both inputs. Run on the emulator and confirm the total moves as you type and toggle.

  5. Self-check & commit

    Walk the acceptance checklist above, then git commit -m "Tip Calculator: state, validation, round-up".

Troubleshooting (post on screen)
  • Field won't accept typing → you passed a constant to value; use value = amountInput + onValueChange = { amountInput = it }.
  • App crashes when the field is empty → you used .toDouble(); use .toDoubleOrNull() ?: 0.0.
  • Number never updates → missing remember, or you stored tip in state instead of deriving it.
  • Letters appear in the bill field → add keyboardOptions = KeyboardOptions(keyboardType = KeyboardType.Number) (note: this changes the keyboard, validation still needed).
  • Unresolved by / getValue → add import androidx.compose.runtime.getValue and setValue.

import androidx.compose.foundation.layout.*
import androidx.compose.material3.*
import androidx.compose.runtime.*
import androidx.compose.ui.Alignment
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp
import androidx.compose.ui.window.ComposeViewport

★ Master it: derived state & the hoisting refactor

This is where the average student and the strongest engineer in the room split apart. Anyone can paste a TextField. The skill is knowing what belongs in state, what should be derived, and who should own it. Work the example, complete the faded version, then drill the misconceptions until they stop biting you in code review.

1 · Worked example: reduce a tip screen to its true state

The hardest idea in this session is not remember — it's minimal state + everything else derived. Let's reason about a real screen before we write a line of UI. Suppose a tip screen shows: the bill, a tip %, a "round up" toggle, the computed tip, the total, and an error hint when the bill is invalid. That's six things on screen. How many should be state?

Reasoning, step by step
  1. Ask of each value: "Could the user change this directly?" The bill text, the tip-% text, and the toggle — yes, those are inputs. They must be state, because nothing else can compute them.
  2. For everything else, ask: "Can I compute it from the inputs?" The tip is bill × pct ÷ 100 (then maybe rounded). The total is bill + tip. The error hint is "is the bill text a valid number?". All three are pure functions of the three inputs → derive them, never store them.
  3. The payoff: three state slots, zero chance of the tip and the total disagreeing. If you stored a separate tip state you'd have to remember to update it in every place the bill or % can change — and the day you forget one, the UI silently lies.

Verdict: 3 state values (amountInput, tipInput, roundUp), 3 derived values recomputed on every recomposition. "Don't store what you can compute" is the single most reusable instinct from this whole session.

Here is the fully worked, idiomatic version. Predict the printed output, then Run to confirm — the harness below proves the derivation logic in pure Kotlin (the same math your composable will run on every keystroke):

Kotlin · runnable
import kotlin.math.ceil

// The pure core of the tip screen — no Compose, just the logic that
// runs on every recomposition. This is exactly what we'll DERIVE in the UI.
fun parseMoney(text: String): Double = text.toDoubleOrNull() ?: 0.0

fun tipOf(bill: Double, pct: Double, roundUp: Boolean): Double {
    val raw = bill * pct / 100.0
    return if (roundUp) ceil(raw) else raw
}

fun totalOf(bill: Double, pct: Double, roundUp: Boolean): Double =
    bill + tipOf(bill, pct, roundUp)

fun main() {
    // bill "50", 18%, no round-up -> tip 9.0, total 59.0
    val bill = parseMoney("50")
    val pct = parseMoney("18")
    println("tip   = " + tipOf(bill, pct, roundUp = false))
    println("total = " + totalOf(bill, pct, roundUp = false))

    // round-up bumps 9.0 -> 9.0 (already whole); try 23% -> 11.5 -> 12.0
    println("tip23 = " + tipOf(bill, parseMoney("23"), roundUp = true))

    // invalid bill text safely becomes 0.0 — no crash, UI just shows $0
    println("bad   = " + parseMoney("abc"))
}
💡
Read the seam

Notice the logic is a set of pure functions with no @Composable, no remember, no Android types. That's deliberate: the composable's only job is to hold the three inputs and call these. Pure logic like this is trivially testable and is the exact shape you'll lift into a ViewModel in Session 8.

2 · Common misconceptions & pitfalls

Each one is a wrong mental model that compiles fine and then misbehaves. Learn the symptom so you can diagnose it in two seconds instead of twenty minutes.

Wrong model: "A tap fires an event, my handler runs and updates the UI — just like an old setOnClickListener, a callback, or a webhook."

Why it's wrong: that is the old, imperative View model (push an event → mutate the widget by hand). Compose is state-driven, not event-driven: UI = f(state). An event's only job is to change state; the UI then updates because Compose re-runs (recomposes) every composable that read that state. The click never touches the UI directly — it is one step removed.

Kotlin · old (imperative) vs Compose (state-driven)
// OLD / webhook-style: the handler DIRECTLY mutates the widget
button.setOnClickListener {
    count++
    counterText.text = "Count: $count"   // you update the UI by hand
}

// COMPOSE: the handler ONLY updates state — never touches the UI
Button(onClick = { count++ }) {           // event's only job: change state
    Text("Count: $count")                 // UI is DERIVED from state
}

Correct model: unidirectional data flow — state flows down into composables (they read it to render), events flow up (a callback updates whoever owns the state). Then state change → recomposition → new UI. Think spreadsheet: you declare B1 = A1 * 2; you never write "when A1 changes, update B1."

Symptom you'd see: someone with the event-driven model hunts for a way to "grab the Text and set it" inside onClick (there isn't one), or mutates a plain var expecting the screen to follow — and nothing changes, because nothing that was read as state was updated.

Wrong model: "mutableStateOf(0) creates the state, so it persists."

Why it's wrong: mutableStateOf only makes a value observable; remember is what keeps it across recompositions. Without remember, every recomposition re-runs var count by mutableStateOf(0) and resets it to 0.

Correct model: remember = "store it across re-runs", mutableStateOf = "make writes trigger a re-run". You almost always need both: remember { mutableStateOf(0) }.

Symptom you'd see: the counter flickers back to 0 (or the field clears) the instant anything else on the screen recomposes — it looks like the tap "didn't register," but it did and was immediately erased.

Wrong model: "Computed values are important, so they deserve their own mutableStateOf."

Why it's wrong: A derived value stored as state has to be manually re-synced everywhere an input changes. Miss one update path and you get two sources of truth that silently disagree.

Correct model: store only inputs; compute tip and total as plain vals in the composable body. They recompute for free on every recomposition. Reserve derivedStateOf for the rare case where the derivation is genuinely expensive and reads frequently-changing state — for simple arithmetic, a plain val is correct and faster to read.

Symptom you'd see: the total updates but the tip line is stale (or vice-versa) after editing a field — the classic "the numbers don't add up" bug.

Wrong model: "I drop in an OutlinedTextField, the user types, I read its text later."

Why it's wrong: Compose text fields are controlled: they render exactly the value you pass and report edits via onValueChange. If you don't write the new value back into state, the field keeps re-displaying the old value — so typing appears to do nothing.

Correct model: value = text + onValueChange = { text = it }. Value flows down, the edit event flows up into your state; the new state flows back down and the field shows it.

Symptom you'd see: you press keys and the field stays empty (or frozen on its initial text). The #1 "my TextField won't type" support ticket.

Wrong model: "The keyboard is numeric, so the text is always a valid number."

Why it's wrong: keyboardType = Number is a hint, not a guarantee — the field is empty initially, can contain "." or "3.4.5", and hardware/paste/voice input bypass the soft keyboard entirely. "".toDouble() throws NumberFormatException and crashes the app.

Correct model: parse defensively with text.toDoubleOrNull() ?: 0.0. Validation (the keyboard hint) and parsing (the null-safe conversion) are two separate jobs — you need both.

Symptom you'd see: the app runs perfectly in the demo, then crashes the moment a tester opens the screen and the field is empty, or clears the field mid-edit.

Wrong model: "It's remembered, so it survives everything."

Why it's wrong: remember survives recomposition only. A configuration change (rotation, dark-mode toggle, font-size change) recreates the Activity and throws away everything held in plain remember.

Correct model: use rememberSaveable for UI state the user would be annoyed to lose (form text, a counter, a selected tab). It persists supported types (primitives, Parcelable, anything with a Saver) across recreation. Keep plain remember for transient/derived things.

Symptom you'd see: the user fills a form, rotates the phone to read it better, and the whole form blanks out — instant rage-quit material.

3 · In production: how a senior engineer wires this

ℹ️
The idiomatic pattern

On a real team, the rule is "hoist state until it has an owner." A leaf input (EditNumberField) is always stateless — it takes value: String + onValueChange: (String) -> Unit and nothing else. The screen-level composable owns the few input states with rememberSaveable. And the moment that logic needs to outlive the screen, talk to the network, or be unit-tested, the state moves one more level up into a ViewModel exposing a single immutable UiState via StateFlow (that's Session 8). The progression is the whole architecture story: local remember → hoisted to parent → lifted to ViewModel. You're learning rung one of a ladder you'll climb for the rest of the course.

Tasteful gotcha: a senior dev never lets a stateless leaf own its own text — but they also never over-hoist. If a piece of state is only ever read and written inside one composable and nobody else cares (e.g. whether a tooltip is expanded), hoisting it just adds noise. Hoist for a reason: a parent needs to read it, two siblings must share it, or you want to test/preview it. "Hoist everything" is as much a smell as "hoist nothing."

⚠️
One more production reflex

Type your event callbacks as the last parameter and your modifier: Modifier = Modifier as the last defaulted one, in that order: fun EditNumberField(label: String, value: String, onValueChange: (String) -> Unit, modifier: Modifier = Modifier). A modifier param that every caller can pass is the unwritten contract of a reusable composable — omit it and your component can't be sized, padded, or laid out by its parent, which is the first thing a reviewer will flag.
