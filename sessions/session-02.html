Lecture / Concept Notes

Work through each big idea below: every one has an explanation, an analogy, and copyable examples. Run the snippets yourself in a Kotlin Scratch file or the Playground as you read.

Kotlin is Google's preferred language for Android (announced 2017, "Kotlin-first" since 2019). It compiles to the same bytecode as Java, runs on the Android Runtime (ART), and calls Java libraries seamlessly — so the entire Android SDK is available.

Why teams switched:

  • Null safety in the type system — the compiler tracks what can be null, eliminating most NullPointerExceptions (Tony Hoare's "billion-dollar mistake").
  • Conciseness — data classes, type inference, and no boilerplate getters/setters cut code dramatically.
  • 100% Java interop — mix Kotlin and Java in one project; migrate file by file.
  • Modern features — first-class functions, extension functions, coroutines for async, and a great standard library.

Analogy: If Java is a manual-transmission car you've driven for years, Kotlin is the same car with an automatic gearbox, lane-keeping, and a seatbelt that physically stops you from the most common crash (the null crash). Same roads (the JVM/Android), far fewer self-inflicted wrecks.

The classic comparison — a simple "model" class:

Kotlin
data class User(val name: String, val age: Int)

fun main() {
    val u = User("Ada", 36)
    println(u)            // User(name=Ada, age=36)  -- toString() is free
    val older = u.copy(age = 37)
    println(u == User("Ada", 36))  // true  -- equals() is free
}
Java
public final class User {
    private final String name;
    private final int age;
    public User(String name, int age) { this.name = name; this.age = age; }
    public String getName() { return name; }
    public int getAge() { return age; }
    @Override public String toString() { return "User{name=" + name + ", age=" + age + "}"; }
    @Override public boolean equals(Object o) { /* ... lots of boilerplate ... */ return false; }
    @Override public int hashCode() { /* ... */ return 0; }
    // copy(): write it yourself.
}

Teaching move

Flip the tabs live. The one-line Kotlin data class vs the ~25 lines of Java is the most persuasive slide in the whole session for "why Kotlin."

val = a read-only reference (like Java's final); reassigning it is a compile error. var = a reassignable variable. Prefer val; reach for var only when the value genuinely changes. This habit makes Compose state predictable later.

Kotlin
val courseCode = "COP4667"   // type inferred as String
var attempts = 0             // mutable
attempts = attempts + 1      // OK

// courseCode = "OTHER"      // ERROR: val cannot be reassigned

val credits: Double = 3.0    // explicit type when you want it
val isFaceToFace = true      // Boolean inferred

// String templates: $var and ${expression}
val name = "Ada"
println("Welcome, $name — you have ${3 - attempts} attempts left.")
// Welcome, Ada — you have 2 attempts left.

Basic types are real objects (no primitives to worry about): Int, Long, Double, Float, Boolean, Char, String. There is no implicit widening — convert explicitly:

Kotlin
val i: Int = 7
val d: Double = i.toDouble()   // explicit; `val d: Double = i` is an ERROR
val text: String = i.toString()
val parsed: Int = "42".toInt()
⚠️
Common misconception

val means the reference is fixed, not that the object is immutable. A val pointing at a mutableListOf() can still have items added. "Can't reassign the box" ≠ "can't change what's in the box."

Functions use fun, name the return type after the params with :, and Unit means "returns nothing" (like void).

Kotlin
fun greet(name: String, punctuation: String = "!"): String {
    return "Hello, $name$punctuation"
}

// Single-expression function: drop the braces and `return`, use `=`
fun square(x: Int) = x * x          // return type inferred as Int

fun main() {
    println(greet("Ada"))                       // Hello, Ada!
    println(greet("Bob", punctuation = "."))    // named arg
    // Default arg lets you skip `punctuation` entirely.
    println(greet(punctuation = "?", name = "Cy"))  // named args in any order
    println(square(9))                          // 81
}

Named arguments make call sites self-documenting and let you skip middle defaults — this is exactly how Compose functions like Text(text = ..., color = ..., fontSize = ...) are called, so it pays off immediately in Session 3.

💡
Looking ahead to Compose

Every Composable you write next week is just a Kotlin function with default and named arguments. Master this now and your Compose call sites will read themselves.

By default a Kotlin type cannot be null. Add ? to opt into nullability. The compiler then forces you to handle the null before you use the value.

Kotlin
var nonNull: String = "hi"
// nonNull = null            // ERROR: null can't be a value of non-null String

var nickname: String? = null  // nullable: opt in with ?

// Safe call ?.  -> returns null instead of crashing if the receiver is null
val len: Int? = nickname?.length    // null (not a crash)

// Elvis ?:  -> provide a fallback when the left side is null
val safeLen: Int = nickname?.length ?: 0   // 0

// let { } -> run a block ONLY when the value is non-null
nickname?.let { n ->
    println("Nickname is $n with ${n.length} chars")  // skipped when null
}

// !! -> "I promise it's not null." Throws NPE if you're wrong. Use sparingly.
// val risky = nickname!!.length   // would crash here

Analogy: ?. is asking politely "if you exist, give me your length." ?: is "…otherwise here's a default." !! is kicking the door in and accepting the consequences.

☠
!! is a code smell

Teach !! only so students recognize and distrust it. In code review, !! almost always means "I should have used ?., ?:, or let." Each !! is a re-introduced billion-dollar mistake.

In Kotlin, if and when are expressions — they return a value. There is no ternary ?:-for-conditions operator because if already does that job.

Kotlin
val score = 87

// if as an expression
val passed = if (score >= 65) "pass" else "fail"   // "pass"

// when as an expression — replaces long if/else-if chains AND switch
val letter = when {
    score >= 90 -> "A"
    score >= 80 -> "B"
    score >= 70 -> "C"
    score >= 65 -> "D"
    else -> "F"
}                                              // "B"

// when on a value, with ranges and multiple labels
val band = when (score) {
    in 90..100 -> "excellent"
    in 70..89  -> "solid"
    0, 1, 2    -> "barely started"
    else       -> "needs work"
}

Ranges & loops:

Kotlin
for (i in 1..5) print(i)          // 12345     (inclusive range)
println()
for (i in 1 until 5) print(i)     // 1234      (excludes 5)
println()

println()

val fruits = listOf("apple", "kiwi", "mango")
for (f in fruits) println(f)

var n = 3
while (n > 0) { println(n); n-- }
💡
Why this matters for Compose

Because when returns a value, you'll use it directly inside Composables to pick a color, label, or icon from state — no temporary var needed. "Everything is an expression" is the Kotlin habit that makes declarative UI clean.

Primary constructor goes right in the class header. Properties declared with val/var there are auto-created.

Destructuring falls out of data classes for free:

Kotlin
val t = Task(1, "Write notes")
val (id, title, done) = t
println("$id / $title / $done")   // 1 / Write notes / false

Kotlin separates read-only from mutable collections. Default to read-only.

Kotlin
val nums = listOf(1, 2, 3, 4)              // read-only List
val editable = mutableListOf("a", "b")    // MutableList
editable.add("c")

val ages = mapOf("Ada" to 36, "Bob" to 28)  // read-only Map
val unique = setOf(1, 1, 2, 3)               // Set -> [1, 2, 3]

A lambda is a block of code you can pass around: { parameter -> body }. If there's one parameter you can use the implicit name it. A higher-order function is a function that takes or returns a function.

Reading the function type (Int) -> Int

🧩
One-sentence definition

(Int) -> Int is a type — it reads "a function that takes one Int and returns an Int" — and a higher-order function is simply one whose parameter (or return) is itself a function of that kind.

Before the chained example, meet the smallest possible higher-order function on its own — one that takes a function and runs it twice:

Kotlin · higher-order function in isolation
fun applyTwice(x: Int, op: (Int) -> Int): Int = op(op(x))

println(applyTwice(3) { it * 2 })   // 12

Read it piece by piece:

  • op: (Int) -> Int — a parameter named op whose type is "function taking an Int, returning an Int." This is what makes applyTwice a higher-order function: it accepts a function as an argument, just like any other value.
  • = op(op(x)) — the body calls the passed-in function. Inner op(x) runs it once; the outer op(...) feeds that result back in and runs it again.
  • applyTwice(3) { it * 2 } — the trailing { it * 2 } is the lambda supplied as op; it is the lambda's single implicit parameter. So op(op(3)) evaluates to (3 * 2) * 2 = 12.
⚠️
Common confusion

(Int) -> Int is not a value or a call — it's a type, like Int or String, that you can store in a variable, pass as an argument, or return. Nothing runs until you actually call the function: writing op just refers to it, while op(x) invokes the lambda that was passed in.

Kotlin
val nums = listOf(1, 2, 3, 4, 5, 6)

val doubled = nums.map { it * 2 }            // [2, 4, 6, 8, 10, 12]
val evens   = nums.filter { it % 2 == 0 }    // [2, 4, 6]
val total   = nums.sum()                     // 21
nums.forEach { println(it) }                 // prints each on its own line

// Chain them — reads top to bottom like a pipeline
val result = nums
    .filter { it % 2 == 0 }   // keep evens
    .map { it * it }          // square them
    .sum()                    // add them up   -> 4 + 16 + 36 = 56

// A higher-order function you WRITE:
fun applyTwice(x: Int, op: (Int) -> Int): Int = op(op(x))
println(applyTwice(3) { it + 1 })   // 5   (trailing lambda syntax)

Trailing lambda rule: if the last argument is a lambda, it can move outside the parentheses — this is why Compose's Button(onClick = { ... }) { Text("Go") } looks the way it does.

💡
Pipeline mental model

Read a chain bottom-up as a sentence: "sum, of the squares, of the evens." This functional style replaces most hand-written loops and is everywhere in Android (transforming API results into UI lists).

Scope functions run a block "on" an object. They differ in (a) how you reference the object inside (it vs this) and (b) what they return (the object itself vs the lambda's result). Today: recognize them — don't drill them.

FunctionObject asReturnsTypical use
letitlambda resultNull-check + transform a value
applythisthe objectConfigure an object, then keep it
runthislambda resultConfigure + compute a result
withthislambda resultGroup calls on one object
alsoitthe objectSide effect (log/validate), keep object
Kotlin
// apply: build & configure, return the object
val sb = StringBuilder().apply {
    append("Tasks: ")
    append(3)
}                                   // sb is the StringBuilder
println(sb)                         // Tasks: 3

// let: transform a non-null value
val length = "hello".let { it.length }   // 5

// also: tap in for a side effect without breaking the chain
val nums = listOf(1, 2, 3)
    .also { println("before map: $it") }
    .map { it * 10 }
println(nums)                       // [10, 20, 30]
ℹ️
Don't over-teach

Students confuse these. For today: "apply = configure-and-keep; let = null-safe transform." The other three are recognition-only. They will reach for these naturally once they have real objects to configure.

  5. Build a list and transform it (map / filter / sortedBy)

    Kotlin
    val tasks = listOf(

    Task(4, "Stretch",         Priority.MEDIUM, done = true)

  6. Null safety in context: find a task that might not exist

    Kotlin
    val maybe: Task? = tasks.find { it.id == 99 }   // find returns Task? (nullable)

println(label)                                   // No such task

tasks.find { it.id == 3 }?.let { found ->
    println("Found: ${found.title}")             // Found: Read Unit 2
}

    You'll see: the difference between the missing-task path (fallback) and the found path (the let block runs).

Troubleshooting (common kata errors)
  • "Function 'main' is not defined" / nothing prints: in a Scratch file each top-level statement runs, so you may not need main(); in the Playground you DO. Pick one and be consistent.
  • Kata 1: 15 prints "Fizz": put the % 3 == 0 && % 5 == 0 branch first.
  • Kata 2: crash on "x".toInt(): use toIntOrNull() (returns null) instead of toInt() (throws).
  • Kata 4: "Type inference failed": declare the lambda param type or let it infer from the list; ensure the function's generic <T> is before the function name.
  • Type mismatch Int/Double: Kotlin won't auto-convert; call .toInt()/.toDouble() explicitly.

1. val/var & string templates. Read-only vs mutable, and $name / ${expr} interpolation.

Kotlin · runnable
fun main() {
    val name = "Ada"          // val = read-only
    var score = 0             // var = mutable
    score += 10
    println("Hello, $name! Score: $score (doubled: ${score * 2})")
}

2. Null safety. Safe-call ?., the Elvis operator ?:, and a non-null result.

Kotlin · runnable
fun firstInitial(name: String?): Char {
    return name?.trim()?.firstOrNull() ?: '?'
}

fun main() {
    println(firstInitial("  Ada"))   // A
    println(firstInitial(null))      // ?
}

3. Control flow as expressions. A when that returns a value — no temporary var, no break. This is the habit that makes Composables pick a label, color, or icon straight from state.

Kotlin · runnable
fun grade(score: Int): Char = when (score) {
    in 90..100 -> 'A'
    in 80..89  -> 'B'
    in 70..79  -> 'C'
    in 65..69  -> 'D'
    else       -> 'F'
}

fun main() {
    for (s in listOf(96, 84, 73, 66, 50)) {
        println("$s -> ${grade(s)}")
    }
}

4. Data classes + collection operators. filter, map, average — the workhorses of Android data handling.

Kotlin · runnable
data class Student(val name: String, val gpa: Double)

fun main() {
    val roster = listOf(
        Student("Ada", 3.9),
        Student("Linus", 3.2),
        Student("Grace", 4.0)
    )
    val honors = roster.filter { it.gpa >= 3.5 }.map { it.name }
    println("Honor roll: $honors")
    println("Average GPA: ${roster.map { it.gpa }.average()}")
}

5. Lambdas & higher-order functions. A function that takes a function — the basis of Compose’s onClick = { ... }.

Kotlin · runnable
fun runTimes(times: Int, action: (Int) -> Unit) {
    for (i in 1..times) action(i)
}

fun main() {
    runTimes(3) { i -> println("Tick #$i") }
}

Master it — sealed state & the exhaustive when

The single highest-leverage idea in this whole session is modeling a closed set of outcomes with a sealed hierarchy and reducing it with an exhaustive when. Get this one pattern into your fingers and the rest of the course — ViewModel state, network Loading/Success/Error, navigation — is just this shape repeated. Below: a fully worked example, a faded challenge you finish, the misconceptions that bite real engineers, the production pattern, and a retrieval check.

Worked example — reasoning, step by step

We want one function that turns any screen state into a user-facing line of text. The state can be exactly one of: loading, loaded-with-data, loaded-but-empty, or failed. Watch the reasoning:

  1. 1 · Name the closed set of outcomes

    A screen that loads data is never "kind of loading and kind of loaded." It is in exactly one state at a time, and the set of states is fixed at compile time. That is the definition of a sealed type: a hierarchy the compiler knows is closed — no outside code can add a new subtype.

    Kotlin
    sealed interface ScreenState {
    data object Loading : ScreenState
    data class Loaded(val items: List<String>) : ScreenState
    data object Empty : ScreenState
    data class Failed(val message: String) : ScreenState
}

    We use sealed interface (modern, lighter than sealed class when you don't need shared state) and data object for the no-payload cases (free toString()/equals(), and it reads as a value, not a type).

  2. 2 · Reduce the state with when — and let the compiler enforce completeness

    Because the type is sealed, a when over it can be exhaustive without an else. The payoff is enormous: the day someone adds a new state, every when that forgot to handle it becomes a compile error, not a silent bug shipped to users.

    Kotlin
    fun statusLine(state: ScreenState): String = when (state) {
    ScreenState.Loading      -> "Loading…"
    is ScreenState.Loaded    -> "Showing ${state.items.size} items"   // smart-cast: state.items is in scope
    ScreenState.Empty        -> "Nothing here yet"
    is ScreenState.Failed    -> "Error: ${state.message}"             // smart-cast to Failed
}

    Two things to notice. (a) For the no-payload objects we compare by value (ScreenState.Loading), and for the data classes we use is, which smart-casts — inside that branch state is known to be Loaded, so state.items just works. (b) There is no else; delete the Empty branch and the file won't compile.

  3. 3 · Prove it runs

    Predict the printed output first, then Run to confirm.

    Kotlin · runnable
    sealed interface ScreenState {
    data object Loading : ScreenState
    data class Loaded(val items: List<String>) : ScreenState
    data object Empty : ScreenState
    data class Failed(val message: String) : ScreenState
}

fun statusLine(state: ScreenState): String = when (state) {
    ScreenState.Loading   -> "Loading…"
    is ScreenState.Loaded -> "Showing ${state.items.size} items"
    ScreenState.Empty     -> "Nothing here yet"
    is ScreenState.Failed -> "Error: ${state.message}"
}

fun main() {
    println(statusLine(ScreenState.Loading))
    println(statusLine(ScreenState.Loaded(listOf("Read", "Run", "Rest"))))
    println(statusLine(ScreenState.Empty))
    println(statusLine(ScreenState.Failed("No network")))
}

Kotlin
fun toState(rawResult: List<String>?): LoadState = when {
    rawResult == null                       -> LoadState.Loading
    rawResult.firstOrNull() == "__ERROR__"  -> LoadState.Failure(rawResult[1])
    rawResult.isEmpty()                     -> LoadState.Empty
    else                                    -> LoadState.Success(rawResult)
}

fun label(state: LoadState): String = when (state) {
    LoadState.Loading     -> "loading"
    is LoadState.Success  -> "ok:${state.names.size}"
    LoadState.Empty       -> "empty"
    is LoadState.Failure  -> "fail:${state.reason}"
}

Notice the when in toState has an else (it tests arbitrary boolean conditions), but the when in label has none — it switches over a sealed type, so the compiler proves it's complete.

Common misconceptions & pitfalls

Each of these is a real mistake that compiles-and-runs-wrong (or compiles-and-crashes) for working engineers. Open one, read the symptom, fix the model.

Wrong mental model: val items = mutableListOf(...) is safe to expose because val means immutable.

Why it's wrong: val freezes the reference, not the object. A val pointing at a MutableList can still have items added, removed, or cleared by anyone who holds it.

Correct model: immutability is about the type. Declare the property's type as the read-only List<T> (back it with a private MutableList if you must mutate internally). "Can't reassign the box" ≠ "can't change what's in the box."

Symptom you'd see: a "read-only" list mysteriously grows because a caller you forgot about kept a reference and called .add(). In Compose this shows up as state changing from a layer that was supposed to be the source of truth.

Kotlin
val items = mutableListOf("a")
items.add("b")          // compiles fine — val didn't stop this
println(items)          // [a, b]

val safe: List<String> = mutableListOf("a")
// safe.add("b")        // ERROR: List has no add() — the TYPE made it read-only

Wrong mental model: a catch-all else on a when over a sealed type is harmless defensive programming.

Why it's wrong: the else silences the compiler's best feature. When you add a new subtype (say Refreshing), the else swallows it and your screen falls through to whatever the else branch did — no error, no warning. You've traded a compile-time guarantee for a runtime surprise.

Correct model: over a sealed type, list every case and omit else. Now adding a state breaks the build at every when that needs updating — exactly where you want to be told.

Symptom you'd see: you add a new UiState and the app compiles, ships, and the new state renders as the stale fallback (often a blank or "loading" screen forever) because an else ate it.

Wrong mental model: value!! is the normal way to use a nullable value once you "know" it's set.

Why it's wrong: !! doesn't check anything — it asserts and throws NullPointerException the instant you're wrong. It re-introduces the exact billion-dollar mistake Kotlin's type system was designed to prevent, and it moves the crash far from the real cause.

Correct model: handle the null explicitly with ?. (safe call), ?: (Elvis fallback), or ?.let { } (run only when present). Reserve !! for the rare case where a non-null is a true invariant you can't express — and treat every !! in review as a question to answer.

Symptom you'd see: a NullPointerException in production on a line that "can't" be null, usually after a config change, a slow network, or an empty API field — the very moments your app is most visible.

Wrong mental model: assigning an Int where a Double is expected (or mixing them) widens silently.

Why it's wrong: Kotlin has no implicit widening. val d: Double = anInt is a compile error; you must call .toDouble(). This is deliberate — it removes a class of silent precision/overflow bugs.

Correct model: convert explicitly with .toDouble() / .toInt() / .toLong(), and remember integer division truncates: 7 / 2 == 3, while 7 / 2.0 == 3.5.

Symptom you'd see: a "type mismatch: required Double, found Int" red squiggle while wiring a tip/total/average calculation, or — worse, if you cast wrong — a tip of 0.0 because you divided two Ints before converting.

Kotlin
val parts = 7
val whole = 2
// val avg: Double = parts / whole        // = 3 (Int division!) then needs .toDouble anyway
val avg: Double = parts.toDouble() / whole  // 3.5  — convert BEFORE dividing
println(avg)

Wrong mental model: task.copy(done = true) deep-copies everything, so the copy shares nothing with the original.

Why it's wrong: copy() is a shallow copy. Scalar/immutable fields are fine, but if a field is a mutable reference (a MutableList, another mutable object), the copy points at the same instance — mutate it through one and both "see" the change.

Correct model: keep data class fields immutable (val + read-only List). Then copy() is effectively safe because there's nothing mutable to alias. This is why idiomatic Compose state is built from immutable data classes.

Symptom you'd see: you copy() a state object to "snapshot" it, mutate the original's inner list, and the snapshot changes too — a classic source of "the UI updated when it shouldn't have."

In production

🏭
How a senior engineer actually writes this

In a real Android codebase the sealed-state pattern is the contract between the ViewModel and the screen. The idiomatic shape today is:

  • Sealed interface, data object for empty cases. sealed interface UiState { data object Loading : UiState; data class Success(...) : UiState; data object Empty : UiState; data class Error(val message: String) : UiState }. sealed interface is lighter than sealed class when there's no shared state, and data object (Kotlin 1.9+) gives the singletons a sensible toString() for logs.
  • One immutable UiState per screen, exposed read-only and copied with copy() on every change. The screen does an exhaustive when (state) — no else — so a new state is a compile error, not a blank screen.
  • Model all four states from day one: Loading / Success / Error / Empty. Empty is a real state (search returned nothing), not "Success with a zero-length list you forgot to special-case."

Tasteful gotcha: don't let your sealed states leak implementation types. Error should carry a clean message: String (or a typed app error), never a raw Throwable/HttpException/SQLException — that couples your UI to your data layer and makes the when branch reach for things the screen has no business knowing. Map exceptions to a user-facing message at the boundary, and keep the state model about what the screen shows, not how it failed under the hood.
