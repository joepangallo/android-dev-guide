val declares a read-only reference (cannot be reassigned). var is mutable. Note: a val can still point to a mutable object — only the reference is fixed.

Kotlin 
val name = "Ada"        // read-only; reassigning is a compile error
var count = 0           // mutable
count += 1              // OK

// Null safety
val nickname: String? = null          // nullable type
val len = nickname?.length            // safe call -> Int? (null if nickname is null)
val shown = nickname ?: "anonymous"   // Elvis operator: fallback when null
// val bad = nickname!!.length        // !! throws NullPointerException if null — avoid

// when as an expression (returns a value)
fun label(score: Int): String = when {
    score >= 90 -> "A"
    score in 80..89 -> "B"
    score in 70..79 -> "C"
    else -> "needs work"
}

// data class: auto-generates equals/hashCode/toString/copy/componentN
data class User(val id: Int, val name: String)
val u1 = User(1, "Ada")
val u2 = u1.copy(name = "Grace")      // copy with one field changed
println(u1 == User(1, "Ada"))         // true — structural equality

// lambda + trailing-lambda syntax
val nums = listOf(1, 2, 3, 4)
val evens = nums.filter { it % 2 == 0 }   // { it ... } is a lambda; it = each element

On a cold launch the callbacks fire in this order: onCreate() → onStart() → onResume(). When leaving: onPause() → onStop() → onDestroy().

  • onCreate() — one-time setup; called once when the Activity is created. In Compose you call setContent { } here.
  • onStart() — Activity is becoming visible.
  • onResume() — Activity is in the foreground and interactive.
  • onPause() — losing focus (partially obscured); keep it fast.
  • onStop() — no longer visible; release heavy resources, persist data.
  • onDestroy() — final cleanup before the Activity is removed.
Kotlin 
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent { AppTheme { Greeting() } }   // build the UI once
    }
    override fun onStop() {
        super.onStop()
        // good place to persist data / release resources you don't need while hidden
    }
}

An explicit intent names the exact component (a specific Activity class) to start — used for navigation inside your own app. An implicit intent declares an action (and optional data) and lets the system find any app that can handle it — used for cross-app actions like opening a URL, sharing text, or dialing a number.

Kotlin 
// Explicit: start a known Activity in this app
val explicit = Intent(this, DetailActivity::class.java)
explicit.putExtra("itemId", 42)
startActivity(explicit)

// Implicit: ask the system to open a web page (any browser may handle it)
val implicit = Intent(Intent.ACTION_VIEW, Uri.parse("https://developer.android.com"))
startActivity(implicit)

// Implicit share
val share = Intent(Intent.ACTION_SEND).apply {
    type = "text/plain"
    putExtra(Intent.EXTRA_TEXT, "Check out COP4667!")
}
startActivity(Intent.createChooser(share, "Share via"))

A composable is a function annotated with @Composable that describes a piece of UI. Modifier chains decorate/layout an element (padding, size, background, click). Layout composables stack children differently:

  • Column — children top-to-bottom (vertical).
  • Row — children left-to-right (horizontal).
  • Box — children stacked on top of each other (z-order / overlap).

Modifier order applies outside-in — each modifier wraps the one after it, so reordering changes the result:

Kotlin 
// padding FIRST: the 16.dp is transparent margin AROUND a gray box
Modifier.padding(16.dp).background(Color.Gray)   // gray drawn INSIDE the padding

// background FIRST: the whole area is gray, THEN the content is inset
Modifier.background(Color.Gray).padding(16.dp)   // gray fills the edges; content padded in
Kotlin 
@Composable
fun ProfileCard(name: String) {
    Column(
        modifier = Modifier
            .fillMaxWidth()
            .padding(16.dp)          // order matters: padding applied around the column
    ) {
        Text(text = name, style = MaterialTheme.typography.titleLarge)
        Row {
            Text("Level ")
            Text("3")
        }
        Box(modifier = Modifier.size(48.dp)) {
            // children here overlap; great for badges over avatars
        }
    }
}

Compose redraws (recomposes) when observed state changes. mutableStateOf creates an observable holder; remember keeps that holder alive across recompositions so the value isn't reset every redraw. State hoisting means moving state up to the caller and passing the value down with an onValueChange callback up — making a composable stateless and reusable.

Kotlin 
// Stateful: owns its own state
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }   // survives recomposition
    Button(onClick = { count++ }) {               // reassigning count triggers recompose
        Text("Clicked $count times")
    }
}

// Hoisted (stateless): value down, event up
@Composable
fun CounterStateless(count: Int, onIncrement: () -> Unit) {
    Button(onClick = onIncrement) { Text("Clicked $count times") }
}

// Caller owns the state and survives config changes with rememberSaveable
@Composable
fun CounterScreen() {
    var count by rememberSaveable { mutableStateOf(0) }
    CounterStateless(count = count, onIncrement = { count++ })
}
